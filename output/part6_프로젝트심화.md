## 1. 전공별 AI 도구 조합 전략

### 1.1 전공 계열별 추천 도구 매트릭스

| 전공 계열 | 핵심 AI 도구 | 보조 도구 | 프로젝트 예시 |
|----------|------------|----------|------------|
| **인문학** | ChatGPT, Claude | NotebookLM, Perplexity | 텍스트 마이닝, 디지털 인문학 |
| **사회과학** | ChatGPT Code Interpreter, Jamovi | Google Forms, Claude | 설문 분석, 정책 시뮬레이션 |
| **경영학** | Perplexity, ChatGPT, Gamma | Notion AI, Claude | 시장 분석 보고서, BM 기획 |
| **자연과학** | ChatGPT, Claude, Wolfram Alpha | NotebookLM, Gemini | 실험 데이터 분석, 논문 리뷰 |
| **공학** | GitHub Copilot, Cursor, Claude Code | Replit, v0 | 프로토타입 개발, 시뮬레이션 |
| **디자인/예술** | Midjourney, DALL-E 3, Canva AI | Gamma, Claude | 포트폴리오, 컨셉 아트 |
| **교육학** | ChatGPT, NotebookLM, JASP | Canva, Gamma | 교수학습 자료, 교육 평가 |
| **의학/간호** | Perplexity, Claude, Elicit | Consensus, ChatGPT | 문헌 리뷰, 임상 데이터 분석 |

### 1.2 멀티 도구 워크플로우 3원칙

```
원칙 1: 도구별 역할 분담
  → Perplexity = 리서치 (최신 정보 + 출처)
  → ChatGPT = 분석·생성 (코드 실행, 콘텐츠)
  → Claude = 심층 분석 (논리 구조, 긴 텍스트)
  → Gamma/Canva = 시각화·발표 자료

원칙 2: 파이프라인 설계
  → 조사(Perplexity) → 분석(ChatGPT/Claude) → 시각화(Artifacts) → 발표(Gamma)

원칙 3: 1도구 1역할
  → 1개 도구로 모든 것을 하려 하지 말 것
  → 각 도구의 최강 기능만 활용
```

> **대학생 92%가 AI를 사용합니다** (DemandSage 2026). 과목당 평균 **2.1개** AI 도구를 조합하고 있으며, 과제에 AI를 활용하는 비율은 **88%**에 달합니다. AI 활용은 선택이 아니라 필수가 되었습니다.

## 2. 핵심 개념: AI 프로젝트의 과학

### 2.1 AI-PBL이 일반 PBL보다 효과적인 이유

**MDPI Education Sciences(2025)** 연구
- AI 통합 프로젝트 기반 학습(AI-PBL) vs 일반 PBL 비교 → AI-PBL이 **유의미하게 높은 효과** (대규모 효과 크기)
- AI 기반 **실시간 피드백 + 맞춤형 콘텐츠** → 학업 불안 감소, 성과 향상
- AI가 프로젝트 전 과정에서 **연속적 피드백과 자기평가** 수단으로 활용

**Harvard 물리학 연구:**
- AI 튜터 사용 학생이 전통 수업 대비 **2배 이상 학습** — 더 짧은 시간에
- Macquarie University: AI 활용 학생 시험 성적 **10% 향상**

### 2.2 4가지 프로젝트 유형별 도구 조합

| 유형 | 도구 조합 | 핵심 흐름 |
|------|----------|----------|
| **리서치형** | Perplexity + Elicit + NotebookLM + Claude | 문헌 탐색 → 분석 → 종합 → 발표 |
| **데이터 분석형** | ChatGPT CI + Claude Artifacts + Jamovi | 수집 → 정제 → 분석 → 시각화 |
| **개발형** | Cursor/Claude Code + Replit + v0 | 기획 → UI → 개발 → 배포 |
| **기획/제안형** | Perplexity + ChatGPT + Claude + Gamma | 조사 → 기획 → 시각화 → 발표 |

### 2.3 AI 프로젝트 공모전 — 최대 25억 사업화 지원

**AI Co-Scientist Challenge Korea (2026):**
- 과기정통부 주최 AI 연구동료 경진대회
- Track 1: AI 활용 과학기술 연구 (논문 형태)
- Track 2: 과학기술 AI Agent 개발 — 대상팀 **최대 25억 원** 사업화 지원 (연 5억 x 5년)
- 개인 또는 팀 참가 가능, 참가비 무료

## 3. 따라하기 실습

### 실습 A: AI로 전공 프로젝트 기획하기 ⭐

> **목표:** 전공 수업 프로젝트를 AI와 함께 체계적으로 기획합니다.
> **소요 시간:** 30분
> **준비물:** ChatGPT 계정, 현재 수강 중인 전공 과목 정보

---

**Step 1: 프로젝트 주제 발상**

ChatGPT에 다음 프롬프트를 입력합니다

```
나는 [전공] [학년]이고, 전공 수업 [과목명]의 학기 프로젝트를 기획하려 합니다.

프로젝트 주제를 5개 제안해주세요:

조건:
1. AI 도구를 적극 활용할 수 있는 주제
2. 데이터 수집 + 분석 + 시각화가 포함된 주제
3. 실제 사회적/산업적 의미가 있는 주제
4. 2~4주 내 완성 가능한 범위
5. 발표 + 보고서 형태의 산출물

각 주제에:
- 주제 설명 (1~2줄)
- 활용할 AI 도구 조합
- 데이터 소스 제안
- 예상 산출물
- 난이도 (★~★★★)
```

📍 **화면 가이드:** ChatGPT가 5개 주제를 도구 조합과 함께 제안합니다. 난이도와 관심도를 기준으로 1개를 선택하세요.

✅ **체크포인트:** 5개 주제가 생성되었나요? 각 주제에 AI 도구 조합과 데이터 소스가 포함되어 있나요?

---

**Step 2: 선택한 주제의 4주 타임라인 설계**

```
위 주제 중 [선택한 주제]를 진행합니다.

4주 프로젝트 계획을 만들어주세요:

조건:
- 팀원 [N]명
- 발표 15분 + 보고서 20페이지
- AI 활용 부분 vs 직접 작업 부분을 구분

포함:
1. 주차별 마일스톤 (Week 1~4)
2. 팀원 역할 분담 제안
3. 각 주차에 활용할 AI 도구와 구체적 프롬프트
4. 데이터 수집 방법 및 출처
5. 중간 체크포인트 (2주차 말 점검 항목)
6. AI 활용 부분에 ★ 표시
```

📍 **화면 가이드:** 주차별 계획이 역할 분담과 함께 나옵니다. AI 활용 부분(★)과 직접 작업 부분을 명확히 구분하세요.

✅ **체크포인트:** 4주 타임라인이 완성되었나요? AI 활용 vs 직접 작업이 구분되어 있나요?

🔧 **트러블슈팅:**
- 범위가 너무 넓다면 → "이 주제 중 2~4주 안에 [N]명이 완성 가능한 범위로 축소해줘. MVP 접근으로"
- 데이터를 어디서 구하는지 모르겠다면 → "한국 공공데이터포털(data.go.kr)에서 이 주제에 활용할 수 있는 데이터셋 3개를 찾아줘"

### 실습 B: 멀티 도구 파이프라인 실행 ⭐⭐

> **목표:** 4개 도구를 조합하여 프로젝트의 한 단계를 실제로 완성합니다.
> **소요 시간:** 60분
> **준비물:** Perplexity, ChatGPT, Claude, Gamma 계정

---

**Step 1: Perplexity로 리서치**

```
[프로젝트 주제]에 대해 배경 조사를 해주세요:

1. 이 주제와 관련된 최신 연구/보고서 5건 (2024~2026)
2. 한국 시장/현황 데이터
3. 관련 통계 자료 (수치 포함)
4. 전문가 의견 또는 사례 3건
5. 이 주제의 핵심 쟁점 3가지

모든 항목에 출처를 포함해주세요.
```

✅ **체크포인트:** 출처가 포함된 배경 조사가 완성되었나요?

---

**Step 2: ChatGPT로 분석 프레임워크 구성**

Perplexity 결과를 ChatGPT에 입력합니다

```
다음은 [프로젝트 주제]에 대한 배경 조사 결과입니다:
[Perplexity 결과 붙여넣기]

이 데이터를 바탕으로:
1. 분석 프레임워크 설계 (어떤 관점에서 분석할 것인지)
2. 가설 3가지 수립
3. 데이터 분석 계획 (어떤 데이터를, 어떤 방법으로)
4. 예상 결론 시나리오 3가지
```

---

**Step 3: Claude로 논리 검증**

ChatGPT의 분석 프레임워크를 Claude에 입력합니다

```
다음 프로젝트의 분석 프레임워크를 검토해주세요:
[ChatGPT 결과 붙여넣기]

검토 항목:
1. 논리적 허점이 있는가?
2. 누락된 관점이 있는가?
3. 가설의 검증 가능성은?
4. 더 강화할 수 있는 부분은?
```

---

**Step 4: Gamma로 발표 자료 초안**

검증된 내용으로 Gamma에서 발표 자료를 생성합니다

```
[프로젝트 주제]: [한 줄 요약]

슬라이드 구성:
1. 표지
2. 연구 배경 및 목적
3. 분석 프레임워크
4. 데이터 및 방법론
5. 분석 결과 (예상)
6. 시사점 및 제언
7. 참고문헌
```

📍 **화면 가이드:** Gamma에서 프롬프트를 입력하면 60초 내에 슬라이드가 자동 생성됩니다.

✅ **체크포인트:** Perplexity(리서치) → ChatGPT(분석) → Claude(검증) → Gamma(발표) 파이프라인을 완주했나요?

## 4. 프롬프트 진화 패턴

**🔴 Level 1 (초보):**
```
마케팅 수업 프로젝트 주제 추천해줘
```
→ 어떤 과목, 몇 주, 몇 명인지 모르므로 너무 일반적인 주제가 나옵니다.

**🟡 Level 2 (중급):**
```
경영학과 마케팅원론 수업에서 AI를 활용한 프로젝트를 하려고 해.
데이터 분석이 포함된 주제 3개와 각각의 진행 방법을 알려줘.
```
→ 전공에 맞춘 주제가 나오지만, 구체적 도구와 일정이 빠져 있습니다.

**🟢 Level 3 (고급):**
```
경영학과 4학년 마케팅전략 수업 학기 프로젝트:
■ 조건: 4주, 4인 팀, AI 필수 활용, 발표 15분 + 보고서 20페이지
■ 도구: Perplexity(시장조사) + ChatGPT(분석) + Claude(시각화) + Gamma(발표)
■ 요청:
1. 주제 3개 (MZ세대/ESG/AI 관련) + 차별화 포인트
2. 각 주제별 4주 타임라인 (주차별 마일스톤)
3. 팀원 역할 분담 제안
4. 데이터 소스 (한국 공공데이터/기업 보고서)
5. AI 활용 부분 vs 직접 작업 부분 구분
```
→ 조건, 도구, 일정이 모두 명시되어 바로 실행 가능한 기획안이 나옵니다.

## 5. 실전 사례

### 사례 1: AI-PBL이 일반 PBL보다 효과적 (A급)

MDPI(2025)에 게재된 연구에 따르면, AI 통합 프로젝트 기반 학습(AI-PBL)은 일반 PBL보다 **유의미하게 높은 학습 효과**를 보였습니다. AI가 실시간 피드백과 맞춤형 콘텐츠를 제공하여 학업 불안을 감소시키고 성과를 향상시켰습니다.

### 사례 2: Harvard — AI 튜터로 2배 학습 (A급)

Harvard 물리학 수업에서 AI 튜터를 사용한 학생은 전통 수업 대비 **2배 이상 학습**했으며, 더 짧은 시간이 걸렸습니다. 2026년 기준 대학생 **86%**가 AI를 주요 연구·브레인스토밍 파트너로 활용하고 있습니다.

## 6. 꿀팁 & 주의사항

### 💡 꿀팁 5가지

1. **1도구 1역할 원칙:** Perplexity로 조사, ChatGPT로 분석, Claude로 검증, Gamma로 발표 — 각 도구의 최강 기능만 쓰세요
2. **프로젝트 과정을 문서화하라:** AI와의 대화 기록, 의사결정 과정을 기록하면 보고서에 "연구 방법론" 섹션이 됩니다
3. **교수에게 AI 활용 범위를 미리 확인하라:** 수업마다 AI 허용 범위가 다릅니다. 선제적으로 확인하면 불필요한 불이익을 피할 수 있습니다
4. **AI가 못하는 부분을 직접 하라:** AI가 해준 부분보다 직접 한 부분이 학점과 면접에서 가치가 있습니다
5. **팀 프로젝트에서 AI 역할을 명시하라:** "팀원 A: 데이터 수집, 팀원 B: AI 분석 관리, 팀원 C: 시각화, 팀원 D: 보고서" 식으로 AI를 팀원처럼 역할 분담하세요

### ⚠️ Anti-Pattern 1: "코드는 돌아가지만 이해는 못한다" — AI 프로젝트의 도구 의존 함정

**상황:** 대학생 I가 AI 도구로 멋진 데이터 분석 프로젝트를 완성. 차트도 예쁘고 결론도 깔끔.

**잘못된 방법:** AI가 생성한 코드, 분석 결과, 결론을 이해 없이 그대로 사용.

**결과:**
- 교수: "이 회귀분석에서 R-squared가 0.95인데, 과적합 가능성은 검토했나요?"
- 학생 I: "......" (AI가 만든 분석이라 과적합이 뭔지도 모름)
- 교수: "결과는 인상적인데, 직접 한 것 같지 않네요. C학점."

**올바른 방법:**
1. AI 생성 코드를 **한 줄씩 이해** — "이 코드가 뭘 하는지 설명해줘"를 AI에게 요청
2. 분석 결과의 **의미를 직접 해석** — AI에게 "이 결과가 틀릴 수 있는 경우를 알려줘"
3. 프로젝트 과정을 **문서화** — AI와의 대화 기록, 의사결정 근거 포함
4. **"AI가 못하는 부분"을 직접 해결** — 이것이 핵심 학습

**차이:** AI는 결과를 만들어주지만, **결과를 이해하고 설명하는 능력**은 학생 본인의 것이어야 합니다. 발표와 면접에서 이 차이가 드러납니다.

---

### ⚠️ Anti-Pattern 2: "AI가 추천한 기능을 다 넣었더니 아무것도 못 끝냈다" — 범위 초과 실패

**상황:** 대학생 J가 AI로 졸업 프로젝트를 기획. ChatGPT가 추천한 기능: 식단 분석 + 운동 추천 + 수면 추적 + 감정 분석 + 커뮤니티.

**잘못된 방법:** AI가 제안한 모든 기능을 4개월 프로젝트에 포함.

**결과:**
- 5개 기능 중 어느 것도 완성하지 못함
- 교수: "아이디어는 좋은데, 핵심 기능 하나도 작동을 안 하네요."
- 4개월을 투자했지만 결과물 없음

**올바른 방법:**
1. AI에게 "이 기능 중 4개월 안에 혼자서 구현 가능한 핵심 1~2개를 골라줘"
2. **MoSCoW 기법**으로 Must/Should/Could/Won't 분류
3. **MVP(최소 기능 제품)** 접근 — 핵심 1가지를 완벽하게 만들기
4. 나머지 기능은 "향후 확장 계획"으로 보고서에 포함

**차이:** AI는 아이디어를 무한히 생성하지만, **범위를 정하는 것은 인간의 판단**입니다. 적은 기능을 완벽하게 구현하는 것이 많은 기능을 미완성하는 것보다 훨씬 좋은 평가를 받습니다.

## 7. 워크플로우 일반화: 전공 AI 프로젝트 파이프라인

```
[Phase 1: 기획] (Week 0~1)
  → ChatGPT로 주제 발상 + 범위 설정 (MVP 접근)
  → 교수에게 AI 활용 범위 확인
  → 팀원 역할 분담 + AI 도구 배정

[Phase 2: 조사] (Week 1~2)
  → Perplexity로 배경 조사 (출처 포함)
  → NotebookLM/Elicit로 문헌 분석
  → 데이터 소스 확보 + 수집

[Phase 3: 분석] (Week 2~3)
  → ChatGPT Code Interpreter로 데이터 분석
  → Claude로 논리 검증
  → Claude Artifacts로 시각화

[Phase 4: 산출물] (Week 3~4)
  → ChatGPT + Claude로 보고서 구조화
  → Gamma로 발표 자료 생성
  → 팀 리허설 + AI 피드백

[전 과정 필수]
  → AI 활용 부분 vs 직접 작업 부분 구분 문서화
  → AI 생성 결과물에 대한 본인의 이해 확보
  → 교수/면접관에게 설명할 수 있는 수준까지
```

이 파이프라인은 전공에 관계없이 모든 프로젝트에 적용할 수 있습니다. 핵심은 **"AI = 생산성 도구, 나 = 이해와 판단의 주체"**라는 역할 분담입니다.

**잘못된 방법 — AI에게 보고서 전체를 맡기기**

```
이 주제로 10페이지짜리 보고서를 처음부터 끝까지 써줘
```

**결과**: AI가 일관된 톤의 보고서를 생성하지만, 본인의 분석과 비판적 사고가 빠져 있습니다. 교수님이 "이건 AI가 쓴 것 같은데?"라고 피드백합니다. 본문 Anti-Pattern 1의 "코드는 돌아가지만 이해는 못한다" 사례와 동일 — R-squared 0.95의 과적합 가능성을 모르면 C학점입니다.

**올바른 방법**: 에이전트 팀이 **"재료"**(조사 결과, 데이터, 시각화)를 제공하고, **"분석과 해석"**은 본인이 직접 작성합니다. reporter는 형식과 구조를 잡아주지만, 핵심 논증은 사람의 것입니다.

**차이**: CLAUDE.md에 "AI는 재료 공급, 분석은 사람" 규칙을 설정하면 이 경계가 명확해집니다.

---

**잘못된 방법 — 데이터 검증 없이 AI 결과 신뢰**

```
이 분야의 시장 규모와 성장률을 알려줘. 그대로 보고서에 넣을 거야.
```

**결과**: AI가 생성한 "2026년 시장 규모 1.2조원"이 실제 데이터와 다를 수 있습니다. 검증 없이 제출하면 학점은 물론 학술 윤리 문제로 이어집니다.

**올바른 방법**: researcher 에이전트가 출처를 명시하도록 하고, builder가 "⚠️ 출처 미확인" 태그를 자동으로 표시합니다. CLAUDE.md에 "모든 데이터에 출처 필수, 미확인 데이터는 경고 태그" 규칙을 설정합니다.

**차이**: 시스템 수준에서 검증되지 않은 데이터가 최종 보고서에 포함되는 것을 방지합니다.

> - 기존 방식: 전공 프로젝트 전 과정 수동 — 주당 10~20시간 × 4주 = 40~80시간
> - 하네스 활용: 하네스 구축 40분 + 에이전트 실행 + 본인 분석·수정
> - 핵심 가치: "작업"에서 해방되어 "분석"과 "사고"에 집중

---

### 🔄 이 워크플로우를 다른 상황에 적용하기

| 이 실습의 사례 | → 이렇게 바꾸면 | 다른 활용 |
|--------------|----------------|----------|
| 전공 과제 보고서 | 졸업논문 초안 | 졸업논문 챕터별 자동화 |
| 사례 분석 프로젝트 | 시장 조사 보고서 | 창업 프로젝트 (Ch19 연계) |
| 데이터 분석 프로젝트 | 공모전 분석 리포트 | 데이터 공모전 참가 |
| 팀 프로젝트 가이드 | 스터디 그룹 관리 | 학습 모임 운영 |

**핵심 패턴**: `브리프(입력) + PM(기획) + 리서처(조사) + 빌더(구현) + 리포터(보고) = 프로젝트 자동화 시스템` — 기획→조사→실행→보고의 구조를 가진 모든 프로젝트에 이 패턴을 적용할 수 있습니다.

### 📊 하이브리드 전략 비교표: 본문 도구 vs CC 하네스

| 프로젝트 단계 | 본문 도구 조합 | CC 하네스 | 최적 전략 |
|-------------|--------------|----------|----------|
| **1. 주제 발상** | ChatGPT로 5개 주제 제안 | PM 에이전트가 브리프 분석 → 주제 추천 | 본문으로 브레인스토밍 → CC로 체계화 |
| **2. 배경 조사** | Perplexity(출처 포함 리서치) | researcher 에이전트(체계적 수집·분류) | Perplexity로 초기 데이터 → CC researcher로 구조화 |
| **3. 분석·구현** | ChatGPT CI(데이터 분석) + Claude(논리 검증) | builder 에이전트(코드·시각화·분석) | 본문 도구로 탐색 → CC builder로 재현 가능한 코드 생성 |
| **4. 보고서 작성** | ChatGPT + Claude로 구조화 | reporter 에이전트(APA 형식 자동) | 본문으로 초안 아이디어 → CC reporter로 형식 완성 |
| **5. 발표 자료** | Gamma(60초 슬라이드 생성) | reporter(마크다운 슬라이드) | **Gamma가 시각적으로 우위** — CC는 내용 구조화에 강점 |
| **6. 팀 프로젝트** | 팀원 수동 역할 분담 | 에이전트 ↔ 팀원 매핑 가이드 | CC 에이전트 구조를 팀원 역할의 참고 모델로 활용 |

**핵심**: 본문의 "Perplexity→ChatGPT→Claude→Gamma" 4도구 파이프라인은 **각 도구를 수동 전환**하는 방식이고, CC 하네스는 **에이전트 파이프라인이 자동 연결**되는 방식입니다. 반복적 프로젝트(매 학기 과제)에서는 CC 하네스가 압도적으로 효율적이고, 단발성 탐색에서는 본문 도구가 빠릅니다. Harvard 물리학 연구의 "AI 튜터로 2배 학습" 결과는 두 방식 모두에서 AI를 **학습 도구**로 활용할 때 극대화됩니다.

---

> **핵심 정리**: 전공 프로젝트에서 Claude Code의 진짜 가치는 **"에이전트 팀의 역할 분담"**입니다. project-manager가 계획을 세우고, researcher가 자료를 모으고, builder가 구현하고, reporter가 정리합니다. 이 구조는 팀 프로젝트의 역할 분담을 그대로 반영하므로, 팀원 각자가 담당 에이전트를 참고하여 작업할 수 있습니다. CLAUDE.md의 **"AI는 재료 공급, 분석은 사람"** 규칙이 핵심 — AI가 아무리 훌륭한 보고서를 생성해도, 본인의 비판적 분석과 고유한 관점이 없으면 학술적 가치가 없습니다.

## 1. 디자인 씽킹 × AI 도구 매핑

### 1.1 디자인 씽킹 5단계별 AI 활용

| 단계 | 목적 | 추천 AI 도구 | 활용법 |
|------|------|------------|-------|
| **1. 공감(Empathize)** | 사용자 이해 | Perplexity, ChatGPT | 사용자 리서치, 인터뷰 질문 생성, 페르소나 |
| **2. 정의(Define)** | 문제 정의 | Claude, ChatGPT | HMW 질문, 문제 재정의 |
| **3. 아이디어(Ideate)** | 해결책 발상 | ChatGPT, Claude, Gemini | SCAMPER, 역발상, 아이디어 매트릭스 |
| **4. 프로토타입(Prototype)** | 시제품 제작 | v0, Lovable, Bolt.new, Replit | 노코드 웹앱/UI 빠른 제작 |
| **5. 테스트(Test)** | 검증 | ChatGPT, Google Forms+Gemini | 테스트 시나리오, 피드백 분석 |

### 1.2 노코드 프로토타입 도구 비교

| 도구 | 가격 | 핵심 기능 | 코딩 필요 | 추천 상황 |
|------|------|----------|----------|----------|
| **v0 (Vercel)** | Free 제한 / 유료 | 자연어→React UI 생성 | ⚠️ 수정 시 도움됨 | **UI 프로토타입 1순위** |
| **Lovable** | Free / Pro $25/월 | 풀스택 앱, Supabase DB 통합 | ❌ 노코드 가능 | **비개발자, DB 필요 앱** |
| **Bolt.new** | Free / ~$20/월 | 빠른 프로토타이핑, 다중 프레임워크 | ⚠️ 편집 가능 | **래피드 프로토타입** |

## 2. 핵심 개념: AI가 창의적 문제 해결에 미치는 영향

### 2.1 연구가 입증한 효과

**ScienceDirect(2024) 실험 연구:**
- ChatGPT가 대학생의 **창의적 문제 해결 성과를 향상**
- **유창성(fluency)과 유연성(flexibility)** 증가
- 단, **참신성(novelty)**은 상대적으로 감소 — AI가 "안전한 답"을 선호하는 경향

**Taylor & Francis(2025) RCT:**
- 비즈니스 전공 학생 대상: ChatGPT 사용 그룹이 **유의미하게 높은 창의적 성과**
- ChatGPT + 개념맵(concept mapping) → **비판적 사고·문제 해결 능력 향상**

### 2.2 SPARK 프레임워크 — 디자인 씽킹 + AI

디자인 씽킹과 ChatGPT를 결합한 **SPARK 5단계 프레임워크**

| 단계 | 의미 | AI 활용 |
|------|------|--------|
| **S**cenario | 현재 상황 파악 | Perplexity로 배경 데이터 수집 |
| **P**roblem | 해결할 문제 정의 | ChatGPT로 HMW 질문 생성 |
| **A**spiration | 목표 설정 | Claude로 목표 구체화 |
| **R**esources | 가용 자원 확인 | 도구·시간·팀원 역량 정리 |
| **K**eys | 핵심 성공 요인 도출 | AI로 우선순위 매트릭스 |

### 2.3 AI 문제 해결의 핵심 한계

**AI는 공감하지 못합니다.** 디자인 씽킹의 출발점은 "사용자의 고통을 이해하는 것"이지만, AI는 데이터 기반 최적화를 할 뿐 **사용자의 감정과 맥락을 진정으로 이해하지 못합니다**.

> **ResearchGate(2023) 논문:** "AI는 아이디어 생성(ideation)과 프로토타입 단계에서 가장 효과적이지만, 공감(empathy) 단계는 인간이 주도해야 한다."

## 3. 따라하기 실습

### 실습 A: AI로 캠퍼스 문제 해결 기획하기 ⭐

> **목표:** 캠퍼스에서 겪는 실제 문제를 디자인 씽킹 + AI로 해결합니다.
> **소요 시간:** 40분
> **준비물:** ChatGPT 계정

---

**Step 1: 문제 발견 + HMW 질문 생성**

```
대학 캠퍼스에서 학생들이 겪는 일상적인 문제 10가지를 나열해줘.
각 문제에 대해:
1. 문제 설명 (1줄)
2. 영향받는 학생 수 (추정)
3. 현재 해결 방법의 한계
```

가장 공감가는 문제 1개를 선택한 후

```
다음 문제 상황을 HMW(How Might We) 질문으로 변환해주세요:

문제 상황: [선택한 문제]
사용자: 대학생
맥락: [문제가 발생하는 구체적 상황]

요구:
1. HMW 질문 5가지 (관점을 달리하여)
2. 각 질문에 대한 해결 아이디어 3가지씩
3. 실현 가능성 × 영향력 매트릭스로 아이디어 평가
4. 상위 3개 아이디어로 프로토타입 방향 제안
```

📍 **화면 가이드:** HMW 질문이 5가지 관점으로 생성되고, 각각에 해결 아이디어가 매핑됩니다.

✅ **체크포인트:** HMW 질문 5개와 아이디어 매트릭스가 완성되었나요? 상위 3개 아이디어가 선정되었나요?

---

**Step 2: AI 브레인스토밍**

상위 아이디어를 선택하고 심화합니다

```
다음 아이디어를 심화해주세요: [선택한 아이디어]

다음 기법으로 각각 확장해주세요:
1. SCAMPER (대체/결합/적용/수정/다른용도/제거/뒤집기)
2. 역브레인스토밍 (이 문제를 더 악화시키려면? → 반대로)
3. 6-3-5 시뮬레이션 (6명의 가상 전문가: 학생/교수/개발자/디자이너/심리학자/행정담당자)

최종적으로 상위 3개 아이디어를 선정해 구체화해주세요:
- 아이디어명
- 핵심 기능 3가지
- 타겟 사용자
- 차별화 포인트
```

📍 **화면 가이드:** SCAMPER, 역발상, 6-3-5 기법이 각각 적용된 아이디어가 나옵니다.

✅ **체크포인트:** 3가지 기법으로 확장된 아이디어가 있나요? 상위 3개가 구체화되었나요?

🔧 **트러블슈팅:**
- 아이디어가 너무 비현실적이다면 → "대학생 2~3명이 4주 안에 프로토타입을 만들 수 있는 범위로 축소해줘"
- 모든 아이디어가 비슷하다면 → "완전히 다른 분야(게임, 소셜미디어, 오프라인)의 관점에서 해결 방법을 제안해줘"

### 실습 B: 노코드 프로토타입 만들기 ⭐⭐

> **목표:** 아이디어를 노코드 도구로 실제 프로토타입으로 구현합니다.
> **소요 시간:** 60분
> **준비물:** v0 또는 Lovable 계정

---

**Step 1: MVP 기능 정의**

```
다음 서비스의 MVP 기능을 정의해주세요:

서비스: [선택한 아이디어 설명]
타겟: 대학생
목표: [검증하고 싶은 가설]

MoSCoW 기법으로 분류:
- Must Have: MVP에 반드시 포함 (핵심 가치 검증)
- Should Have: 가능하면 포함
- Could Have: 시간 여유 시
- Won't Have: V2에서 개발

각 기능에:
- 기능명 + 1줄 설명
- 유저 스토리
- 예상 개발 시간 (노코드 기준)
```

📍 **화면 가이드:** Must Have 기능만으로 프로토타입을 만듭니다. Should/Could/Won't는 무시하세요.

✅ **체크포인트:** Must Have 기능이 2~3개로 명확히 정의되었나요?

---

**Step 2: v0로 UI 프로토타입 생성**

v0(v0.dev)에 접속하여

```
[서비스명] 의 메인 페이지를 만들어줘:

기능:
1. [Must Have 기능 1]
2. [Must Have 기능 2]

디자인: 깔끔하고 모던, 모바일 반응형
색상: [선호하는 컬러 스킴]
React + TypeScript + shadcn/ui 기반
```

📍 **화면 가이드:** v0가 React 코드와 함께 프리뷰를 생성합니다. 수정이 필요하면 자연어로 추가 요청하세요.

✅ **체크포인트:** 프로토타입 프리뷰가 작동하나요? Must Have 기능이 UI에 반영되었나요?

---

**Step 3: 테스트 시나리오 생성**

```
다음 프로토타입의 사용자 테스트 시나리오를 만들어줘:

프로토타입: [서비스 설명]
타겟: 대학생 5명

5가지 테스트 시나리오:
1. [시나리오: 사용자가 어떤 상황에서 어떤 행동을 하는지]
2. 각 시나리오의 성공 기준
3. 관찰할 포인트
4. 피드백 질문 3가지
```

✅ **체크포인트:** 5가지 테스트 시나리오와 성공 기준이 완성되었나요?

🔧 **트러블슈팅:**
- v0가 원하는 UI를 못 만든다면 → Lovable로 전환하여 "자연어로 전체 앱을 만들어줘" 요청
- 프로토타입이 너무 복잡해진다면 → Must Have 1개 기능만 남기고 나머지 삭제

### 실습 C: 4개 도구로 아이디어 확장 비교 ⭐⭐

## 5. 실전 사례

### 사례 1: ChatGPT가 창의적 문제 해결 능력을 향상시킨다 (A급)

ScienceDirect(2024) 실험 연구에 따르면, ChatGPT를 사용한 학생은 창의적 문제 해결에서 **유창성(fluency)과 유연성(flexibility)**이 증가했습니다. 학생들은 AI 사용 시 **호기심과 침착함**을 가장 많이 느꼈습니다. 다만 참신성(novelty)은 상대적으로 감소하여, AI의 "안전한 답" 선호 경향에 주의가 필요합니다.

### 사례 2: ChatGPT + 개념맵 — 비판적 사고 향상 (A급)

Taylor & Francis(2025) 연구: ChatGPT 지원 개념맵(concept mapping) 접근법으로 학생의 **학업 성취도, 비판적 사고, 문제 해결 능력**이 모두 유의미하게 향상되었습니다.

### 사례 3: 비즈니스 문제 해결 — RCT로 입증 (A급)

Taylor & Francis(2025) RCT: 비즈니스 전공 학생 대상 실험에서 ChatGPT 사용 그룹이 **유의미하게 높은 창의적 성과**를 보였습니다. 실제 비즈니스 문제 해결 과제에서 검증된 결과입니다.

### 사례 4: AI Lean Canvas — 85% 더 빠른 의사결정 (B급)

AI 기반 BMC 도구를 사용한 스타트업의 **85%**가 더 빠른 의사결정을 보고했습니다. F/MS BMC Tool은 2개 질문만으로 AI가 맞춤 Lean Canvas + PDF + Google Sheet를 생성합니다.

## 6. 꿀팁 & 주의사항

### 💡 꿀팁 5가지

1. **문제 정의는 현장에서:** AI에게 문제를 정의해달라고 하지 말고, 직접 관찰·인터뷰한 후 AI에게 구조화를 요청하세요
2. **MoSCoW로 범위를 잡아라:** AI가 아무리 많은 기능을 제안해도, Must Have 2~3개만 프로토타입에 포함하세요
3. **v0는 UI 프로토타입 최강:** 자연어 한 문장으로 React UI를 생성하고, 수정도 자연어로 가능합니다
4. **Lovable은 비개발자 최강:** DB가 필요한 앱을 코딩 없이 만들 수 있습니다. Supabase가 자동 연결됩니다
5. **테스트는 AI가 아닌 실제 사람에게:** AI에게 "이 아이디어 어때?"라고 묻지 말고, 실제 사용자 5명에게 프로토타입을 보여주세요

### ⚠️ Anti-Pattern 1: "AI는 공감하지 못한다" — 문제 해결의 출발점은 현장

**상황:** 팀 프로젝트에서 ChatGPT에게 "대학생이 겪는 문제 10가지"를 물어봄. AI가 추천한 "캠퍼스 식당 혼잡" 문제를 선택.

**잘못된 방법:** 현장 관찰 없이 AI가 정의한 문제를 그대로 수용. AI가 "평균 대기시간 20분"이라고 했으므로 그것을 팩트로 사용.

**결과:**
- 실제 관찰: 식당마다 혼잡 패턴이 완전히 다름 (A식당 12분, B식당 35분)
- AI가 놓친 것: 학생들은 "대기시간"보다 "메뉴 정보 부족"을 더 큰 불편으로 느낌
- 잘못된 문제를 풀었기 때문에, 솔루션이 사용자에게 외면받음

**올바른 방법:**
1. **먼저 현장 관찰** — 직접 식당에 가서 대기 패턴, 학생 행동, 불편 사항 관찰
2. **인터뷰 5명 이상** — "가장 불편한 점이 뭐예요?" 직접 질문
3. 관찰·인터뷰 데이터를 **AI에게 입력** → 패턴 분석 + HMW 질문 생성
4. AI는 **데이터 분석과 아이디어 확장**에 활용, 문제 정의는 현장에서

**차이:** AI는 데이터 기반 최적화를 하지만, **사용자의 진짜 고통**은 현장에서만 발견됩니다. 공감 없는 솔루션은 기술적으로 완벽해도 외면받습니다.

---

### ⚠️ Anti-Pattern 2: "완벽한 AI 계획서, 2주차부터 지연" — 이상 vs 현실

**상황:** ChatGPT로 12주 프로젝트 계획서를 만듦. 역할 분담, 마일스톤, 리스크 관리까지 포함된 "완벽한" 계획.

**잘못된 방법:** AI가 만든 계획을 팀원들의 실제 역량과 가용 시간을 고려하지 않고 그대로 실행.

**결과:**
- AI 계획: "Week 2에 데이터 수집 완료" → 실제: 설문 응답이 30명밖에 안 모임
- AI 계획: "Week 4에 분석 완료" → 실제: 팀원 2명이 중간고사 기간과 겹침
- 2주차부터 계획 대비 지연 → 팀 사기 저하 → 프로젝트 품질 하락

**올바른 방법:**
1. AI 계획을 **초안**으로만 활용
2. 팀원들과 함께 **현실적 일정으로 조정** — 각자의 시간표, 중간고사 등 고려
3. **버퍼 시간** 포함 (계획의 70%만 채워넣기)
4. 매주 **체크포인트**에서 일정 재조정

**차이:** AI는 이상적 조건에서 계획을 세우지만, 팀의 현실(역량, 시간, 동기)은 인간이 조정해야 합니다.

## 7. 워크플로우 일반화: AI 문제 해결 파이프라인

```
[Phase 1: 공감] — 인간 주도
  → 현장 관찰 + 인터뷰 (최소 5명)
  → 관찰 데이터를 ChatGPT에 입력 → 패턴 분석
  → 페르소나 생성

[Phase 2: 정의] — 인간 + AI 협업
  → 관찰 데이터 기반 HMW 질문 생성 (ChatGPT)
  → 문제 재정의 (Claude로 논리 검증)
  → SPARK 프레임워크 적용

[Phase 3: 아이디어] — AI 주도
  → SCAMPER + 역발상 + 6-3-5 (ChatGPT)
  → 실현가능성 × 영향력 매트릭스
  → 상위 3개 아이디어 선정

[Phase 4: 프로토타입] — AI 도구 활용
  → MVP 기능 정의 (MoSCoW)
  → v0/Lovable/Bolt.new로 노코드 프로토타입
  → Must Have 기능만 구현

[Phase 5: 테스트] — 인간 주도
  → 실제 사용자 5명에게 프로토타입 테스트
  → Google Forms로 피드백 수집
  → ChatGPT로 피드백 분석 → 개선 방향
```

핵심 원칙: **공감과 테스트는 인간이, 아이디어와 프로토타입은 AI가** 주도합니다.

# Problem Analyst — 문제 분석가

## 핵심 역할
- 문제를 MECE(Mutually Exclusive, Collectively Exhaustive) 원칙으로 분해합니다.
- 5 Whys 기법으로 근본 원인을 탐색합니다.
- 이해관계자 분석과 영향 범위를 정리합니다.

## 분석 도구
- 5 Whys: "왜?" 5번 반복으로 근본 원인 도달
- MECE 분해: 중복 없이, 누락 없이 문제 요소 분류
- 피시본(Ishikawa) 다이어그램: 원인을 카테고리별로 정리

**잘못된 방법 — 분석 없이 바로 해결책 요청**

```
참여율을 높일 방법을 알려줘. 분석은 필요 없어, 답만 줘.
```

**결과**: 근본 원인을 모르니 증상만 치료하는 해결책이 나옵니다. "홍보를 더 하자"라고 했는데, 실제 원인이 "모임 시간이 수업과 겹침"이었다면 홍보는 무의미합니다. 본문 Anti-Pattern 1 "AI는 공감하지 못한다"와 동일 — 현장 관찰 없이 AI가 정의한 "평균 대기시간 20분"을 팩트로 수용했더니, 실제로는 식당마다 패턴이 달랐고 학생들은 "대기시간"보다 "메뉴 정보 부족"을 더 큰 불편으로 느꼈습니다.

**올바른 방법**: problem-analyst가 5 Whys로 근본 원인을 파악한 후, 그 원인에 맞는 솔루션을 설계합니다.

**차이**: CLAUDE.md의 **"분석 없는 해결책 금지"** 규칙이 성급한 결론을 방지합니다.

---

**잘못된 방법 — AI가 추천한 1순위 솔루션만 실행**

```
1순위 솔루션만 알려줘. 나머지는 필요 없어.
```

**결과**: 1순위가 실행 불가능하거나 예상치 못한 부작용이 있을 때 대안이 없습니다.

**올바른 방법**: evaluator가 최소 3개 솔루션을 비교 평가하고, 최적안 + 차선안(Plan B)을 함께 제시합니다.

**차이**: 항상 대안이 준비되어 있어야 실전에서 유연하게 대응할 수 있습니다.

새로운 문제를 해결할 때 사용할 스크립트를 만들어줘.

사용법: bash scripts/solve_problem.sh [문제명]

동작:
1. input/problem_statement.md 확인
2. 3단계 순차 실행 (분석→설계→평가)
3. 결과물 output/에 정리
4. 요약 출력
```

✅ **체크**: 스크립트로 전 과정이 자동 실행되면 성공

---

#### Step 4. 디자인 씽킹 워크숍 시뮬레이션 ⏱️ 5분

```
디자인 씽킹 5단계를 CC 에이전트로 시뮬레이션해줘.

1. 공감(Empathize): 사용자 인터뷰 시나리오 생성
2. 정의(Define): 문제 재정의 (HMW)
3. 아이디에이션(Ideate): 솔루션 브레인스토밍
4. 프로토타입(Prototype): 간단한 프로토타입
5. 테스트(Test): 사용자 피드백 시나리오

각 단계를 에이전트가 수행.
```

✅ **체크**: 디자인 씽킹 5단계가 시뮬레이션되면 성공

---

#### 🎉 완성! — 결과물 미리보기

분석 → 솔루션 → 평가 → 실행 → 디자인 씽킹까지 **체계적 문제 해결 시스템**을 구축했습니다.

> 📊 **효과**
> - 기존 방식: 팀 회의에서 "아이디어 나열" → 가장 인기 있는 안 선택 — 근거 없음
> - 하네스 활용: 구조화된 분석 + 다수 솔루션 + 객관적 평가 — 논리적이고 투명한 의사결정
> - 핵심 가치: "좋은 해결책"보다 "좋은 문제 정의"가 먼저

### 🔄 이 워크플로우를 다른 상황에 적용하기

| 이 실습의 사례 | → 이렇게 바꾸면 | 다른 활용 |
|--------------|----------------|----------|
| 캠퍼스 문제 해결 | 기업 과제 해결 | 사례 분석 과제 |
| 5 Whys 분석 | 버그 원인 분석 | 소프트웨어 디버깅 |

**핵심 패턴**: `문제 정의(입력) + 구조적 분석(에이전트) + 다수 솔루션(에이전트) + 객관적 평가(에이전트) = 체계적 문제 해결 시스템` — 원인 분석 → 해결책 설계 → 평가의 구조를 가진 모든 의사결정에 이 패턴을 적용할 수 있습니다.

### 📊 하이브리드 전략 비교표: 본문 도구 vs CC 하네스

| 문제 해결 단계 | 본문 도구 조합 | CC 하네스 | 최적 전략 |
|--------------|--------------|----------|----------|
| **1. 공감(Empathize)** | 현장 관찰 + Perplexity 리서치 + ChatGPT 페르소나 | problem-analyst(인터뷰 시나리오 생성) | **현장 관찰은 인간 필수** → CC로 관찰 데이터 구조화 |
| **2. 정의(Define)** | ChatGPT로 HMW 질문 + Claude 논리 검증 | problem-analyst(5 Whys + MECE + fishbone) | 본문 HMW로 시작 → CC로 근본 원인 심층 분석 |
| **3. 아이디어(Ideate)** | ChatGPT SCAMPER + 역발상 + 6-3-5 시뮬레이션 | solution-designer(브레인스토밍 + SCAMPER + 제약 기반) | 본문 기법으로 발산 → CC로 3개 솔루션 구조화 |
| **4. 프로토타입** | **v0/Lovable/Bolt.new(시각적 UI 최강)** | CC 에이전트(마크다운 와이어프레임·코드 MVP) | **노코드 UI는 본문 도구 우위** — CC는 프로세스·데이터 프로토타입에 강점 |
| **5. 테스트** | Google Forms + ChatGPT 피드백 분석 | evaluator(평가 매트릭스 + KPI + Plan B) | 본문으로 실사용자 피드백 수집 → CC evaluator로 객관적 평가 |
| **6. 반복(Iterate)** | 새 대화에서 재시작 | 스크립트로 즉시 재실행 | CC의 **재현성**이 압도적 — 피드백 반영 후 전체 파이프라인 재실행 |

**핵심**: 본문의 디자인 씽킹 5단계와 SPARK 프레임워크는 **"무엇을 할 것인가"**를 정의하고, CC 에이전트는 **"어떻게 체계적으로 실행할 것인가"**를 자동화합니다. ScienceDirect(2024) 연구가 지적한 AI의 "참신성 감소" 문제는 solution-designer의 SCAMPER·역발상 기법이 보완하고, ResearchGate(2023) 논문이 강조한 "공감 단계는 인간 주도" 원칙은 CLAUDE.md 규칙으로 강제합니다.

---

> **핵심 정리**: 문제 해결에서 Claude Code의 진짜 가치는 **"분석과 해결을 분리"**하는 것입니다. problem-analyst가 근본 원인을 파악하고, solution-designer가 다수 해결책을 제안하고, evaluator가 객관적으로 평가합니다. 이 3단계 분리가 "감에 의존한 해결"을 "구조화된 해결"로 전환합니다. CLAUDE.md의 **"분석 없는 해결책 금지"** 규칙이 핵심 — 문제를 제대로 정의하면, 해결책의 절반은 이미 나온 것입니다.

## 2. 핵심 개념 이해

### 2.1 린 캔버스(Lean Canvas)란?

린 캔버스는 에릭 리스(Eric Ries)의 린 스타트업 방법론을 Ash Maurya가 1페이지 프레임워크로 정리한 것입니다. 기존 사업계획서가 수십 페이지를 요구하는 것과 달리, **9개 블록에 핵심만 담습니다.**

| 블록 | 질문 | 핵심 |
|------|------|------|
| 1. 문제 (Problem) | 고객의 상위 3가지 문제는? | 가장 아픈 곳 |
| 2. 고객 세그먼트 | 누가 이 문제를 가지고 있는가? | 얼리어답터 특정 |
| 3. 고유 가치 제안 | 왜 이 솔루션을 써야 하는가? | 한 줄 차별점 |
| 4. 솔루션 | 각 문제를 어떻게 해결하는가? | 핵심 기능 3개 |
| 5. 채널 | 고객에게 어떻게 도달하는가? | 마케팅 경로 |
| 6. 수익 모델 | 어떻게 돈을 버는가? | 과금 방식 |
| 7. 비용 구조 | 무엇에 돈이 드는가? | 고정/변동 비용 |
| 8. 핵심 지표 | 성공을 어떻게 측정하는가? | KPI 3개 |
| 9. 경쟁 우위 | 쉽게 따라할 수 없는 것은? | 해자(moat) |

### 2.2 MVP(Minimum Viable Product)란?

MVP는 **핵심 가설을 검증할 수 있는 최소 기능의 제품**입니다. "완벽한 제품"이 아니라 "학습을 위한 제품"입니다.

**MoSCoW 기법**으로 기능의 우선순위를 정합니다

- **M**ust Have: MVP에 반드시 포함 (핵심 가치 검증용)
- **S**hould Have: 가능하면 포함
- **C**ould Have: 시간 여유가 있으면
- **W**on't Have: V2에서 개발

### 2.3 피치덱(Pitch Deck)이란?

투자자나 심사위원에게 사업을 설명하는 **10~15장짜리 프레젠테이션**입니다. 구조는 보통 다음 10장을 따릅니다

1. 표지 (서비스명 + 한줄 소개)
2. 문제 (고객의 핵심 페인포인트)
3. 솔루션 (우리의 해결 방법)
4. 시장 규모 (TAM/SAM/SOM)
5. 비즈니스 모델 (수익 구조)
6. 경쟁 우위 (차별화 포인트)
7. 제품 데모/목업 (핵심 화면)
8. 팀 소개 (왜 우리 팀이 해야 하는지)
9. 재무 계획 (3년 매출 예측)
10. Ask (투자 요청 금액 / 대회 목표)

## 3. 따라하기 실습: AI로 4주 창업 기획 완성하기

> **실습 시나리오:** "대학생 대상 AI 학습 도우미 앱"을 기획합니다. 시장 조사부터 피칭까지 AI 도구만으로 전 과정을 진행합니다.

### 실습 A: Perplexity로 시장 조사하기 ⭐

**사용 도구:** Perplexity (perplexity.ai)

**Step 1:** Perplexity에 접속하여 로그인합니다.

**Step 2:** 다음 프롬프트를 입력합니다

```
"대학생 AI 학습 도우미" 서비스의 시장성을 분석해주세요:

1. TAM/SAM/SOM 시장 규모 추정 (한국 기준)
2. 시장 성장률 및 트렌드 (최근 3년)
3. 주요 경쟁사 5곳 비교표 (제품, 가격, 강점, 약점)
4. 타겟 고객 세그먼트 및 규모
5. 진입 장벽 및 기회 요인

모든 데이터에 출처를 포함해주세요.
```

**Step 3:** Perplexity의 답변에서 **출처가 포함된 데이터**를 확인합니다.

✅ **체크포인트:** TAM/SAM/SOM 수치와 출처가 포함되어 있나요? 경쟁사 비교표(5곳 이상)와 시장 성장률 수치가 출력에 있나요?

**Step 4:** 후속 질문으로 깊이를 더합니다

```
위 분석에서 한국 대학생 시장에 집중해서 추가 분석해주세요:
- 한국 대학생 약 300만 명 중 AI 학습 도구 사용 비율
- 월 지불 의향(WTP) 추정
- 학기 중 vs 방학 중 사용 패턴 차이
```

🔧 **트러블슈팅:** Perplexity가 한국 데이터를 충분히 제공하지 않으면, "한국어로 검색해서" 또는 "site:kr" 키워드를 추가로 안내해보세요.

> **왜 Perplexity인가?** 시장 조사에서 가장 중요한 것은 **데이터의 출처**입니다. ChatGPT나 Claude는 출처를 생성(hallucination)할 수 있지만, Perplexity는 실시간 검색 결과에 **인라인 출처(Citation)**를 제공하므로 신뢰도가 높습니다.

### 실습 B: ChatGPT로 린 캔버스 작성하기 ⭐⭐

다음 사업 아이디어로 린 캔버스(Lean Canvas)를 작성해주세요:

사업 아이디어: 대학생 AI 학습 도우미 — 강의 노트를 업로드하면

핵심 문제: 학습 자료 정리에 시간이 너무 많이 걸림

린 캔버스 9개 블록 모두 채워주세요:
1. 문제 (Problem) — 상위 3가지 문제

6. 수익 모델 (Revenue Streams)
7. 비용 구조 (Cost Structure)
8. 핵심 지표 (Key Metrics)
9. 경쟁 우위 (Unfair Advantage)

각 블록에 구체적 근거/수치를 포함하고,
표 형태로 한눈에 볼 수 있게 정리해주세요.
```

**Step 2:** 결과를 복사해 **Notion이나 Google Docs**에 붙여넣습니다.

**Step 3:** Claude에 같은 린 캔버스를 요청하여 **교차 검증**합니다

```
[ChatGPT가 생성한 린 캔버스를 붙여넣고]

위 린 캔버스를 비판적으로 검토해주세요:
1. 각 블록에서 약한 점이나 누락된 부분
2. 가설(Hypothesis)이 명확하지 않은 블록
3. 검증 우선순위 Top 3
4. MVP로 검증 가능한 실험 3가지 제안
```

✅ **체크포인트:** 9개 블록이 모두 채워져 있나요? 수익 모델에 구체적 가격/과금 방식이 포함되어 있나요? 경쟁 우위가 "AI 기술" 같은 모호한 표현이 아닌, 구체적인가요?

> **핵심 팁:** AI가 생성한 린 캔버스는 **초안**입니다. 가장 중요한 작업은 "이 가설이 맞는가?"를 실제 사용자에게 물어보는 것입니다. AI는 구조를 잡아주지만, **검증은 사람이 합니다.**

## 4. 프롬프트 진화 패턴

### 창업 아이디어 프롬프트의 진화

**🔴 Level 1 — 단순 요청:**

```
대학생 창업 아이디어 추천해줘
```

→ 일반적이고 누구에게나 해당되는 답변. 실행 불가능.

**🟡 Level 2 — 맥락 추가:**

```
대학생 대상 AI 기반 서비스 창업 아이디어 5개와
각각의 비즈니스 모델을 제안해줘.
```

→ 분야(AI)와 산출물(BM)을 지정했으나, 구체적 제약 조건이 없음.

**🟢 Level 3 — 완전한 맥락 + 제약 조건:**

```
대학생 창업 경진대회(예비창업패키지) 준비:
■ 분야: EdTech (AI 학습 도우미)
■ 요청:
1. 린 캔버스 9블록 (구체적 수치 포함)
2. TAM/SAM/SOM (한국 대학생 600만명 기준)
3. 경쟁사 5곳 포지셔닝 맵 (기능 × 가격)
4. MVP 기능 MoSCoW 분류 (Lovable 개발 기준)
5. 3년 재무 모델 (MAU 시나리오별)
6. 피치덱 10장 구조 + 슬라이드별 스크립트
■ 대회: 발표 10분 + Q&A 5분, 심사위원 = VC + 교수
```

→ 대회명, 분야, 도구, 청중까지 모두 지정. AI가 맞춤형 결과를 생성.

## 5. 도구 비교 실험: 같은 프롬프트, 4개 도구

> **실험 프롬프트:** "대학생 대상 AI 학습 도우미 서비스의 린 캔버스를 작성해줘. 9개 블록을 모두 채우고, 각 블록에 구체적 수치를 포함해줘."

| 평가 항목 | ChatGPT | Claude | Gemini | Perplexity |
|----------|---------|--------|--------|------------|
| **구조 완성도** | ★★★★★ | ★★★★★ | ★★★★☆ | ★★★☆☆ |
| **수치/데이터** | ★★★★☆ | ★★★★★ | ★★★★☆ | ★★★★★ |
| **비판적 관점** | ★★★☆☆ | ★★★★★ | ★★★☆☆ | ★★★☆☆ |
| **출처 포함** | ★★☆☆☆ | ★★☆☆☆ | ★★★☆☆ | ★★★★★ |
| **실행 가능성** | ★★★★★ | ★★★★☆ | ★★★★☆ | ★★★☆☆ |

**도구별 특징:**

- **ChatGPT:** 가장 완성도 높은 캔버스. 빠르게 구조를 잡고 실행 가능한 수준으로 제시. 다만 수치의 출처가 불명확할 수 있음.
- **Claude:** 가설을 비판적으로 분석. "이 가정이 약합니다"라고 직접 지적. BM 설계에서 논리 검증에 최적.
- **Gemini:** Google 검색 연동으로 최신 데이터 반영. 다만 캔버스 포맷이 다소 느슨할 수 있음.
- **Perplexity:** 출처가 가장 풍부. 시장 규모 등 **데이터 검증**에 최적이나, 캔버스 구조 완성도는 낮음.

**최적 조합:** Perplexity(시장 데이터) → ChatGPT(캔버스 초안) → Claude(비판적 검증)

## 6. 실전 사례: AI 창업은 이미 일어나고 있다

### 사례 1: Y Combinator — 코드의 95%가 AI 생성

Y Combinator 2025년 Winter 배치에서 **스타트업의 25%가 코드베이스의 95% 이상을 AI로 생성**했습니다. 이 스타트업들의 공통점은

- 기술이 아닌 **문제**에서 출발함
- MVP를 **2~4주 내**에 구축함
- 빠른 반복(iteration)으로 **제품-시장 적합성(PMF)**을 검증함
- 수직형(vertical) AI — 범용이 아닌 **도메인 특화** 솔루션

2026년 핵심 트렌드는 **수직형 AI**입니다. 범용 챗봇을 만드는 것이 아니라, 특정 도메인(의료, 법률, 교육, 물류 등)의 문제를 AI로 해결하는 서비스가 투자를 받고 있습니다.

## 7. 꿀팁 & 주의사항

### 꿀팁 5선

**Tip 1: 도구 조합의 황금 공식**

```
시장 데이터: Perplexity (출처 포함)
      ↓
BM 설계: ChatGPT (초안) → Claude (검증)
      ↓
MVP: Lovable (비개발자) 또는 Cursor (개발자)
      ↓
피치덱: Gamma (60초 생성)
      ↓
발표 준비: ChatGPT (스크립트 + Q&A)
```

**Tip 2: MVP는 "최소"에 집중하라**

기능을 5개 이하로 줄이세요. Lovable이나 Bolt.new는 기능이 많아질수록 에러가 급증합니다. Must Have만 구현하고 나머지는 V2로 미루세요.

**Tip 3: 린 캔버스는 1주일마다 업데이트하라**

AI가 처음 생성한 린 캔버스는 가설(hypothesis)입니다. 매주 실제 피드백을 반영하여 업데이트하세요. 특히 "문제"와 "고객 세그먼트" 블록이 가장 많이 바뀝니다.

**Tip 4: Perplexity 시장 조사에서 PEST 분석을 추가하라**

시장 규모(TAM/SAM/SOM)만으로는 부족합니다. "PEST 분석(정치/경제/사회/기술)"을 함께 요청하면 심사위원이 감탄하는 시장 분석이 됩니다.

**Tip 5: 피치덱은 "문제 슬라이드"가 80%를 결정한다**

심사위원은 첫 2장(표지 + 문제)에서 관심 여부를 결정합니다. 문제가 공감되지 않으면 나머지 8장은 의미가 없습니다. 문제 슬라이드에 가장 많은 시간을 투자하세요.

### ⚠️ Anti-Pattern 1: "기술이 있어서 만든 것 vs 문제가 있어서 만든 것"

**상황:** 팀 K가 "GPT API가 있으니까 뭔가 만들자"로 시작.

**잘못된 방법:** 기술(GPT API)에서 출발 → "이걸로 뭘 할 수 있지?" → 범용 챗봇 제작.

**결과:**
- 차별점 없는 서비스
- 심사위원: "기존 ChatGPT와 뭐가 다른가요?"
- 사용자도 ChatGPT를 직접 쓰면 되므로 전환 이유 없음

**올바른 방법:**
1. 문제에서 출발 → "대학생이 학습 자료 정리에 주 8시간 쓴다"
2. 해결 방법으로 AI를 적용 → 강의 노트 자동 요약 + 퀴즈 생성
3. **수직형(vertical) AI** — 범용이 아닌 도메인 특화

**차이:** "기술이 있어서 만든 것"은 시장이 없고, "문제가 있어서 만든 것"은 시장이 있습니다. Getting Smart(2025)의 조언: **"실제 문제를 해결하라."** Y Combinator 데이터도 동일합니다 — 코드의 95%를 AI로 생성할 수 있어도, **문제 정의와 시장 검증은 여전히 인간의 몫**입니다.

---

### ⚠️ Anti-Pattern 2: "AI API 비용을 간과한 서비스 기획"

**상황:** 팀 J가 ChatGPT API로 "대학생 맞춤 학습 도우미" 프로토타입 완성.

**잘못된 방법:** 무료 서비스로 출시 → 사용자 100명 확보 → API 비용 월 50만 원 발생.

**결과:**
- 수익 모델 없이 운영 → 2개월 만에 서비스 중단
- "사용자가 늘수록 적자가 커지는" 구조

**올바른 방법:**
1. 기획 단계에서 비용 구조를 설계: API 호출당 비용 × 예상 사용량 = 월 비용
2. 과금 모델을 역산하여 Freemium이라도 비용 상한(cap)을 설계
3. 무료 사용자에게는 응답 횟수 제한, Pro 사용자에게만 무제한 제공

**차이:** AI API 비용은 사용량에 비례합니다. 무료 서비스로는 지속 불가능하며, 기획 단계에서 반드시 비용 구조를 설계해야 합니다.

**비용 계산 예시:**
- GPT-4o API: 입력 $2.5/100만 토큰, 출력 $10/100만 토큰
- 사용자 1명이 하루 평균 5,000 토큰 사용 시
- 100명 × 30일 × 5,000 토큰 = 1,500만 토큰/월
- 비용: 약 $15~$150/월 (모델에 따라 다름)
- **반드시 기획 단계에서 계산하세요**

## 8. 워크플로우 일반화: AI 창업 기획의 보편 원칙

지금까지의 실습에서 도출한 창업 기획의 보편 원칙입니다. **어떤 사업 아이디어에든 적용할 수 있습니다.**

### 원칙 1: 문제 → 솔루션 → 기술 순서를 지켜라

```
❌ 기술(AI가 있으니까) → 솔루션(뭔가 만들자) → 문제(누가 쓸까?)
✅ 문제(누구의 어떤 고통?) → 솔루션(어떻게 해결?) → 기술(AI가 최선인가?)
```

### 원칙 2: AI는 속도의 도구이지, 판단의 도구가 아니다

| AI가 해주는 것 | AI가 해줄 수 없는 것 |
|---------------|-------------------|
| 시장 데이터 수집 | 이 시장에 진입할지 결정 |
| 린 캔버스 구조화 | 가설이 맞는지 검증 |
| MVP 코드 생성 | 사용자가 원하는지 확인 |
| 피치덱 디자인 | 청중을 설득하는 열정 |

### 원칙 3: 4주 파이프라인


| 주차 | 활동 | 핵심 도구 | 산출물 |
|------|------|----------|--------|
| **Week 1** | 시장 조사 + 문제 정의 | Perplexity, ChatGPT | 시장 분석 보고서, 페르소나 |
| **Week 2** | BM 설계 + 재무 모델 | ChatGPT, Claude, CI | 린 캔버스, 재무 모델 |
| **Week 3** | MVP 개발 | Lovable / Replit | 작동하는 프로토타입 |
| **Week 4** | 피치덱 + 발표 준비 | Gamma, ChatGPT | 피치덱 10장, 스크립트 |

### 원칙 4: 검증 우선 순위

모든 가설을 한꺼번에 검증할 수 없습니다. 다음 순서로 검증하세요

1. **문제 가설:** "이 문제가 진짜 존재하는가?" → 잠재 사용자 10명 인터뷰
2. **솔루션 가설:** "우리 방법이 이 문제를 해결하는가?" → MVP 사용 테스트
3. **비즈니스 가설:** "돈을 낼 의향이 있는가?" → 가격 실험 (A/B 테스트)

#### 🟡 Level 2 — 중급 프롬프트

```
대학생 대상 AI 학습 도우미 서비스를 기획하려고 해.
시장 규모, 경쟁사, BM을 분석해줘.
```

**결과**: 개괄적 분석이 나오지만 단편적이고, 피치 수준의 설득력은 없습니다.

📋 **평가**
- 정확성: ★★★☆☆
- 활용도: ★★★☆☆
- 개선점: 구체적 아이디어와 요청 항목
- 아쉬운 점: 체계적 분석 없음, 피치덱 없음

---

#### 🟢 Level 3 — 전문가 프롬프트 (하네스 활용)

```
input/idea.md와 input/target_customer.md를 읽고
창업 기획을 시작해줘.

@market-researcher TAM/SAM/SOM + 경쟁사 5개 비교 매트릭스

제약 조건: input/constraints.md 참조.
```

📋 **평가**
- 정확성: ★★★★★
- 활용도: ★★★★★
- 핵심 차이: 체계적 시장 분석 + 린 캔버스 + 투자자 관점 피치덱

### ❌ 흔한 실수: 창업 기획의 함정

**상황**: Claude Code로 창업 기획을 할 때

---

**잘못된 방법 — AI 기획서를 그대로 발표**

```
사업계획서 써줘. 10분 발표에 써야 해.
```

**결과**: AI가 만든 사업계획서는 "논리적으로 완벽"하지만, 심사위원의 질문에 답하지 못합니다. "이 수익 모델의 근거가 뭔가요?"라는 질문에 "AI가 만들어줬는데요..."는 탈락입니다.

**올바른 방법**: 에이전트 팀이 **재료(시장 데이터, BM 프레임워크, 슬라이드 구조)**를 제공하고, **핵심 의사결정(가격, 타겟, 차별화)은 본인**이 합니다. pitch-builder의 스크립트를 참고하되 본인 말로 재구성합니다.

**차이**: CLAUDE.md에 "가설과 검증 사실 구분 표시" 규칙이 있어, 본인이 검증해야 할 부분이 명확히 보입니다.

---

**잘못된 방법 — TAM을 과대 추정**

```
AI 교육 시장이 100조원이니까 우리가 1%만 가져와도 1조원!
```

**결과**: 투자자와 심사위원이 가장 싫어하는 패턴입니다. "전체 시장 × 1%" 공식은 설득력이 전혀 없습니다. 본문 Anti-Pattern 2에서 경고한 **API 비용 간과** 문제도 동일 — 무료 서비스로 100명 확보 후 월 50만 원 API 비용이 발생하여 2개월 만에 서비스가 중단된 사례가 있습니다.

**올바른 방법**: market-researcher가 TAM(전체) → SAM(유효) → SOM(수익)으로 단계적으로 축소하고, 각 단계의 **산출 근거**를 명시합니다. SOM이 현실적이어야 합니다.

**차이**: CLAUDE.md의 "TAM 과대 추정 금지, SOM은 현실적으로" 규칙이 허황된 시장 분석을 방지합니다.

|--------------|----------------|----------|

**핵심 패턴**: `아이디어(입력) + 시장 분석(에이전트) + BM 설계(에이전트) + 피치 제작(에이전트) + 피벗 자동화(스크립트) = 린 스타트업 시스템` — 아이디어에서 발표까지의 구조를 가진 모든 기획에 이 패턴을 적용할 수 있습니다.

### 📊 하이브리드 전략 비교표: 본문 도구 vs CC 하네스

| 창업 기획 단계 | 본문 도구 조합 | CC 하네스 | 최적 전략 |
|--------------|--------------|----------|----------|
| **1. 시장 조사** | **Perplexity(출처 포함 실시간 검색)** | market-researcher(TAM/SAM/SOM + PEST + 5 Forces) | **Perplexity로 최신 데이터 수집** → CC로 프레임워크 분석 체계화 |
| **2. 경쟁 분석** | Perplexity + ChatGPT 비교표 | market-researcher(경쟁사 5개 매트릭스 자동) | 본문으로 경쟁사 탐색 → CC로 구조화된 매트릭스 생성 |
| **3. BM 설계** | ChatGPT(린 캔버스 초안) + Claude(비판적 검증) | bm-designer(9블록 + 수익 모델 + MVP 정의) | 본문 "ChatGPT 초안 → Claude 검증" 패턴과 CC 에이전트 병행 |
| **4. 재무 모델** | ChatGPT CI(매출 예측·손익분기점) | bm-designer(보수적/낙관적 2시나리오) | **ChatGPT CI가 계산에 강점** — CC는 시나리오 구조화에 강점 |
| **5. 피치덱** | **Gamma(60초 시각적 슬라이드 최강)** | pitch-builder(10슬라이드 + 발표 스크립트) | **Gamma로 시각적 디자인** + CC로 내용 전략·스크립트 구성 |
| **6. 피벗** | 새 대화에서 전체 재요청 | `build_pitch.sh` — idea.md만 수정하면 전체 재생성 | CC의 **피벗 자동화**가 압도적 — Build-Measure-Learn 반복에 최적 |

**핵심**: 본문의 "Perplexity(시장 데이터) → ChatGPT(캔버스 초안) → Claude(비판적 검증)" 최적 조합은 **탐색과 검증**에 강하고, CC 하네스는 **자동화와 피벗**에 강합니다. 본문 Tip 5 "피치덱은 문제 슬라이드가 80%를 결정한다"는 CC pitch-builder에서도 핵심 원칙입니다. UC Berkeley TeachShare 사례의 핵심 조언 — "실제 문제를 해결하라, 빠르게 만들고, 일찍 피드백을 받아라" — 이 CC 하네스의 빠른 피벗과 정확히 일치합니다.

## 1. 바이브 코딩 시대가 열리다

### 1.1 바이브 코딩이란?

2025년 초, OpenAI 공동창업자 **Andrej Karpathy**가 "바이브 코딩(Vibe Coding)"이라는 용어를 만들었습니다. 정의는 간단합니다

> **AI에게 자연어로 설명하면 코드를 생성한다 — 코드를 이해하지 않고도 결과물을 만들 수 있다.**

숫자로 보면 이 변화의 규모가 보입니다

| 지표 | 수치 | 출처 |
|------|------|------|
| 미국 개발자 중 AI 코딩 도구 일상 사용 | 92% | Google Cloud, 2025 |
| AI 에이전트가 작성하는 코드량 (인간 대비) | 약 10배 | IBM, 2026 |
| Fortune 500 중 바이브 코딩 플랫폼 도입 | 87% | Wikipedia |
| GitHub Copilot 사용 시 코딩 속도 향상 | 55% | GitHub/arXiv, 2023 |
| Copilot 사용자 수 | 150만+ | GitHub, 2025 |
| DORA Report: 전문 개발자 AI 도구 사용 | 90% 이상 | DORA, 2025 |

하지만 **바이브 코딩에는 그림자**도 있습니다. 이 부분은 Anti-Pattern에서 자세히 다룹니다.

## 2. 핵심 개념 이해

### 2.1 AI 코딩 도구의 3가지 유형

**유형 1: 자동완성형 (코드 어시스턴트)**

기존 IDE에 플러그인으로 설치. 코드를 타이핑하면 AI가 다음 줄을 제안합니다.

- 예: GitHub Copilot, Windsurf
- 장점: 기존 작업 환경 유지
- 적합: 코딩 경험이 있는 학생

**유형 2: AI 네이티브 IDE**

에디터 전체가 AI 중심으로 설계. 코드베이스를 이해하고 대규모 리팩토링 가능.

- 예: Cursor
- 장점: 코드베이스 전체 맥락 이해, 멀티파일 수정
- 적합: 중급 이상 개발자

**유형 3: 노코드 AI 빌더**

자연어로 설명하면 전체 앱을 생성. 코딩 지식 불필요.

- 예: Replit Agent, Lovable, Bolt.new
- 장점: 코딩 경험 없이 앱 제작 가능
- 적합: 비전공자, 아이디어 검증

### 2.2 코딩 에이전트란?

2025~2026년 AI 코딩의 가장 큰 변화는 **코딩 에이전트**의 등장입니다. 기존 AI 코딩 도구가 "질문에 답하는" 수준이었다면, 에이전트는 **자율적으로 코딩 작업을 수행**합니다

- GitHub 이슈에서 자동으로 PR(Pull Request) 생성
- 에러를 감지하고 자동으로 수정
- 테스트를 작성하고 실행하여 검증
- 여러 파일을 동시에 수정

GitHub Copilot, Cursor, Claude Code 모두 에이전트 기능을 탑재하고 있습니다.

## 3. 따라하기 실습: AI로 웹앱 만들기

### 실습 A: Replit으로 할일 관리 앱 만들기 ⭐

**사용 도구:** Replit (replit.com)

**Step 1:** replit.com에 접속하여 회원가입합니다 (Google 계정으로 가능).

**Step 2:** "Create Repl" 버튼을 클릭합니다.

**Step 3:** Agent에 다음 프롬프트를 입력합니다

```
할일 관리 웹앱을 만들어줘.

기능:
1. 할일 추가 (텍스트 입력)
2. 완료 체크 (체크박스)
3. 삭제 버튼
4. 완료된 할일과 미완료 할일 분리 표시
5. 오늘 날짜 자동 표시

디자인:
- 깔끔한 모던 UI
- 모바일 반응형
- 다크모드 지원
```

**Step 4:** Replit Agent가 자동으로
- 프로젝트 구조 생성
- 코드 작성
- 내장 브라우저에서 실시간 미리보기

**Step 5:** 기능을 추가하고 싶다면 채팅에 요청합니다

```
마감일(due date) 기능을 추가해줘.
마감일이 지난 할일은 빨간색으로 표시해줘.
```

**Step 6:** 배포하기 — "Deploy" 버튼 클릭 → URL 생성 → 누구나 접속 가능

✅ **체크포인트:** 할일 추가/삭제가 동작하나요? 완료 체크가 작동하나요? 배포 URL이 생성되었나요?

🔧 **트러블슈팅:**
- **"에러가 발생합니다":** 에러 메시지를 그대로 Replit Agent 채팅에 붙여넣으면 자동 수정
- **"화면이 안 보여요":** 내장 브라우저의 "Refresh" 클릭
- **"느려요":** 무료 플랜은 성능 제한이 있음. 복잡한 앱은 Core 플랜 고려

> **왜 Replit인가?** 설치가 필요 없고, 브라우저에서 바로 시작할 수 있으며, AI Agent가 전체 앱을 자동으로 만들어줍니다. 비전공자가 AI 코딩을 경험하는 **가장 쉬운 경로**입니다.

### 실습 B: GitHub Copilot으로 코딩 생산성 높이기 ⭐⭐

**사용 도구:** VS Code + GitHub Copilot

**Step 1:** GitHub 학생 인증하기

1. github.com에 로그인
2. education.github.com/pack에 접속
3. "Get your pack" 클릭
4. 학교 이메일(.ac.kr)로 인증 또는 학생증 업로드
5. 승인 후 **Copilot Student 플랜 활성화** (무료, 200만+ 학생 사용 중)

1. VS Code 열기
2. Extensions → "GitHub Copilot" 검색 → Install
3. GitHub 계정으로 로그인
4. 하단 상태바에 Copilot 아이콘 확인

**Step 3:** Copilot 자동완성 체험

```python
# Python 파일에서 다음을 입력합니다:
def calculate_gpa(grades):
    # 여기서 Tab을 누르면 Copilot이 나머지를 제안합니다
```

**Step 4:** Copilot Chat 활용

VS Code에서 `Ctrl+I` (Mac: `Cmd+I`)를 누르고

```
이 함수에 에러 핸들링을 추가해줘.
빈 리스트가 입력되면 0.0을 반환하고,
유효하지 않은 학점이 있으면 ValueError를 발생시켜줘.
```

**Step 5:** 테스트 코드 자동 생성

```
# Copilot Chat에 요청:
calculate_gpa 함수의 유닛 테스트를 pytest로 작성해줘.
정상 케이스, 빈 리스트, 유효하지 않은 학점 케이스를 포함해줘.
```

✅ **체크포인트:** Copilot이 코드 제안을 회색 텍스트로 표시하나요? Tab으로 제안을 수락할 수 있나요? Copilot Chat에서 질문에 답변하나요?

🔧 **트러블슈팅:**
- **"Copilot이 제안을 안 해요":** 하단 상태바에서 Copilot 아이콘 확인, 비활성화 상태면 클릭하여 활성화
- **"학생 인증이 안 돼요":** .ac.kr 이메일이 없으면 학생증 사진으로 인증 가능 (1~7일 소요)
- **"제안이 영어만 나와요":** 한국어 주석을 쓰면 한국어 맥락의 코드를 제안합니다

## 4. 프롬프트 진화 패턴

### AI 코딩 프롬프트의 진화

**🔴 Level 1 — 단순 요청:**

```
파이썬으로 계산기 만들어줘
```

→ 기본적인 콘솔 계산기가 나옴. 에러 핸들링, UI, 테스트 없음.

**🟡 Level 2 — 조건 추가:**

```
React로 할일 관리 앱을 만들어줘.
CRUD 기능, 로컬 스토리지 저장, 반응형 디자인.
```

→ 기능이 구체적이나, 기술 스택과 품질 기준이 없음.

**🟢 Level 3 — 완전한 스펙:**

```
Next.js 14 + TypeScript + Supabase로 팀 프로젝트 관리 앱:
■ 기능: 프로젝트 CRUD, 태스크 칸반보드, 팀원 초대, 실시간 알림
■ 인증: Supabase Auth (Google OAuth)
■ DB: Supabase PostgreSQL (RLS 적용)
■ UI: shadcn/ui + Tailwind CSS, 다크모드
■ 배포: Vercel
■ 추가: 에러 핸들링, 로딩 상태, 빈 상태(empty state) 포함
각 기능을 모듈별로 분리하고, 테스트 코드 포함.
```

→ 기술 스택, 인증, DB, UI, 배포, 품질 기준까지 모두 지정. AI가 프로덕션 수준의 코드를 생성.

| 평가 항목 | ChatGPT | Claude | Gemini | Perplexity |
|----------|---------|--------|--------|------------|
| **코드 완성도** | ★★★★★ | ★★★★★ | ★★★★☆ | ★★★☆☆ |
| **코드 설명** | ★★★★☆ | ★★★★★ | ★★★★☆ | ★★★☆☆ |
| **에러 핸들링** | ★★★★☆ | ★★★★★ | ★★★☆☆ | ★★★☆☆ |
| **테스트 코드** | ★★★★☆ | ★★★★★ | ★★★☆☆ | ★★☆☆☆ |
| **디버깅 능력** | ★★★★☆ | ★★★★★ | ★★★★☆ | ★★★☆☆ |

**도구별 특징:**

- **ChatGPT:** 가장 대중적. 코드 생성 속도가 빠르고, Code Interpreter로 즉시 실행 가능. 하지만 긴 코드에서 맥락을 잃을 수 있음.
- **Claude:** 코드 품질이 가장 높음. 에러 핸들링, 엣지 케이스까지 고려. "이 코드의 잠재적 문제점은..."을 자발적으로 설명. 디버깅에 최적.
- **Gemini:** Google 생태계와 통합. 코드 실행 환경 내장. 다만 복잡한 로직에서 불안정할 수 있음.
- **Perplexity:** 코딩보다는 "어떤 도구/라이브러리를 써야 하나?" 같은 기술 조사에 적합. 코드 생성은 보조적.

**최적 조합:** ChatGPT/Claude(코드 생성) → Perplexity(기술 조사) → GitHub Copilot/Cursor(IDE에서 실행)

## 6. 실전 사례: AI 코딩의 빛과 그림자

### 사례 1: GitHub Copilot — Accenture 대규모 RCT로 입증된 생산성

Accenture 개발자를 대상으로 한 **무작위 대조 실험(RCT)**에서 GitHub Copilot의 효과가 과학적으로 입증되었습니다

| 지표 | 결과 |
|------|------|
| PR(Pull Request) 증가 | +8.69% |
| PR 병합률 증가 | +15% |
| 성공적 빌드 증가 | +84% |

| 반복 작업 정신적 부담 감소 | -70% |

## 7. 꿀팁 & 주의사항

### 꿀팁 5선

**Tip 1: GitHub Student Developer Pack을 반드시 신청하라**

GitHub Copilot 무료 외에도: .tech 도메인 무료, Canva Pro 무료, GitHub Pages(포트폴리오 호스팅), Codespaces(클라우드 개발 환경)가 포함됩니다. education.github.com/pack에서 신청하세요.

**Tip 2: "코드를 설명해줘"가 최고의 학습법이다**

AI 코딩 도구의 가장 강력한 기능은 코드 **생성**이 아니라 **설명**입니다

```
이 코드를 한 줄씩 설명해줘.
초보자가 이해할 수 있는 수준으로,
각 줄이 왜 필요한지 포함해서.
```

**Tip 3: 테스트 코드를 먼저 요청하라**

AI에게 코드를 요청하기 전에 테스트를 먼저 요청하세요

```
학생 성적 관리 프로그램의 유닛 테스트를 먼저 작성해줘.
정상 케이스, 엣지 케이스, 에러 케이스를 포함해줘.
그 다음에 이 테스트를 통과하는 코드를 작성해줘.
```

**Tip 4: Cursor의 @codebase를 적극 활용하라**

Cursor에서 `@codebase`를 붙이면 전체 프로젝트를 맥락으로 참조합니다. 대규모 프로젝트에서 특히 유용합니다

```
@codebase 이 프로젝트에서 사용자 인증 로직이 어디에 있어?
```

**Tip 5: AI 생성 코드를 커밋 전에 반드시 읽어라**

AI가 생성한 코드를 "돌아가니까 커밋"하지 마세요. 최소한 다음을 확인하세요
- 하드코딩된 API 키나 비밀번호가 없는가?
- 불필요한 console.log가 남아 있지 않은가?
- 에러 핸들링이 적절한가?

### ⚠️ Anti-Pattern 1: "코드는 돌아가지만 품질은 위험" — 바이브 코딩의 그림자

**상황:** 학생 L이 Replit으로 웹앱을 완성. "AI가 다 해줬다! 완벽해!"

**잘못된 방법:** AI 생성 코드를 읽지 않고 그대로 배포. 테스트 없음, 코드 리뷰 없음.

**결과:**
- 보안 취약점(SQL 인젝션) 발견. 사용자 데이터 유출 위험
- 로직 오류로 계산 결과 불일치
- 서비스 신뢰도 붕괴

**올바른 방법:**
1. AI 생성 코드를 반드시 리뷰 — 특히 보안 취약점 확인
2. 테스트 코드를 AI에게 먼저 요청 — 코드보다 테스트가 먼저
3. 보안 검사 수행 — API 키, 인젝션, XSS 등 기본 체크

**차이:** 2025년 12월 대규모 분석에 따르면, AI가 공동 작성한 코드는 인간이 작성한 코드 대비 주요 이슈 **1.7배**, 로직 오류 **75%** 더 많습니다. 전문 개발자의 **72%가 프로덕션 코드에 바이브 코딩을 사용하지 않는** 이유입니다. **"돌아간다 ≠ 안전하다"**를 기억하세요.


---

### ⚠️ Anti-Pattern 2: "AI가 코딩해줘도 디버깅은 내가 해야 한다"

**상황:** 대학생 K가 Cursor로 웹앱을 완성. "AI가 다 해줬다."

**잘못된 방법:** 배포 후 에러 발생 → 에러 메시지를 AI에 복사 → AI 수정 코드 적용 → 새로운 에러 → 반복.

**결과:**
- 에러가 누적. 코드가 점점 복잡해지며 AI도 해결 불가
- 결국 처음부터 다시 시작
- 시간과 노력 모두 낭비

**올바른 방법:**
1. 에러 메시지를 직접 읽고 이해 — 에러의 유형과 위치 파악
2. AI에게 "이 에러가 왜 발생하는지 설명해줘"로 학습 → 직접 수정
3. 스택 트레이스를 이해하여 에러의 근본 원인 추적

**차이:** AI 코딩 도구는 "에러 없는 코드"를 만드는 게 아니라 "에러를 더 빠르게 만나게" 해줍니다. 기초 디버깅 능력(에러 메시지 읽기, 스택 트레이스 이해)은 AI 시대에도 필수입니다. 특히 "이 에러 고쳐줘"(❌)가 아닌 "이 TypeError가 왜 발생하는지 설명하고, 원인별로 해결법을 알려줘"(✅)로 AI에게 질문하는 것이 핵심입니다.

## 8. 워크플로우 일반화: AI 코딩의 보편 원칙

### 원칙 1: 경험 수준에 맞는 도구를 선택하라

| 코딩 경험 | 추천 도구 | 이유 |
|-----------|----------|------|
| 코딩 불가 (비전공자) | Lovable, Replit Agent | 코딩 없이 앱 제작 |
| 코딩 가능 (초보~중급) | Cursor, GitHub Copilot | 자동완성 + 코드 설명 + 학습 |
| 코딩 전문 (개발자) | Claude Code, Cursor, Copilot | 대규모 리팩토링, 멀티에이전트 |

### 원칙 2: AI 코딩 = 80% 가속 + 20% 검증

AI가 코드의 80%를 빠르게 생성해주지만, 나머지 20%는 인간의 검증이 필요합니다

```
AI 코딩 워크플로우:
[1] AI에게 요구사항 전달 (프롬프트)
[2] AI 생성 코드 리뷰 (이해 + 검증)
[3] 테스트 작성 및 실행 (자동/수동)
[4] 보안 검사 (API 키, 취약점)
[5] 커밋 + 배포
```


### 원칙 3: 학습 목적의 AI 코딩 vs 생산성 목적의 AI 코딩

| 목적 | 접근법 | 핵심 질문 |
|------|--------|----------|
| **학습** | AI 생성 코드를 한 줄씩 이해 | "왜 이렇게 동작하지?" |
| **생산성** | AI에게 반복 작업 위임 | "이 작업을 더 빠르게 할 수 있나?" |

프로그래밍 수업에서는 **학습 목적**으로, 프로젝트/창업에서는 **생산성 목적**으로 사용하세요. 두 목적을 혼동하면 "AI가 다 해줬는데 아무것도 모르겠다"는 상태가 됩니다.

### 원칙 4: 카테고리당 1개 도구만 마스터하라

도구가 너무 많습니다. 모든 도구를 써보려 하지 말고, 각 카테고리에서 1개만 깊이 익히세요

- **자동완성:** GitHub Copilot (학생 무료)
- **AI IDE:** Cursor 또는 Windsurf (택 1)
- **노코드 빌더:** Replit 또는 Lovable (택 1)
- **UI 생성:** v0

## 1. Claude Code로 개발하는 것의 의미

### 1-1. 전통 개발 보조 도구 vs Claude Code

| 항목 | GitHub Copilot | ChatGPT (코드 생성) | Claude Code |
|------|---------------|-------------------|-------------|
| 작동 방식 | IDE 내 자동완성 | 대화형 코드 생성 | **프로젝트 전체 이해 + 실행** |
| 맥락 범위 | 현재 파일 중심 | 대화창에 붙여넣은 코드 | **전체 코드베이스** |
| 파일 관리 | 한 파일 수정 | 복사-붙여넣기 | **직접 파일 생성·수정** |
| 실행 | 불가 | 불가 (Code Interpreter 제외) | **터미널 직접 실행** |
| 테스트 | 불가 | 코드만 생성 | **테스트 작성 + 실행 + 수정** |
| 디버깅 | 제한적 | 에러 붙여넣기 → 해결책 | **에러 확인 → 수정 → 재실행** |
| 프로젝트 구조 | 인식 제한 | 없음 | **폴더 구조 인식 + 설계** |
| 벤치마크 (SWE-Bench) | 해결률 56.0%, 속도 89.9초 | — | **코드베이스 전체 이해 + 멀티에이전트** |
| 제안 수용률 | 30% (70%는 개발자가 거부) | — | CLAUDE.md 규칙으로 품질 기준 강제 |

### ❌ 흔한 실수: AI 코딩의 함정

**상황**: Claude Code로 코딩 프로젝트를 할 때

---

**잘못된 방법 — AI가 생성한 코드를 이해하지 않고 사용**

```
코드 전체를 만들어줘. 동작만 하면 돼.
```

**결과**: AI가 만든 코드가 동작하지만, 왜 이렇게 작동하는지 모릅니다. 과제 발표에서 "이 부분은 어떻게 동작하나요?"라는 질문에 답하지 못합니다. 본문 Anti-Pattern 2 "디버깅은 내가 해야 한다"와 동일 — 에러 메시지를 AI에 복사 → AI 수정 코드 적용 → 새로운 에러 → 반복의 악순환에 빠지면 코드가 점점 복잡해져 결국 처음부터 다시 시작하게 됩니다.

**올바른 방법**: developer 에이전트가 모든 코드에 **학습용 주석**("📚 학습 포인트")을 달도록 CLAUDE.md에 규칙을 설정합니다. 코드를 받은 후 주석을 읽고, 이해가 안 되는 부분은 CC에게 추가 설명을 요청합니다.

**차이**: CC는 **"코드를 대신 써주는 도구"**가 아니라, **"코드를 가르쳐주는 도구"**입니다.

---

**잘못된 방법 — 보안을 무시한 코드**

```
빨리 동작하게만 만들어줘. 보안은 나중에.
```

**결과**: API 키가 코드에 하드코딩되고, SQL 인젝션에 취약한 코드가 생성됩니다. 배포 후 문제가 발생합니다.

**올바른 방법**: CLAUDE.md에 보안 원칙(환경변수, 파라미터 바인딩, XSS 방지)을 설정하면, developer가 처음부터 안전한 코드를 작성합니다.

**차이**: "나중에 고치자"는 대부분 "영원히 안 고친다"가 됩니다. CLAUDE.md로 **처음부터 안전하게**.

### 📊 하이브리드 전략 비교표: 본문 도구 vs CC 하네스

| 개발 단계 | 본문 도구 조합 | CC 하네스 | 최적 전략 |
|----------|--------------|----------|----------|
| **1. 설계** | ChatGPT/Claude로 아키텍처 상담 | architect 에이전트(폴더 구조·API·데이터 모델 자동) | CC architect가 **파일까지 직접 생성**하여 압도적 |
| **2. UI 프로토타입** | **v0(React UI 최강)** + Lovable(풀스택 노코드) | developer 에이전트(HTML/CSS/JS 코드 생성) | **v0/Lovable로 시각적 UI** → CC로 백엔드 연동·확장 |
| **3. 코드 구현** | GitHub Copilot(자동완성) + Cursor(코드베이스 이해) | developer 에이전트(학습 주석 + 보안 규칙 자동 적용) | Copilot/Cursor로 IDE 내 코딩 → CC로 멀티파일 리팩토링 |
| **4. 테스트** | Copilot Chat으로 테스트 코드 생성 | tester 에이전트(단위·API·시나리오 + **실행까지**) | CC tester가 **테스트 작성+실행+수정** 일관 수행으로 우위 |
| **5. 디버깅** | 에러를 ChatGPT에 붙여넣기 → 해결책 | CC가 **에러 확인→수정→재실행** 직접 수행 | CC의 **터미널 직접 실행**이 압도적 |
| **6. 배포** | Vercel/Render 수동 설정 | scripts/deploy.sh 자동화 | CC 스크립트로 **원클릭 배포** |

**핵심**: 본문의 "경험 수준별 도구 선택" 플로우차트 — 코딩 불가(Lovable/Replit) → 초보~중급(Cursor/Copilot) → 전문가(Claude Code) — 는 CC 하네스가 **전문가 수준의 개발 프로세스**를 제공한다는 것을 의미합니다. Accenture RCT의 "코딩이 더 즐거워졌다" **95%** 응답은 AI 코딩 도구 전체에 해당하지만, CC의 차별화는 **"학습 주석 규칙"**입니다. 본문의 "학습 목적 vs 생산성 목적" 구분에서 CC의 "📚 학습 포인트" 주석은 두 목적을 동시에 달성합니다.
