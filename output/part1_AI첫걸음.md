- AI의 기술적 원리(트랜스포머, 어텐션 메커니즘 등)를 깊이 알고 싶은 분 → 이 교재의 범위 밖입니다

## 2. 이 챕터에서 배우는 것

### 학습목표

이 챕터를 마치면 다음을 할 수 있습니다

1. **ChatGPT, Gemini, Claude, Perplexity** 4대 AI 도구의 특성과 차이를 설명할 수 있다
2. 같은 질문이라도 **프롬프트를 어떻게 쓰느냐**에 따라 결과가 극적으로 달라진다는 것을 체감한다
3. **역할 부여, 조건 명시, 출력 형식 지정** 등 프롬프트 엔지니어링 핵심 기법 3가지를 적용할 수 있다
4. AI 답변의 **신뢰성을 검증하는 방법**(교차검증, 팩트체킹)을 실행할 수 있다
5. **학업 상황별 최적 도구**를 선택할 수 있다

### 완성 후 얻는 결과물

> 이 챕터의 실습을 마치면, 동일한 과제 프롬프트를 4개 AI 도구에 입력하고 결과를 비교 분석한 **"나만의 AI 도구 비교 보고서"**를 갖게 됩니다. 이 보고서는 앞으로 어떤 과제를 만나도 "어떤 AI 도구를 어떻게 쓸까?"를 판단하는 나침반이 됩니다.

### 🔍 도구 비교 실험: "경영학 레포트 개요"를 4개 도구에 물어보면?

동일한 프롬프트를 4개 도구에 입력했을 때 어떤 차이가 나는지 직접 비교해봅시다.

**실험 프롬프트:**

```
당신은 경영학 교수입니다. 대학교 3학년 경영학과 학생이
"디지털 트랜스포메이션이 중소기업 경쟁력에 미치는 영향"이라는 주제로
15페이지 분량의 학기말 레포트를 작성하려고 합니다.

다음 조건에 맞는 레포트 개요를 작성해주세요:
1. 서론-본론-결론 구조
2. 각 섹션별 핵심 논점 3개 이상
3. 참고할 수 있는 이론/프레임워크 제안
4. 각 섹션 예상 분량 (페이지 수)
5. 한국 중소기업 사례 포함 권장
```

**비교 결과:**

| 항목 | ChatGPT | Gemini | Claude | Perplexity |
|------|---------|--------|--------|------------|
| **구조 체계성** | ★★★★★ | ★★★★☆ | ★★★★★ | ★★★★☆ |
| **이론/프레임워크** | ★★★★☆ | ★★★★☆ | ★★★★★ | ★★★★☆ |
| **한국 맥락 반영** | ★★★★☆ | ★★★☆☆ | ★★★★☆ | ★★★★★ |
| **출처/참고문헌** | ★★☆☆☆ | ★★☆☆☆ | ★★☆☆☆ | ★★★★★ |
| **실행 가능 구체성** | ★★★★★ | ★★★★☆ | ★★★★★ | ★★★★☆ |

**도구별 특징 요약:**

- **ChatGPT**: Canvas 기능으로 개요를 즉시 편집·발전시킬 수 있습니다. 포터의 5 Forces, TAM 모델 등 이론을 풍부하게 제안하지만, 출처가 구체적이지 않습니다.
- **Gemini**: Google Scholar 연동으로 실제 논문·보고서를 찾아 제안합니다. 다만 한국어 맥락 이해가 상대적으로 약합니다.
- **Claude**: 가장 세밀하고 학술적인 개요를 제공합니다. "왜 → 무엇을 → 어떻게 → 그래서"의 논리적 흐름이 뛰어나고, 분량 배분이 가장 현실적입니다.
- **Perplexity**: 모든 주장에 출처(URL)를 포함합니다. 최신 한국 중소기업 디지털 전환 통계를 실시간 검색하여 제시하지만, 개요 자체의 구조적 완성도는 상대적으로 낮습니다.

**결론 — 상황별 최적 도구:**

> 하나의 도구가 모든 면에서 최고는 아닙니다. **자료 조사는 Perplexity**, **구조 설계는 Claude**, **초안 작성은 ChatGPT**, **교차 검증은 Gemini**로 조합하는 것이 가장 효과적입니다.

## 4. 핵심 개념

### AI는 "검색"이 아니라 "생성"입니다

생성형 AI를 처음 쓰는 분들이 가장 많이 하는 실수는 **Google 검색하듯이** 사용하는 것입니다.

| | Google 검색 | 생성형 AI |
|--|-----------|----------|
| **작동 방식** | 이미 있는 웹페이지를 찾아줌 | 학습한 패턴을 바탕으로 새로운 텍스트를 **생성**함 |
| **출처** | 원본 웹페이지 링크 제공 | 출처 없이 답변 (Perplexity 제외) |
| **정확도** | 원본 문서의 정확도에 의존 | **할루시네이션** 가능 — 없는 정보를 그럴듯하게 지어낼 수 있음 |
| **최적 질문** | 키워드 나열: "디지털 트랜스포메이션 중소기업 논문" | 맥락+조건+형식 구조화: "경영학 3학년이 15페이지 레포트를 쓰려고 합니다..." |

### 프롬프트 엔지니어링 — AI에게 "잘 말하는 기술"

프롬프트(prompt)란 AI에게 보내는 입력 텍스트입니다. 같은 AI라도 프롬프트를 어떻게 쓰느냐에 따라 결과물의 품질이 **하늘과 땅 차이**입니다.

프롬프트 엔지니어링의 핵심 요소 5가지

| 요소 | 설명 | 예시 |
|------|------|------|
| **역할 부여** | AI에게 특정 전문가 역할을 부여 | "당신은 경영학 교수입니다" |
| **맥락 제공** | 본인의 상황과 배경 설명 | "3학년 경영학과, 학기말 레포트" |
| **조건 명시** | 구체적인 요구사항 나열 | "서론-본론-결론, 15페이지, 한국 사례 포함" |
| **출력 형식** | 원하는 결과물의 형태 지정 | "마크다운 표와 계층적 번호 매기기로" |
| **예시 제공** | 참고할 수 있는 예시 힌트 | "포터의 경쟁전략, TAM 모델 등" |


### 🔬 프롬프트 진화 실험

같은 작업을 3단계 프롬프트로 시도하여, 프롬프트 품질의 차이를 직접 체감합니다.

#### 🔴 Level 1 — 초보 프롬프트

```
경영학 레포트 개요 써줘
```

**결과**: 일반적인 경영학 레포트 구조를 나열합니다. "서론-본론-결론"이라는 뻔한 틀에, 주제도 학년도 분량도 모른 채 작성한 막연한 개요가 나옵니다.

📋 **평가**
- 정확성: ★★☆☆☆
- 활용도: ★☆☆☆☆
- 문제점: 맥락 없음, 주제 불명확, 분량 기준 없음. 이대로는 쓸 수 없음

---

#### 🟡 Level 2 — 중급 프롬프트

```
디지털 트랜스포메이션이 중소기업 경쟁력에 미치는 영향이라는 주제로
경영학 레포트 개요를 작성해줘. 서론-본론-결론 구조로, 15페이지 분량이야.
```

**결과**: 주제에 맞는 개요가 나옵니다. 본론에 2~3개 소주제가 포함되고, 디지털 전환의 정의·사례·시사점이 언급됩니다. 하지만 이론적 깊이가 부족하고, 한국 맥락이 없습니다.

📋 **평가**
- 정확성: ★★★★☆
- 활용도: ★★★☆☆
- 개선점: Level 1 대비 주제와 구조가 명확해짐
- 아쉬운 점: 이론 프레임워크 부재, 한국 사례 미포함, 각 섹션 분량 미제시

---

#### 🟢 Level 3 — 전문가 프롬프트

```
당신은 경영학 교수입니다. 대학교 3학년 경영학과 학생이
"디지털 트랜스포메이션이 중소기업 경쟁력에 미치는 영향"이라는 주제로
15페이지 분량의 학기말 레포트를 작성하려고 합니다.

다음 조건에 맞는 레포트 개요를 작성해주세요:
1. 서론-본론-결론 구조
2. 각 섹션별 핵심 논점 3개 이상
3. 참고할 수 있는 이론/프레임워크 제안 (포터의 경쟁전략, TAM 모델 등)
4. 각 섹션 예상 분량 (페이지 수)
5. 한국 중소기업 사례 포함 권장 (중소벤처기업부 자료 참고)

출력 형식: 마크다운 표와 계층적 번호 매기기로 구성
```

**결과**: 즉시 활용 가능한 체계적 개요가 나옵니다. 포터의 5 Forces, TAM 모델, 자원기반관점(RBV) 등 이론이 적절히 배치되고, 한국 중소기업 사례가 포함되며, 각 섹션별 페이지 수까지 제시됩니다.

📋 **평가**
- 정확성: ★★★★★
- 활용도: ★★★★★
- 핵심 차이: 역할 부여 + 구체적 조건 5가지 + 예시 힌트 + 출력 형식까지 지정

---

#### 💡 교훈: 프롬프트의 마법

| 요소 | Level 1 | Level 2 | Level 3 |
|------|---------|---------|---------|
| 역할 부여 | ✗ | ✗ | ✓ ("경영학 교수") |
| 맥락 제공 | ✗ | △ (주제만) | ✓ (학년, 과목, 분량) |
| 조건 명시 | ✗ | △ (구조만) | ✓ (5가지 상세 조건) |
| 출력 형식 | ✗ | ✗ | ✓ (마크다운 표) |
| 예시/참고 | ✗ | ✗ | ✓ (포터, TAM) |
| **결과 품질** | **20%** | **60%** | **95%** |

> 📌 **이 프롬프트가 좋은 이유:**
> - **역할 부여** ("경영학 교수") → 학술적 톤과 전문성을 유도합니다
> - **구체적 맥락** (3학년, 15페이지, 학기말) → AI가 적절한 수준과 분량을 조절합니다
> - **5가지 제약조건** → AI가 빠짐없이 체크하며 답변을 구성합니다
> - **예시 힌트** (포터, TAM) → AI가 같은 수준의 다른 이론도 추가합니다
> - **출력 형식 지정** → 바로 사용할 수 있는 깔끔한 결과가 나옵니다

### 도구별 프롬프트 최적화 전략

같은 프롬프트라도 도구마다 조금씩 다르게 쓰면 더 좋은 결과를 얻을 수 있습니다

| 도구 | 프롬프트 팁 | 이유 |
|------|-----------|------|
| **ChatGPT** | 역할 부여 + 출력 형식 지정이 가장 효과적. Canvas 모드에서 반복 수정 활용 | 지시를 충실히 따르는 스타일 |
| **Gemini** | "최신 자료를 검색하여~"라고 명시. 이미지/PDF 업로드 후 분석 요청 활용 | 검색 연동이 강점 |
| **Claude** | "단계별로 생각하여~" 명시. 긴 문서를 통째로 붙여넣고 분석 요청. XML 태그(`<context>...</context>`) 활용 | 분석적이고 신중한 스타일 |
| **Perplexity** | "~에 대한 최신 연구/통계를 찾아줘" 형태가 최적. Focus 모드(Academic) 활용 | 검색 특화, 사실 중심 |

### 할루시네이션 — AI의 "자신감 있는 거짓말"

생성형 AI의 가장 큰 위험은 **할루시네이션(hallucination)**입니다. 틀린 정보를 마치 사실인 것처럼, 때로는 더 확신에 찬 어조로 말합니다.

**2025~2026년 할루시네이션 현황:**

| 모델 | 할루시네이션률 | 특징 |
|------|-------------|------|
| Gemini 2.0 Flash | **0.7%** | 가장 낮음 |
| Perplexity Sonar | ~1.5% | 검색 기반으로 보완 |
| Claude Opus 4 | ~2.5% | "모르겠다"고 인정하는 경향 |
| GPT-5 | ~3% | 범용 성능 우수 |
| OpenAI o3 (추론 특화) | **33%** | 추론 모델의 역설 |

> ⚠️ **주의**: 최고 성능 모델의 할루시네이션률이 0.7%까지 떨어졌지만, **완전히 0%가 될 수는 없습니다.** 2025년 수학적 증명으로 현재 LLM 아키텍처에서 할루시네이션 완전 제거가 불가능하다는 것이 확인되었습니다. AI가 "확실히~", "분명히~"라고 말할 때 오히려 더 의심해야 합니다 — Zhang et al. (2025)의 연구에 따르면 AI가 할루시네이션할 때 34% 더 높은 확률로 확신 표현을 사용합니다.

**준비물**: 웹 브라우저, 각 도구의 무료 계정 (회원가입만 하면 됩니다)
**사용 도구**: ChatGPT, Gemini, Claude, Perplexity

하버드대학교 최대 물리학 수업(Physical Sciences 2)에서 194명의 학생을 대상으로 AI 튜터와 전통적 능동학습 교실을 비교하는 무작위 대조 실험이 진행되었습니다.

**결과:**
- AI 튜터 그룹의 사후검사 중앙값: **4.5** vs 능동학습 교실: **3.5**

- 학생들의 참여도, 동기, 만족도 모두 향상

> 출처: Kestin et al. (2025), Harvard University, PubMed (pubmed.ncbi.nlm.nih.gov/40537565)

### 사례 2. 메타분석으로 확인된 AI의 학업 효과 (✅ A급)

51개 연구를 종합 분석한 메타분석(Wang & Fan, 2025) 결과

|------|-------------|------|

> 출처: Wang & Fan (2025), *Humanities and Social Sciences Communications*, Nature (nature.com/articles/s41599-025-04787-y)

### 사례 3. 대학생의 AI 활용 실태 — "이미 일상이 되었다" (✅ A급)

Anthropic이 574,740건의 대학생 Claude 대화를 분석한 결과

- **콘텐츠 생성 39.3%**: 연습 문제 설계, 에세이 편집, 학술 자료 요약

하지만 **47%의 대화가 "직접적" 패턴** — 최소한의 노력으로 즉각적 답변만 요구하는 방식이었습니다. 이것이 바로 "제대로 배워야 하는" 이유입니다.

## 7. 꿀팁 & 주의사항

### 전문가 팁

1. **프롬프트는 대화다**: 한 번에 완벽한 결과를 기대하지 마세요. "여기에 한국 사례를 추가해줘", "본론 2번을 더 구체화해줘"처럼 **후속 대화**로 결과를 발전시키세요.

2. **Temporary Chat을 활용하세요**: 개인 정보가 포함된 과제를 작업할 때, ChatGPT의 Temporary Chat 모드를 사용하면 대화 내용이 학습에 사용되지 않습니다.

3. **Perplexity는 "검증 도구"로 쓰세요**: 다른 AI의 답변에서 의심스러운 내용이 있으면, Perplexity에 "~이 사실인지 확인해줘"라고 물어보세요. 출처까지 확인해줍니다.

### ❌ 흔한 실수: AI 답변을 검증 없이 믿기

**상황**: 레포트에 넣을 통계 수치를 AI에게 물어봤습니다.

**잘못된 방법**
```
한국 중소기업의 디지털 전환율이 몇 %인지 알려줘
```
→ AI가 "2024년 기준 한국 중소기업의 디지털 전환율은 42.3%입니다"라고 자신있게 답변합니다.
→ 이 수치를 그대로 레포트에 인용합니다.

**결과**: 이 수치는 할루시네이션일 수 있습니다. 변호사들조차 AI가 만들어낸 가짜 판례를 법정에 인용하여 제재를 받은 사례가 **전 세계 486건**이나 됩니다. 128명의 변호사와 2명의 판사가 실제로 징계를 받았습니다.

**올바른 방법**
```
한국 중소기업의 디지털 전환 현황에 대한 최신 통계를 찾아줘.
반드시 출처(기관명, 보고서명, 발행연도)를 함께 알려줘.
```
→ Perplexity에서 검색하면 출처가 포함된 답변을 받을 수 있습니다.
→ 제시된 출처 링크를 직접 클릭하여 원본 보고서에서 수치를 확인합니다.

**차이**: "출처를 요구"하는 한 줄을 추가하고, Perplexity를 활용한 것만으로 할루시네이션 위험을 크게 줄였습니다. 출처를 직접 확인하는 마지막 단계가 핵심입니다.

### AI 답변 교차검증 5단계 워크플로우

AI가 알려준 정보를 레포트에 쓰기 전, 반드시 이 순서로 검증하세요

```
Step 1. 동일 질문을 최소 2개 도구에 입력하여 답변 비교
Step 2. Perplexity로 핵심 주장의 출처 확인
Step 3. 구체적 수치/통계는 원본 출처(논문, 보고서)에서 직접 확인
Step 4. AI가 "확실히~", "분명히~" 표현을 사용하면 오히려 의심하기
Step 5. 최신 정보일수록 검증 필수 (AI 학습 데이터 마감일 이후 정보는 부정확 가능)
```

| 학업 상황 | 1순위 | 2순위 | 이유 |

## 1. Claude Code 설치와 기본 사용법

### 1-1. Claude Code란?

본문에서 ChatGPT, Gemini, Claude 웹, Perplexity 4개 도구를 비교해봤습니다. 이 도구들은 모두 **웹 브라우저(GUI)**에서 작동합니다. Claude Code는 이와 다르게 **터미널(명령줄, CLI)**에서 Claude AI와 대화하는 도구입니다. 브라우저를 열지 않아도 터미널 창 하나에서 질문하고, 로컬 파일을 직접 읽고, 코드를 실행하고, 반복 작업을 자동화할 수 있습니다.

### 1-2. 설치 방법

#### macOS

```bash
# 1. Node.js 설치 (Homebrew 이용)
brew install node

# 2. Claude Code 설치
npm install -g @anthropic-ai/claude-code

# 3. 설치 확인
claude --version
```

✅ **체크**: `claude v1.x.x` 형태의 버전 번호가 출력되면 정상입니다.

> 🔧 **안 되나요?**
> - `permission denied` 오류 → `sudo npm install -g @anthropic-ai/claude-code`로 재시도하거나, nvm을 설치하여 Node.js를 관리하면 sudo 없이 사용 가능합니다.
> - `brew: command not found` → Homebrew 먼저 설치: `/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"`
> - `node --version`으로 Node.js 18 이상인지 확인하세요. 18 미만이면 `brew upgrade node`로 업그레이드하세요.

#### Windows

```powershell
# 1. Node.js 설치 — https://nodejs.org 에서 LTS 버전 다운로드 후 설치
# 또는 winget 사용:
winget install OpenJS.NodeJS.LTS

# 2. Claude Code 설치 (PowerShell 또는 명령 프롬프트)
npm install -g @anthropic-ai/claude-code

# 3. 설치 확인
claude --version
```

✅ **체크**: `claude v1.x.x` 형태의 버전 번호가 출력되면 정상입니다.

> 🔧 **안 되나요?**
> - `npm: command not found` → Node.js 설치 후 **터미널을 새로 열어야** PATH가 적용됩니다.
> - PowerShell 실행 정책 오류 (`cannot be loaded because running scripts is disabled`) → 관리자 PowerShell에서 `Set-ExecutionPolicy RemoteSigned` 실행 후 재시도하세요.
> - `claude: command not found` → `npx @anthropic-ai/claude-code`로 실행하거나, `npm config get prefix` 경로가 시스템 PATH에 포함되어 있는지 확인하세요.

#### Linux (Ubuntu/Debian)

```bash
# 1. Node.js 설치 (nvm 권장 — sudo 불필요)
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.40.1/install.sh | bash
source ~/.bashrc
nvm install --lts

# 2. Claude Code 설치
npm install -g @anthropic-ai/claude-code

# 3. 설치 확인
claude --version
```

✅ **체크**: `claude v1.x.x` 형태의 버전 번호가 출력되면 정상입니다.

> ⚠️ **주의**: Node.js 18 이상이 필요합니다. `node --version`으로 버전을 확인하세요.
>
> 🔧 **안 되나요?**
> - `nvm: command not found` → `source ~/.bashrc` (또는 `source ~/.zshrc`) 실행 후 재시도하세요.
> - 글로벌 설치 권한 오류 → nvm을 사용하면 sudo 없이 설치할 수 있습니다.

### 1-3. 처음 실행하기

```bash
# 터미널에서 Claude Code 시작
claude
```

처음 실행 시 Anthropic 계정 인증이 필요합니다. 터미널에 표시되는 안내를 따라 로그인하면 됩니다.

✅ **체크**: `>` 프롬프트가 나타나면 대화 준비 완료입니다.

### 🔬 프롬프트 진화 실험: Claude Code에 질문하기

같은 작업을 3단계 프롬프트로 시도하여, 프롬프트 품질의 차이를 직접 체감합니다.

#### 🔴 Level 1 — 초보 프롬프트

```
파일 요약해줘
```

📋 **평가**
- 정확성: ★☆☆☆☆
- 활용도: ★☆☆☆☆
- 문제점: 대상 파일, 요약 기준, 출력 형식이 모두 불명확

---

#### 🟡 Level 2 — 중급 프롬프트

```
report.pdf를 읽고 내용을 요약해줘
```

**결과**: 파일을 읽고 전체 내용을 장황하게 요약합니다. 길이 조절이 안 되고, 어떤 관점에서 요약해야 하는지 불명확합니다.

📋 **평가**
- 정확성: ★★★☆☆
- 활용도: ★★★☆☆
- 개선점: Level 1 대비 파일이 특정되어 실제 작업이 수행됨
- 아쉬운 점: 요약 분량, 관점, 출력 형식이 미지정

---

#### 🟢 Level 3 — 전문가 프롬프트

```
report.pdf를 읽고 다음 기준으로 요약해줘:
- 분량: 5줄 이내의 핵심 요약
- 관점: 대학생 과제 활용 관점
- 형식: 번호 매기기
- 포함 사항: 핵심 주장, 사용된 방법론, 주요 결론
- 결과를 summary.md로 저장
```

**결과**: 정확히 5줄 이내로, 과제에 바로 활용할 수 있는 구조화된 요약이 생성되고, 파일로 저장됩니다.

📋 **평가**
- 정확성: ★★★★★
- 활용도: ★★★★★
- 핵심 차이: 분량·관점·형식·저장까지 한 번에 지정하여 추가 작업이 불필요

## 3. GUI(ChatGPT/Gemini/Claude 웹/Perplexity) vs CLI(Claude Code) 비교

본문에서 4개 도구를 "상황별 최적 도구"로 조합하는 워크플로우를 배웠습니다. Claude Code는 이 조합에 **자동화 레이어**를 추가합니다.

| 기준 | GUI (웹 기반 채팅) | CLI (Claude Code) |
|------|-------------------|-------------------|
| **접근성** | 브라우저만 있으면 사용 가능 | 터미널 설치 필요 |
| **파일 처리** | 파일을 직접 업로드/다운로드 | 로컬 파일 시스템에 직접 접근 |
| **대량 작업** | 파일을 하나씩 업로드해야 함 | 폴더 전체를 한 번에 처리 가능 |
| **자동화** | 수동 반복 작업 | 스크립트로 자동화 가능 |
| **코드 실행** | 제한적 (일부 플랫폼만) | 로컬 환경에서 바로 실행 |
| **버전 관리** | 대화 기록만 저장 | Git과 연동, 변경 이력 추적 |
| **프로젝트 규모** | 단일 대화 중심 | 프로젝트 전체를 맥락으로 활용 |

### 언제 GUI가 유리한가?

- 간단한 질문/답변이 필요할 때
- 이미지 생성이나 시각적 결과물이 필요할 때
- 터미널에 익숙하지 않은 초보 사용자
- 모바일 환경에서 빠르게 접근할 때

### 언제 CLI(Claude Code)가 유리한가?

- 여러 파일을 동시에 처리해야 할 때
- 반복 작업을 자동화하고 싶을 때
- 코딩 프로젝트를 진행할 때
- 로컬 파일을 직접 읽고 수정해야 할 때
- 워크플로우를 재사용 가능하게 만들고 싶을 때

## 4. 간단한 자동화 예시

### 4-1. 파일 일괄 이름 변경

학기 중에 "과제1_홍길동.docx", "과제1_김철수.docx" 같은 파일들의 이름 형식을 통일하고 싶을 때

```
> 현재 폴더의 모든 docx 파일 이름을 "2024_AI과목_학번_이름.docx" 형식으로 변경해줘.
> 파일 내용에서 학번과 이름을 추출해서 사용해.

Claude: [파일들을 분석하고 이름을 변경합니다]
        - 과제1_홍길동.docx → 2024_AI과목_20210001_홍길동.docx
        - 과제1_김철수.docx → 2024_AI과목_20210042_김철수.docx
        ...
```

### 4-2. 텍스트 요약 자동화

여러 논문 PDF를 한 번에 요약하고 싶을 때

```
> papers 폴더에 있는 모든 PDF 파일을 각각 읽고,
> 각 논문의 제목, 저자, 핵심 주장, 방법론을 표로 정리해서
> summary.md 파일로 저장해줘

Claude: [papers 폴더를 스캔합니다]
        [5개의 PDF 파일을 발견했습니다]
        [각 파일을 분석하는 중...]
        [summary.md 파일을 생성했습니다]
```

### 4-3. 비대화형 모드로 자동화 파이프라인 만들기

Claude Code는 `-p` (print) 플래그를 사용하면 비대화형 모드로 실행됩니다. 이를 활용하면 셸 스크립트에서 자동화가 가능합니다.

```bash
# 단일 명령 실행
claude -p "현재 디렉터리의 파일 목록을 알려줘"

# 파일을 읽고 결과를 다른 파일로 저장
claude -p "report.txt를 읽고 영어로 번역해줘" > report_en.txt

# 셸 스크립트로 여러 파일 일괄 처리
for file in papers/*.pdf; do
  claude -p "$file 을 읽고 3문장으로 요약해줘" > "summaries/$(basename "$file" .pdf)_summary.txt"
done
```

```
> 각 섹션에 들어갈 핵심 논점을 정리해서
> draft_outline.md로 저장해줘

Claude: [보고서 개요를 구성합니다]
        [draft_outline.md를 생성했습니다]
```

## 5. 더 나아가기: 하네스 맛보기

> Ch 2 본문에서 "에이전트 AI"가 2024~2025년 사이 문의량 **1,445% 급증**한 핵심 트렌드임을 배웠습니다. Claude Code의 **하네스(Harness)**는 바로 이 에이전트 AI를 직접 체험할 수 있는 기능입니다.

**하네스**란 여러 AI 에이전트를 팀처럼 구성하여 복잡한 작업을 분담시킬 수 있는 환경입니다. 본문에서 배운 "Perplexity→Claude→ChatGPT→Gemini" 4개 도구 조합 워크플로우를, AI 에이전트가 자동으로 수행하는 것이라고 생각하면 됩니다.

예를 들어, 교재 한 챕터를 작성할 때
- **리서처 에이전트**: 관련 자료를 웹에서 수집 (Perplexity 역할)
- **작성 에이전트**: 수집된 자료를 바탕으로 본문 작성 (ChatGPT 역할)
- **편집 에이전트**: 작성된 본문의 문법과 흐름 검수 (교차검증 역할)

이런 구성을 `.claude/agents/` 폴더에 마크다운 파일로 정의하면, Claude Code가 각 역할을 자동으로 수행합니다. 자세한 내용은 이후 챕터에서 다룹니다.

```bash
#!/bin/bash
# summarize.sh — docs 폴더의 모든 텍스트 파일을 일괄 요약

mkdir -p summaries

for file in docs/*.txt; do
  filename=$(basename "$file" .txt)
  echo "📄 $file 처리 중..."
  claude -p "$file 을 읽고 핵심을 2문장으로 요약해줘" > "summaries/${filename}_summary.txt"
  echo "✓ 완료 → summaries/${filename}_summary.txt"
done

echo "🎉 모든 파일 요약 완료!"
```

```bash
chmod +x ~/ai-practice/summarize.sh
```

✅ **체크**: `cat ~/ai-practice/summarize.sh`로 내용이 정상적으로 저장되었는지 확인

### AI 생성물 저작권 — 이용약관 비교

AI가 만들어준 결과물, 과연 "내 것"일까요?

| 항목 | ChatGPT | Gemini | Claude | Perplexity |
|------|---------|--------|--------|------------|
| **출력물 소유권** | ✅ 사용자 소유 | ✅ 사용자 소유 | ✅ 사용자 소유 | ❓ 불명확 |
| **상업적 사용** | ✅ 허용 | ✅ 허용 | ✅ 허용 | ✅ 허용 |
| **AI 사용 표시 의무** | ⚠️ 2026년 기준 필수 공개 요구 | 권장 | 권장 | 권장 |
| **다른 사용자에게 같은 결과** | ⚠️ 가능 | ⚠️ 가능 | ⚠️ 가능 | ⚠️ 가능 |

**핵심 구분 — 계약상 소유 vs 법적 저작권:**

- **"계약상 소유"**: 플랫폼이 "출력물은 당신의 것"이라고 이용약관에 명시 → 플랫폼이 소유권을 주장하지 않겠다는 의미
- **"법적 저작권"**: 실제로 법적 보호를 받아 제3자의 무단 사용을 막을 수 있는 권리 → **인간의 상당한 창의적 기여** 없이는 인정되지 않음

> 즉, AI 출력물을 "소유"할 수는 있지만, 다른 사람이 같은 내용을 사용하는 것을 **법적으로 막을 수는 없을 수 있습니다.**

| 수준 | 설명 | 예시 | 판단 |

| **Level 4: AI 출력 직접 사용** | AI 답변을 거의 수정 없이 제출 | AI가 쓴 에세이를 복붙 | ❌ 대부분 부정행위 |
| **Level 5: AI 대리 제출** | AI에게 과제 전체를 위임 | 프롬프트만 입력하고 결과 제출 | ❌ 명백한 부정행위 |

> 💡 **핵심 기준**: Level 1~2는 거의 모든 대학에서 허용, Level 3은 교수에 따라 다름, Level 4~5는 거의 모든 대학에서 부정행위.

### 2024~2026 AI 5대 트렌드

AI 기술은 빠르게 변하고 있습니다. 대학생으로서 알아야 할 핵심 트렌드 5가지입니다.

#### 1. 멀티모달 AI의 본격화

텍스트, 이미지, 비디오, 오디오를 동시에 처리하는 AI가 2025년 프로덕션 수준에 도달했습니다.

**학생에게 의미하는 것:**
- PDF 논문 속 차트/그래프까지 AI가 분석 가능
- 강의 영상을 업로드하여 요약/노트 생성 가능

#### 2. 에이전트 AI — "AI가 알아서 일하는 시대"

2026년은 멀티 에이전트 시스템이 본격 프로덕션에 진입하는 해입니다. 에이전트 관련 문의량이 2024~2025년 사이 **1,445% 급증**했습니다.

- **ChatGPT**: Codex 에이전트 (코드 자동 작성/실행)
- **Claude**: Claude Code (터미널 기반 자율 코딩), Agent SDK
- **Gemini**: Jules (다중 에이전트 워크플로우)
- **Perplexity**: Comet 브라우저 (AI 기반 브라우징)

**학생에게 의미하는 것:** AI가 단순 답변을 넘어 "과제를 수행"하는 시대. 윤리적 경계 설정이 더욱 중요해집니다.

#### 3. 추론 모델 — "더 깊이 생각하는 AI"

OpenAI o3, Gemini 2.5 Deep Think 등 추론 특화 모델이 등장했습니다.

> ⚠️ **역설적 발견**: 추론 특화 모델(o3)이 팩트 벤치마크에서 **33% 할루시네이션률**을 기록했습니다 (이전 모델 16%의 2배). 복잡한 문제를 잘 풀지만, 지식 공백을 그럴듯한 추측으로 메우는 경향이 있습니다.

#### 4. 오픈소스 AI의 약진

오픈소스와 상용 모델의 성능 격차가 2024년 8%에서 2025년 **1.7%**로 급격히 축소되었습니다. DeepSeek-V3.2, Meta Llama 4 등 고성능 무료 모델이 다수 등장했습니다.

#### 5. MCP(Model Context Protocol) 표준화

AI 에이전트가 외부 도구·API에 연결하는 방식이 표준화되었습니다. AI 도구들이 서로 연동되기 쉬워져, "도구 조합 워크플로우"가 더 실용적으로 변하고 있습니다.

## 5. 따라하기 실습 (교재의 핵심)

### 🎯 실습: 자교 AI 사용 가이드라인 분석 보고서 만들기 ⭐⭐

**목표**: 자기 학교의 AI 사용 가이드라인을 찾아 분석하고, "어떤 상황에서 AI를 어떻게 써도 되는가?"를 판단하는 기준표를 만듭니다.
**소요 시간**: 30~40분
**준비물**: 웹 브라우저, Perplexity 계정 (무료 OK)
**사용 도구**: Perplexity (검색), ChatGPT 또는 Claude (분석 보조)

---

#### Step 1. 자기 학교의 AI 가이드라인 찾기 ⏱️ 10분

Perplexity에서 다음 프롬프트를 입력합니다

```
[자기 학교 이름] 생성형 AI 사용 가이드라인 또는 학술 윤리 지침을
찾아줘. 공식 문서나 공지사항 링크를 포함해줘.
```

> 📍 **화면 안내**: Perplexity(perplexity.ai)의 검색창에 입력합니다. 결과에 출처 링크가 숫자로 표시됩니다.

> 💡 **커스텀 포인트**: `[자기 학교 이름]`을 본인 학교로 바꾸세요. 예: "성균관대학교", "연세대학교", "한양대학교"

> 🔧 **안 되나요?**: 학교 가이드라인이 검색되지 않으면
> 1. 학교 홈페이지 → 학사공지 → "AI" 또는 "생성형" 검색
> 2. 학교 교무처/교육혁신원 홈페이지 확인
> 3. 정 없다면 교육부의 "대학 AI 활용 윤리 가이드라인"을 사용하세요

✅ **체크**: 가이드라인 문서(PDF 또는 웹페이지)를 찾았으면 성공입니다. 못 찾았다면 교육부 가이드라인을 대신 사용하세요.

---

#### Step 2. 가이드라인 핵심 내용 파악하기 ⏱️ 10분

찾은 가이드라인을 AI에게 분석 요청합니다. ChatGPT 또는 Claude에 다음 프롬프트를 입력하세요

```
다음은 [학교 이름]의 AI 사용 가이드라인입니다.
[가이드라인 내용 붙여넣기 또는 PDF 업로드]

이 가이드라인에서 다음 항목을 정리해줘:
1. AI 사용이 명시적으로 허용되는 경우
2. AI 사용이 명시적으로 금지되는 경우
3. 교수 재량에 맡기는 경우
4. AI 사용 시 반드시 지켜야 할 의무 (공개, 인용 등)
5. 위반 시 제재 내용

표 형식으로 정리해줘.
```

> 📍 **화면 안내**: PDF 가이드라인이라면 ChatGPT나 Claude의 파일 업로드 버튼(📎)을 클릭하여 업로드하세요. 웹페이지라면 내용을 복사하여 붙여넣으세요.

✅ **체크**: AI가 5개 항목을 표 형식으로 정리해주면 성공입니다. 이 표를 저장해두세요.

---

#### Step 3. 상황별 판단 기준표 만들기 ⏱️ 10분

다음 프롬프트를 이어서 입력합니다

```
위 가이드라인을 바탕으로, 다음 상황에서 AI를 사용해도 되는지 판단해줘:

1. 레포트 주제 브레인스토밍에 ChatGPT 사용
2. AI로 생성한 개요를 바탕으로 레포트 작성 (AI 사용 명시)
3. AI가 작성한 문단을 거의 수정 없이 레포트에 포함
4. 시험 준비를 위해 AI에게 예상 문제 생성 요청
5. 코딩 과제에서 AI에게 정답 코드 요청
6. 발표 스크립트를 AI로 생성하여 사용
7. AI가 제시한 참고문헌을 검증 없이 인용

각 상황에 대해 ✅허용 / ⚠️조건부 / ❌금지 로 판단하고,
근거를 가이드라인에서 인용해줘.
```

✅ **체크**: 7가지 상황에 대해 각각 판단과 근거가 제시되면 성공입니다.

> ⚠️ **주의**: AI의 판단이 항상 정확한 것은 아닙니다. 애매한 경우에는 반드시 **교수님이나 학과 사무실**에 직접 확인하세요.

---

#### Step 4. 나만의 "AI 활용 원칙서" 작성하기 ⏱️ 5분

Step 2~3의 결과를 종합하여, 아래 템플릿을 채워 "나만의 AI 활용 원칙서"를 완성하세요

```

학과: [학과]
작성일: 2026년 3월

### 1. 내가 다니는 학교의 AI 정책 핵심
- 허용: [정리 내용]
- 금지: [정리 내용]
- 교수 재량: [정리 내용]

### 2. 나의 AI 활용 원칙
1. AI는 [       ]의 용도로만 사용한다
2. AI를 사용할 때는 반드시 [       ]한다
3. AI의 결과물은 반드시 [       ]한다
4. 절대로 [       ]하지 않는다

### 3. 내가 주로 쓸 AI 도구와 데이터 보호 설정
- [도구 1]: 학습 비활성화 ✅/❌

```

✅ **체크**: 모든 빈칸이 채워지고, "나의 AI 활용 원칙" 4가지를 본인의 언어로 적었다면 성공입니다.

하지만 학생 측이 2개의 추가 AI 검사를 실시한 결과, 모두 **"인간 작성"**으로 판정되었습니다. 대학은 이 증거를 무시했고, 학생은 소송을 제기했습니다.

> **교훈**: AI 탐지 도구도 AI입니다. 맹신하면 무고한 학생이 피해를 받을 수 있습니다. 역으로, AI를 사용했다면 투명하게 밝히는 것이 가장 안전합니다.

## 7. 꿀팁 & 주의사항

### 전문가 팁

1. **수강 첫날 강의계획서 확인**: 교수마다 AI 정책이 다릅니다. 강의계획서에 AI 관련 조항이 있는지 반드시 확인하세요. 없다면 직접 교수에게 물어보세요.

2. **AI 사용 기록 남기기**: 어떤 도구에 어떤 프롬프트를 입력했고, 결과를 어떻게 수정했는지 기록해두면 나중에 문제가 생겼을 때 자신을 보호할 수 있습니다.

3. **데이터 보호 설정은 지금 당장**: 4개 도구 모두 첫 사용 시 학습 비활성화를 설정하세요. 나중에 하려면 이미 데이터가 수집된 후입니다.

### ❌ 흔한 실수: AI가 만들어준 참고문헌을 검증 없이 인용하기

**상황**: 레포트에 참고문헌이 필요해서 AI에게 요청했습니다.

**잘못된 방법**
```
디지털 트랜스포메이션 관련 참고문헌 10개를 APA 형식으로 만들어줘
```
→ AI가 그럴듯한 저자명, 논문 제목, 저널명, 연도를 포함한 참고문헌 목록을 생성합니다.
→ 이것을 그대로 레포트 참고문헌에 넣습니다.

**결과**: AI가 생성한 참고문헌의 상당수는 **실제로 존재하지 않는 가짜 논문**입니다. 교수가 확인하면 학술 사기로 간주될 수 있습니다. Deloitte가 작성한 호주 정부 보고서(A$440,000 비용)에서도 가짜 학술 출처가 발견되었습니다.

**올바른 방법**
```
디지털 트랜스포메이션과 중소기업에 관한 최신 연구를 찾아줘.
실제 존재하는 논문만 알려주고, 각 논문의 DOI 또는 URL을 포함해줘.
```
→ Perplexity에서 검색하면 실제 존재하는 논문과 출처 링크가 제공됩니다.
→ **제시된 링크를 하나하나 클릭**하여 논문이 실제로 존재하는지 확인합니다.

**차이**: Perplexity를 사용하고, 출처를 직접 확인하는 두 단계를 추가한 것만으로 가짜 참고문헌 위험을 제거했습니다.

### 학생을 위한 데이터 보호 체크리스트

```
✅ 모든 AI 도구에서 "학습/개선 참여" 옵션 비활성화하기
✅ 학번, 주민번호, 연락처 등 개인 식별 정보 절대 입력하지 않기
✅ 민감한 과제(개인 에세이, 의료/법률 관련)는 Temporary Chat 모드 사용
✅ 학교 계정과 개인 계정 분리하여 사용
✅ 정기적으로 대화 기록 삭제하기
❌ AI 채팅에 친구/동료의 개인정보 입력하지 않기
❌ 기밀 연구 데이터를 무료 플랜에 업로드하지 않기
```

### 🔄 이 워크플로우를 다른 상황에 적용하기

이 실습에서 배운 "가이드라인 검색 → AI 분석 → 상황별 판단표 → 원칙서 작성" 패턴은 다음 상황에도 적용할 수 있습니다

| 이 실습의 사례 | → 이렇게 바꾸면 | 다른 활용 |
|--------------|----------------|----------|
| 학교 AI 가이드라인 분석 | "기업 AI 사용 정책"으로 변경 | 인턴/취업 후 사내 AI 정책 파악 |
| 상황별 AI 사용 판단표 | "저작권 판단표"로 변경 | 졸업작품/포트폴리오의 AI 활용 범위 결정 |
| 나만의 AI 활용 원칙서 | "팀 프로젝트 AI 활용 계약서"로 변경 | Ch 5: 팀플에서 AI 사용 합의 |

**핵심 패턴**: "공식 정책을 찾아 → AI로 분석하고 → 상황별 판단 기준을 만들어 → 나만의 원칙으로 내면화한다."

## 1. CLAUDE.md로 프로젝트 규칙 설정 — 윤리적 사용 원칙 코드화

### 1-1. CLAUDE.md란?

`CLAUDE.md`는 프로젝트 루트에 놓는 설정 파일입니다. Claude Code가 대화를 시작할 때 이 파일을 자동으로 읽고, 파일에 적힌 규칙을 따릅니다. 즉, **AI의 행동 원칙을 코드처럼 관리**할 수 있습니다.

본문에서 배운 교육부 '대학 AI 활용 윤리 가이드라인' 5대 핵심원칙(학문적 진실성, 인간 중심성, 투명성, 공정성, 개인 보호)을 CLAUDE.md에 명시하면, AI가 매번 이 원칙을 자동으로 따르게 됩니다.

### 1-2. 윤리적 사용 원칙을 CLAUDE.md에 작성하기

프로젝트 폴더에서 다음과 같이 `CLAUDE.md`를 작성합니다

```markdown
# 프로젝트 규칙

## 윤리 원칙
- 출처가 불명확한 정보를 사실처럼 제시하지 않습니다.
- 다른 사람의 글을 그대로 복사하지 않고, 반드시 재구성합니다.
- 편향된 시각을 피하고 다양한 관점을 포함합니다.
- 개인정보가 포함된 데이터를 처리할 때 민감 정보를 마스킹합니다.

## 작성 규칙
- 한국어로 작성합니다.
- 학술적 어투를 사용하되, 대학생이 이해할 수 있는 수준을 유지합니다.
- 주장에는 반드시 근거를 함께 제시합니다.
```

### 1-3. CLAUDE.md 적용 확인하기

```bash
# 프로젝트 폴더로 이동
cd ~/my-project

# Claude Code 시작 — CLAUDE.md가 자동으로 로드됩니다
claude
```

```
> "AI가 일자리를 모두 대체할 것이다"라는 주장에 대한 글을 써줘

Claude: [CLAUDE.md 규칙에 따라 다양한 관점을 포함하여 작성합니다]
        "AI가 일자리에 미치는 영향에 대해서는 다양한 시각이 존재합니다.
         대체론 측에서는... 한편 보완론 측에서는... 실증 연구에 따르면..."
```

CLAUDE.md에 "편향된 시각을 피하고 다양한 관점을 포함합니다"라고 명시했기 때문에, Claude는 자동으로 균형 잡힌 서술을 합니다.

### 1-4. 계층적 규칙 설정

CLAUDE.md는 여러 수준에서 설정할 수 있습니다

```
~/.claude/CLAUDE.md          ← 전역 규칙 (모든 프로젝트에 적용)
~/my-project/CLAUDE.md       ← 프로젝트 규칙
~/my-project/src/CLAUDE.md   ← 하위 디렉터리 규칙
```

예를 들어, 전역 규칙에는 기본 윤리 원칙을, 프로젝트 규칙에는 해당 과제의 세부 지침을 넣을 수 있습니다.

**전역 CLAUDE.md 예시** (`~/.claude/CLAUDE.md`)

```markdown
# 전역 규칙
- 표절하지 않습니다.
- 확인되지 않은 통계 수치를 임의로 생성하지 않습니다.
- 한국어로 응답합니다.
```

**프로젝트 CLAUDE.md 예시** (`~/ai-ethics-report/CLAUDE.md`)

```markdown
# AI 윤리 보고서 프로젝트
- APA 인용 형식을 사용합니다.
- 2020년 이후 자료를 우선 인용합니다.
- 반론을 반드시 포함합니다.
```

---

### ❌ 흔한 실수: CLAUDE.md 작성 시 흔한 실수

**상황**: CLAUDE.md를 작성했지만 기대한 대로 동작하지 않을 때

---

**잘못된 방법 1 — 너무 모호한 규칙**
```markdown
# 규칙
- 좋은 글을 써주세요.
- 정확하게 해주세요.
```

**결과**: "좋은 글"과 "정확하게"는 AI가 해석할 기준이 없습니다. 모호한 지시는 없는 것과 같습니다.

**올바른 방법**
```markdown
# 규칙
- 모든 문단은 3~5문장으로 구성합니다.
- 통계 수치 인용 시 출처(저자, 연도)를 괄호 안에 표기합니다.
- "~인 것 같다" 대신 "~이다/~로 알려져 있다" 형태의 단정적 서술을 사용합니다.
```

**차이**: 구체적인 기준(문장 수, 인용 형식, 어투)이 있으면 AI가 일관되게 따릅니다.

---

**잘못된 방법 2 — 상충하는 지시**
```markdown
# 규칙
- 최대한 간결하게 작성합니다.
- 모든 주장에 대해 상세한 근거와 예시를 3개 이상 포함합니다.
```

**결과**: "간결하게"와 "상세한 근거 3개 이상"은 모순됩니다. AI가 어느 쪽을 우선해야 할지 혼란스러워 결과가 일관되지 않습니다.

**올바른 방법**
```markdown
# 규칙
- 핵심 주장은 간결하게 1~2문장으로 서술합니다.
- 각 주장 아래에 근거 1개를 간략히(1문장) 제시합니다.
```

**차이**: 우선순위가 명확하고 모순이 없는 규칙은 예측 가능한 결과를 만듭니다.

---

**잘못된 방법 3 — CLAUDE.md 위치 오류**
```bash
# 프로젝트 폴더가 ~/my-project인데,
# 홈 디렉터리에 CLAUDE.md를 넣음
~/CLAUDE.md          ← ❌ 잘못된 위치
~/my-project/CLAUDE.md  ← ✅ 올바른 위치
```

**결과**: Claude Code는 실행된 디렉터리와 그 상위 경로의 CLAUDE.md만 읽습니다. 엉뚱한 위치에 놓으면 무시됩니다.

**올바른 방법**: 반드시 프로젝트 루트 디렉터리에 `CLAUDE.md`를 배치하세요.

## 2. 프롬프트에 윤리 가이드라인 내장하기

### 2-1. 시스템 프롬프트로 윤리 기준 설정

CLAUDE.md 외에도, 개별 대화에서 윤리 기준을 직접 명시할 수 있습니다.

```
> 다음 규칙을 지켜서 답변해줘:
> 1. 출처를 명시할 수 없는 정보는 "확인이 필요합니다"라고 표시
> 2. 통계 수치를 인용할 때는 출처와 연도를 함께 표시
> 3. 특정 입장을 지지하지 말고 양쪽 논거를 균형있게 제시
>
> 질문: 대학에서 AI 사용을 허용해야 하는가?

Claude: [규칙에 따라 균형 잡힌 답변을 작성합니다]

        **AI 사용 허용 찬성 측 논거:**
        - ...  (출처: UNESCO, 2023)

        **AI 사용 허용 반대 측 논거:**
        - ...  (출처: Nature, 2024)

        **참고:** 각 대학의 학칙 규정은 확인이 필요합니다.
```

### 2-2. 비대화형 모드에서 윤리 가이드라인 적용

자동화 스크립트에서도 윤리 규칙을 포함할 수 있습니다

```bash
claude -p "
다음 규칙을 반드시 지켜서 답변하세요:
- 확인되지 않은 사실은 '미확인'으로 표시
- 인용 시 출처 명시

```

### 2-3. 윤리 체크리스트 자동화

과제를 제출하기 전에, Claude Code로 윤리 체크를 자동화할 수 있습니다

```
> 내 보고서(report.md)를 읽고 다음 항목을 점검해줘:
> 1. 출처 없이 사용된 통계 수치가 있는가?
> 2. 특정 집단에 대한 편향적 표현이 있는가?
> 3. 표절 의심 구간이 있는가? (너무 일반적이지 않은 구체적 문장)
> 4. AI가 생성한 내용임을 밝혀야 하는 부분이 있는가?
> 결과를 체크리스트 형식으로 정리해줘.

Claude: [report.md를 분석합니다]

        ## 윤리 점검 결과

        ✅ 출처 없는 통계: 2건 발견
           - 3페이지: "AI 시장은 2030년까지 1.8조 달러 규모" → 출처 필요
           - 7페이지: "대학생 78%가 AI를 사용" → 출처 필요

        ✅ 편향적 표현: 해당 없음

        ⚠️ 표절 의심: 1건
           - 5페이지: "인공지능은 인간의 지능을..." → 매우 일반적인 정의이나,
             원문 확인 권장

        ✅ AI 생성 표시: 결론부에 AI 활용 사실 명시 필요
```

### 🔬 프롬프트 진화 실험: 윤리 체크리스트 요청하기

같은 작업을 3단계 프롬프트로 시도하여, 프롬프트 품질의 차이를 직접 체감합니다.

#### 🔴 Level 1 — 초보 프롬프트

```
내 보고서 검토해줘
```

**결과**: AI가 문법, 논리, 구성 등 무엇을 검토해야 하는지 임의로 판단합니다. 윤리 점검은 빠질 가능성이 높습니다.

📋 **평가**
- 정확성: ★★☆☆☆
- 활용도: ★☆☆☆☆
- 문제점: 검토 기준이 없어 AI가 자의적으로 판단. 윤리 항목 누락.

---

#### 🟡 Level 2 — 중급 프롬프트


## 3. AI 응답의 팩트체킹을 Claude Code로 자동화하는 방법

> Ch 1 본문에서 "AI 답변 교차검증 5단계 워크플로우"를 배웠고, Ch 2 본문에서 변호사 486건의 AI 할루시네이션 사례를 확인했습니다. Claude Code를 사용하면 이 교차검증을 **자동화**하여, 여러 문서를 한 번에 검증할 수 있습니다.

### 3-1. 기본 팩트체킹 워크플로우

AI가 생성한 텍스트를 다시 AI에게 검증시키는 "자기 검증" 패턴입니다

```
> 다음 텍스트의 사실 관계를 검증해줘.
> 각 주장에 대해 (1) 사실/미확인/오류 판정, (2) 근거를 제시해줘.
>
> 텍스트:
> "트랜스포머 아키텍처는 2015년 구글이 발표했으며,
>  GPT-4는 1조 개의 파라미터를 가지고 있다."

Claude:
        | 주장 | 판정 | 근거 |
        |------|------|------|
        | 트랜스포머 2015년 구글 발표 | ❌ 오류 | 트랜스포머는 2017년 논문 "Attention Is All You Need"에서 발표 |
        | GPT-4 파라미터 1조 개 | ⚠️ 미확인 | OpenAI는 GPT-4의 정확한 파라미터 수를 공식 발표하지 않음 |
```

### 3-2. 웹 검색을 활용한 팩트체킹

Claude Code에는 웹 검색 도구(`WebSearch`, `WebFetch`)가 내장되어 있어, 대화 중 실시간으로 웹 정보를 조회할 수 있습니다. 팩트체킹을 요청하면 Claude Code가 자동으로 웹 검색을 수행하여 주장을 교차 검증합니다.

> ⚠️ **주의**: 웹 검색은 Claude Code가 해당 도구를 사용할 수 있도록 권한이 설정된 경우에만 작동합니다. 처음 사용 시 도구 사용 허가 프롬프트가 나타날 수 있습니다.

```
> 내 보고서(report.md)에서 인용된 통계와 사실을 웹 검색으로 교차 검증해줘.
> 각 항목의 정확성을 판단하고, 잘못된 부분은 올바른 정보로 수정 제안해줘.

Claude: [report.md를 분석합니다]
        [웹 검색으로 각 주장을 검증합니다]

        ## 팩트체킹 결과

        1. "전 세계 AI 시장 규모 1.8조 달러 (2030년)"
           → ✅ 사실 — 출처: Statista, 2024년 보고서와 유사한 수치

        2. "ChatGPT 출시일 2022년 12월"
           → ❌ 수정 필요 — 정확한 출시일: 2022년 11월 30일

        3. "한국 대학생 AI 사용률 85%"
           → ⚠️ 출처 불명 — 해당 수치의 원 출처를 찾을 수 없음.
             가장 유사한 조사: [구체적 조사명]에서 XX% 보고
```

### 3-3. 자동화 팩트체킹 스크립트

여러 문서를 일괄 검증하는 스크립트입니다. Claude Code에게 파일 경로를 직접 전달하여 읽게 하는 방식이 안정적입니다

```bash
#!/bin/bash
# factcheck.sh — 폴더 내 모든 마크다운 파일 팩트체킹

mkdir -p factcheck_results

for file in drafts/*.md; do
  filename=$(basename "$file" .md)
  claude -p "
    $file 을 읽고 사실 관계를 검증하세요.
    각 주장에 대해 사실/미확인/오류로 판정하고 근거를 제시하세요.
    표 형식으로 정리하세요.
  " > "factcheck_results/${filename}_checked.md"

  echo "✓ $file 검증 완료"
done

echo "모든 파일 팩트체킹 완료. 결과: factcheck_results/"
```

> ⚠️ **주의**: 이전 버전에서는 `$(cat "$file")`로 파일 내용을 인라인으로 넘기는 방식을 사용했지만, 대용량 파일에서 셸 인자 길이 제한에 걸릴 수 있습니다. 위처럼 Claude Code에게 파일 경로를 전달하여 직접 읽게 하는 것이 더 안정적입니다.

### 3-4. 출처 자동 수집

보고서에 인용된 주장들의 출처를 자동으로 찾아주는 활용법

```
> report.md를 읽고, 출처가 명시되지 않은 모든 주장을 찾아서
> 각각에 대해 적절한 학술 출처를 검색하여 추천해줘.
> 결과를 references.md로 저장해줘.

Claude: [report.md를 분석합니다]
        [출처 미명시 주장 7건 발견]
        [각 주장에 대해 관련 출처를 검색합니다]
        [references.md를 생성했습니다]
```

## 4. 하네스 심화: 윤리적 AI 사용을 위한 에이전트 팀

> 본문에서 "에이전트 AI"가 2026년 핵심 트렌드임을 배웠습니다. Claude Code의 하네스는 이 에이전트 AI를 **윤리적 검수 워크플로우**에 적용하는 구체적 방법입니다. 본문의 "변호사 486건 AI 할루시네이션 사례"에서 보았듯, 검증 없는 AI 사용은 전문가도 위험합니다. 에이전트 팀으로 자동 검수 체계를 구축하면 이 위험을 체계적으로 줄일 수 있습니다.

### 4-1. 윤리 검수 에이전트 팀 구성 예시

복잡한 보고서를 작성할 때, 역할을 나눈 에이전트 팀을 구성할 수 있습니다

```
프로젝트 구조:
.claude/
  agents/
    writer.md        ← 본문 작성 에이전트
    fact-checker.md  ← 팩트체킹 전담 에이전트
    ethics-reviewer.md ← 윤리 검수 에이전트
```

**writer.md** (작성 에이전트)
```markdown
---
name: writer
description: "보고서 본문을 작성하는 에이전트"
---
# Writer
- 주어진 주제에 대해 본문을 작성합니다.
- 모든 주장에 출처를 표기합니다.
- 완료 후 fact-checker에게 검수를 요청합니다.
```

**fact-checker.md** (팩트체킹 에이전트)
```markdown
---
name: fact-checker
description: "작성된 내용의 사실 관계를 검증하는 에이전트"
---
# Fact Checker
- writer가 작성한 본문의 모든 사실 주장을 검증합니다.
- 오류 발견 시 수정 제안을 작성합니다.
- 검증 완료 후 ethics-reviewer에게 전달합니다.
```

**ethics-reviewer.md** (윤리 검수 에이전트)
```markdown
---
name: ethics-reviewer
description: "콘텐츠의 윤리적 적절성을 검토하는 에이전트"
---
# Ethics Reviewer
- 편향성, 차별적 표현, 개인정보 노출 여부를 점검합니다.
- AI 생성 콘텐츠 표시 여부를 확인합니다.
- 표절 가능성을 검토합니다.
- 최종 검수 결과를 보고서에 첨부합니다.
```

이렇게 구성하면, 한 번의 명령으로 작성 → 팩트체킹 → 윤리 검수가 순차적으로 이루어집니다. 마치 실제 편집팀처럼 AI가 역할을 나누어 작업합니다.

### 4-2. 왜 에이전트 팀이 유용한가?

1. **품질 향상**: 작성자와 검수자를 분리하면 "자기 글의 오류를 못 찾는" 문제를 줄임
2. **일관성**: 매번 같은 기준으로 검수가 이루어짐
3. **확장성**: 새로운 검수 기준(예: 접근성 검토)을 에이전트 하나 추가로 해결
4. **재사용**: 한 번 만든 팀 구성을 다른 프로젝트에서도 활용 가능

### 🎯 실습: AI 생성 텍스트 팩트체킹 ⭐⭐

**목표**: AI가 생성한 글의 사실 관계를 같은 AI로 검증하는 워크플로우를 체험합니다.
**소요 시간**: 15분
**준비물**: Claude Code 설치 완료
**사용 도구**: Claude Code

---

#### Step 1. AI에게 글 생성 요청 ⏱️ 3분

```
"인공지능의 역사"라는 주제로 5가지 핵심 사건을 포함한 200자 글을 써줘.
연도와 사실을 포함해서 작성해.
결과를 ai_history.md로 저장해줘.
```

✅ **체크**: `ai_history.md` 파일이 생성되고, 연도와 사건이 포함된 글이 나오면 성공

---

#### Step 2. 팩트체킹 요청 ⏱️ 7분

```
ai_history.md를 읽고 사실 관계를 검증해줘.
각 주장에 대해:
- 판정: 사실 / 미확인 / 오류
- 근거: 왜 그렇게 판정했는지
표 형식으로 정리하고, 결과를 factcheck_result.md로 저장해줘.
```

✅ **체크**: 표 형식의 팩트체킹 결과가 생성되고, 최소 1개 이상의 "미확인" 또는 "오류" 판정이 포함되면 성공 (AI 생성 텍스트에는 거의 항상 부정확한 부분이 포함됩니다)

---

#### Step 3. 결과 분석 ⏱️ 5분

AI의 팩트체킹 결과에서 다음을 확인합니다

1. "오류"로 판정된 항목이 실제로 잘못된 정보인가?
2. "미확인"으로 판정된 항목은 왜 확인이 안 되는가?
3. "사실"로 판정된 항목도 정말 맞는지 독립적으로 확인할 수 있는가?

> ⚠️ **주의**: AI의 팩트체킹도 100% 정확하지 않습니다. AI 팩트체킹은 "1차 필터"이며, 중요한 사실은 반드시 원 출처에서 직접 확인해야 합니다.
>
> 🔧 **안 되나요?**: 모든 항목이 "사실"로 판정되었다면, 의도적으로 잘못된 정보를 포함한 텍스트를 만들어 테스트해보세요. 예: "GPT-1은 2020년에 출시되었다"

AI 생성 → AI 검증의 "이중 검토" 워크플로우를 체험했습니다.

> 📊 **효과**
> - 기존 방식: 각 사실을 하나씩 검색 엔진에서 확인 — 사실 5개 기준 25분
> - AI 활용: 한 번의 요청으로 전체 텍스트 검증 — 3분
> - 절약: 약 22분 (단, AI 검증은 1차 필터이므로 핵심 사실은 직접 확인 권장)

---

### 🎯 실습: 팩트체킹 일괄 자동화 스크립트 ⭐⭐⭐

**목표**: 여러 파일을 한 번에 팩트체킹하는 셸 스크립트를 작성하고 실행합니다.
**소요 시간**: 30분
**준비물**: Claude Code 설치 완료, 앞선 실습에서 만든 파일들 (없으면 새로 생성)
**사용 도구**: Claude Code, bash

---

#### Step 1. 테스트 파일 준비 ⏱️ 5분

```bash
mkdir -p ~/ethics-practice/drafts ~/ethics-practice/factcheck_results
claude -p "인공지능 윤리에 대한 주장 3가지를 연도와 통계를 포함해 작성해줘" > ~/ethics-practice/drafts/ethics.md
claude -p "AI와 저작권에 대한 주장 3가지를 연도와 통계를 포함해 작성해줘" > ~/ethics-practice/drafts/copyright.md
claude -p "AI 편향성에 대한 주장 3가지를 연도와 통계를 포함해 작성해줘" > ~/ethics-practice/drafts/bias.md
```

✅ **체크**: `ls ~/ethics-practice/drafts/`로 3개 파일이 생성되었는지 확인

---

#### Step 2. 팩트체킹 스크립트 작성 ⏱️ 10분

다음 내용을 `~/ethics-practice/factcheck.sh`로 저장합니다

```bash
#!/bin/bash
# factcheck.sh — drafts 폴더의 모든 마크다운 파일을 일괄 팩트체킹

mkdir -p factcheck_results

for file in drafts/*.md; do
  filename=$(basename "$file" .md)
  echo "🔍 $file 검증 중..."
  claude -p "
    $file 을 읽고 사실 관계를 검증하세요.
    각 주장에 대해 사실/미확인/오류로 판정하고 근거를 제시하세요.
    표 형식으로 정리하세요.
  " > "factcheck_results/${filename}_checked.md"
  echo "✓ $file 검증 완료 → factcheck_results/${filename}_checked.md"
done

echo "🎉 모든 파일 팩트체킹 완료!"
```

```bash
chmod +x ~/ethics-practice/factcheck.sh
```

✅ **체크**: `cat ~/ethics-practice/factcheck.sh`로 스크립트 내용 확인

---

#### Step 3. 스크립트 실행 및 결과 확인 ⏱️ 15분

```bash
cd ~/ethics-practice && ./factcheck.sh
```

✅ **체크**: `factcheck_results/` 폴더에 3개의 검증 결과 파일이 생성되면 성공

> ⚠️ **주의**: 각 파일 처리에 API 호출이 발생합니다. 실습에서는 3개 이내로 테스트하세요.
>
> 🔧 **안 되나요?**: `permission denied` → `chmod +x factcheck.sh`로 실행 권한을 부여하세요.

---

#### 🎉 완성! — 결과물 미리보기

셸 스크립트 하나로 여러 문서의 팩트체킹을 자동화했습니다.

> 📊 **효과**
> - 기존 방식: 문서 3개를 각각 수동으로 사실 확인 — 90분 이상
> - AI 활용: 스크립트 한 번 실행 — 10분
> - 절약: 약 80분, 문서가 많을수록 효과 급증

### 🔄 이 워크플로우를 다른 상황에 적용하기

이 실습에서 배운 패턴은 다음 상황에도 적용할 수 있습니다

| 이 실습의 사례 | → 이렇게 바꾸면 | 다른 활용 |
|--------------|----------------|----------|
| 팩트체킹 | "맞춤법과 문법 검사"로 변경 | 보고서 교정 자동화 |
| 마크다운 파일 | PDF 파일로 변경 | 논문 검증 자동화 |
| 사실 판정 표 | "APA 인용 형식 점검"으로 변경 | 참고문헌 형식 검사 |

**핵심 패턴**: CLAUDE.md로 기준을 설정하고, `-p` 플래그로 일괄 자동화 — 이것이 "윤리의 코드화 + 자동화"입니다.

---

> **핵심 정리**: CLAUDE.md는 "AI에게 주는 행동 강령"입니다. 윤리 원칙을 파일로 명시하면, 매번 프롬프트에 반복 입력하지 않아도 AI가 일관되게 원칙을 지킵니다. 이것이 "윤리의 코드화"입니다.
