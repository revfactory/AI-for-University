## 1. 전공 맞춤 에이전트 설계

### 1-1. 왜 전공별 에이전트가 필요한가?

Ch 5에서 범용 에이전트 팀(researcher-writer-fact-checker-editor)을 배웠습니다. 본문에서도 전공별 AI 도구 매핑(인문학→Claude, 사회과학→ChatGPT, 공학→Claude+Copilot 등)을 배웠는데, 에이전트 설계에도 같은 원리가 적용됩니다 — 전공마다 필요한 AI 작업이 다릅니다:

| 전공 계열 | 핵심 작업 | 필요한 전문 에이전트 |
|----------|----------|-------------------|
| **인문/사회** | 문헌 분석, 비판적 논증 | 문헌분석가, 논증검토자 |
| **경영/경제** | 데이터 분석, 시장 조사 | 데이터분석가, 시장조사원 |
| **이공계** | 코드 작성, 실험 설계 | 코더, 실험설계자 |
| **예술/디자인** | 컨셉 개발, 비평 | 컨셉디자이너, 크리틱 |
| **교육** | 교안 설계, 평가 문항 | 교안설계자, 문항개발자 |

### 1-2. 전공 맞춤 에이전트 설계 원칙

1. **전공 용어**: 해당 분야의 전문 용어와 방법론을 명시
2. **도메인 규칙**: 분야 특유의 작업 규칙을 포함 (예: 의학 - "진단 조언 금지")
3. **출력 형식**: 전공에서 사용하는 표준 형식 지정 (예: APA, IEEE, MLA)
4. **품질 기준**: 해당 분야의 학술적 기준 명시

## 2. 전공별 에이전트 팀 예시

### 2-1. 경영학 — 사업 분석 에이전트 팀

```
.claude/
  agents/
    market-analyst.md    ← 시장/산업 분석
    financial-analyst.md ← 재무 분석
    strategy-advisor.md  ← 전략 제안
    report-writer.md     ← 보고서 통합 작성
```

**market-analyst.md**:
```markdown
---
name: market-analyst
description: "시장 규모, 경쟁 환경, 트렌드를 분석하는 에이전트"
---
# Market Analyst — 시장 분석가

## 핵심 역할
- 대상 시장의 규모, 성장률, 주요 트렌드를 분석합니다.
- 경쟁사 현황과 경쟁 구도를 정리합니다.
- Porter의 5 Forces, PEST 분석 프레임워크를 활용합니다.

## 작업 원칙
- 모든 통계에 출처(기관명, 연도)를 포함합니다.
- 한국 시장 데이터를 우선 수집합니다.
- 정량 데이터(숫자, 비율)와 정성 데이터(트렌드, 전망)를 균형있게 포함합니다.

## 출력
- market_analysis.md: 시장 분석 보고서

## 협업
- 완료 후 financial-analyst에게 전달합니다.
- strategy-advisor에게 시장 데이터를 제공합니다.
```

### 2-2. 컴퓨터과학 — 코딩 프로젝트 에이전트 팀

```
.claude/
  agents/
    architect.md    ← 시스템 설계
    coder.md        ← 코드 작성
    tester.md       ← 테스트 작성/실행
    reviewer.md     ← 코드 리뷰
```

**coder.md**:
```markdown
---
name: coder
description: "설계를 바탕으로 실제 코드를 작성하는 에이전트"
---
# Coder — 개발자

## 핵심 역할
- architect의 설계 문서를 바탕으로 코드를 작성합니다.
- 함수 단위로 분리하여 모듈화합니다.
- 코드에 적절한 주석을 포함합니다.

## 작업 원칙
- PEP 8 (Python) 또는 해당 언어의 코딩 컨벤션을 따릅니다.
- 에러 핸들링을 포함합니다.
- 각 함수에 간단한 docstring을 작성합니다.

## 출력
- src/ 폴더에 소스 코드 파일 생성

## 협업
- 완료 후 tester에게 테스트를 요청합니다.
- reviewer의 피드백을 받아 코드를 수정합니다.
```

### 2-3. 교육학 — 수업 설계 에이전트 팀

```
.claude/
  agents/
    curriculum-designer.md  ← 교육과정 설계
    content-creator.md      ← 학습 자료 제작
    assessment-designer.md  ← 평가 문항 개발
    accessibility-checker.md ← 접근성 검토
```

**assessment-designer.md**:
```markdown
---
name: assessment-designer
description: "학습 목표에 맞는 평가 문항을 개발하는 에이전트"
---
# Assessment Designer — 평가 설계자

## 핵심 역할
- 학습 목표에 aligned된 평가 문항을 개발합니다.
- Bloom의 분류 체계에 따라 인지 수준별 문항을 배치합니다.
- 선택형, 서술형, 수행평가를 균형있게 포함합니다.

## 작업 원칙
- 각 문항에 해당 학습 목표와 인지 수준을 태그합니다.
- 채점 기준(rubric)을 함께 제공합니다.
- 난이도(상/중/하)를 표시합니다.

## 출력
- assessments/ 폴더에 평가 문항 파일 생성

## 협업
- curriculum-designer의 학습 목표를 참조합니다.
- accessibility-checker에게 문항의 접근성 검토를 요청합니다.
```

```

사업 분석 보고서를 작성해줘. Porter의 5 Forces를 활용해줘.
```

**결과**: 단일 AI가 분석부터 보고서 작성까지 모두 처리합니다. 각 분석이 깊지 않고, 재무 데이터나 전략 제안이 피상적입니다.

📋 **평가**:

- 활용도: ★★★☆☆
- 개선점: 주제, 프레임워크가 지정됨
- 아쉬운 점: 시장 분석/재무 분석/전략 제안이 구분되지 않아 각각의 깊이가 얕음

---

#### 🟢 Level 3 — 전문가 프롬프트 (전공 맞춤 에이전트 팀)

```
.claude/agents/의 경영학 에이전트 팀을 활용하여
"한국 커피 시장의 경쟁 구도 분석" 보고서를 작성해줘.

1. market-analyst: Porter의 5 Forces + PEST 분석으로 시장 분석
   → market_analysis.md
2. financial-analyst: 주요 기업(스타벅스코리아, 이디야, 메가커피)의
   매출·영업이익 비교 → financial_analysis.md
3. strategy-advisor: 분석 결과 기반 전략 제안 3개
   → strategy_recommendations.md
4. report-writer: 전체를 통합하여 최종 보고서
   → final/business_report.md
```

**결과**: 4개 전문 에이전트가 각자의 영역을 깊이 있게 분석하고, 통합 보고서가 완성됩니다. 시장/재무/전략이 각각 충분한 깊이를 가집니다.

📋 **평가**:
- 정확성: ★★★★★
- 활용도: ★★★★★
- 핵심 차이: 전공 특화 에이전트가 각 분야를 깊이 있게 분석

---

**잘못된 방법 — 범용 에이전트를 그대로 사용**:

Ch 5의 researcher-writer-fact-checker-editor를 모든 전공에 그대로 적용하면, 전공 특유의 분석 방법론이나 평가 기준이 반영되지 않습니다.

**올바른 방법**: 전공에 맞게 에이전트의 역할명, 사용 프레임워크, 출력 형식을 커스터마이즈하세요.

| 범용 에이전트 | → 경영학 맞춤 | → 교육학 맞춤 |
|-------------|------------|------------|
| researcher | market-analyst (Porter, PEST) | curriculum-designer (Bloom 분류) |
| writer | report-writer (사업 보고서 형식) | content-creator (학습 자료 형식) |
| fact-checker | financial-analyst (재무 데이터 검증) | assessment-designer (문항 타당도) |
| editor | strategy-advisor (전략 제안) | accessibility-checker (접근성) |

**차이**: 전공 특화 에이전트는 해당 분야의 언어, 프레임워크, 품질 기준을 내장하여 더 전문적인 결과를 생성합니다. 본문의 GitHub Copilot 연구가 보여주듯, AI 결과를 "세분화 채택"(조각 단위로 검토하여 선택적 수용)하는 것이 "통째 채택"보다 효과적입니다 — 에이전트의 결과물도 마찬가지입니다.

## 3. 하네스 구성 기초

### 3-1. 하네스 구성의 전체 그림

지금까지 배운 Claude Code의 모든 기능을 통합하면:

```
~/my-project/
├── CLAUDE.md                    ← 프로젝트 규칙 (Ch 2)
├── .claude/
│   └── agents/                  ← 에이전트 팀 (Ch 5~6)
│       ├── agent-1.md
│       ├── agent-2.md
│       └── agent-3.md
├── input/                       ← 입력 자료
├── _workspace/                  ← 중간 산출물
└── output/                      ← 최종 결과물
```

이 구조가 바로 **하네스(Harness)**입니다:
- **CLAUDE.md**: 전체 프로젝트의 규칙과 원칙
- **agents/**: 각 역할의 전문가 정의
- **폴더 구조**: 입력-작업-출력의 체계적 관리

### 3-2. 나만의 하네스 설계 체크리스트

전공 맞춤 하네스를 설계할 때 다음 체크리스트를 활용하세요:

```
□ 1. 프로젝트 목표가 명확한가?
□ 2. 필요한 역할이 모두 정의되었는가?
□ 3. 각 에이전트의 입력/출력이 명확한가?
□ 4. 에이전트 간 협업 흐름이 정의되었는가?
□ 5. CLAUDE.md에 전공 특화 규칙이 포함되었는가?
□ 6. 품질 검증 단계가 포함되었는가?
□ 7. 폴더 구조가 워크플로우를 반영하는가?
```

```
내 전공([전공명])의 전문 용어와 방법론을 반영해줘.
```

✅ **체크**: `.claude/agents/` 폴더에 3~4개의 에이전트 파일이 생성되면 성공

> ⚠️ **주의**: AI가 생성한 에이전트 정의를 그대로 사용하지 말고, 본인의 전공 지식으로 검토하고 수정하세요. 특히 전문 용어와 방법론이 정확한지 확인하세요.
