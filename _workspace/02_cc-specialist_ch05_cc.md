## 1. 에이전트 팀이란?

### 1-1. 개념 이해

Ch 3~4에서 특정 작업용 에이전트 팀을 경험했습니다. 이번 챕터에서는 에이전트 팀 설계의 원리와 패턴을 체계적으로 학습합니다.

Ch 2에서 "에이전트 AI"를 2026년 핵심 트렌드로 소개했습니다. Claude Code의 **하네스(Harness)**는 이 에이전트 개념을 실제로 구현하는 기능입니다.

에이전트 팀이란:
- 각각 **고유한 역할과 전문성**을 가진 AI 에이전트 여러 개를 정의
- `.claude/agents/` 폴더에 마크다운 파일로 역할을 기술
- 팀 리더가 작업을 조율하여 **자동으로 분업**

본문에서 4명의 팀원에게 강점 기반으로 역할을 분담했던 것을 기억하세요(A: 자료조사, B: 디자인, C: 발표, D: 데이터). 에이전트 팀도 동일한 원리입니다 — AI 에이전트에게 "researcher", "writer", "fact-checker", "editor" 같은 명확한 역할을 부여하면 품질이 향상됩니다.

### 1-2. 에이전트 정의 파일 구조

각 에이전트는 `.claude/agents/` 폴더에 마크다운 파일로 정의합니다:

```markdown
---
name: 에이전트이름
description: "이 에이전트가 하는 일을 한 줄로 설명"
---

# 역할명

## 핵심 역할
- 이 에이전트가 수행하는 구체적 작업들

## 작업 원칙
- 작업 시 지켜야 할 규칙들

## 입력/출력
- 어떤 파일을 읽고, 어떤 파일을 생성하는지

## 협업
- 다른 에이전트와 어떻게 소통하는지
```

## 2. 조별과제 분업 시뮬레이션: 에이전트 팀 설계

### 2-1. 시나리오: "AI 윤리 보고서" 조별과제

4명이 팀을 이루어 "AI 윤리와 대학 교육"에 대한 보고서를 작성하는 상황을 에이전트 팀으로 시뮬레이션합니다.

```
.claude/
  agents/
    researcher.md      ← 자료 조사 담당
    writer.md          ← 본문 작성 담당
    fact-checker.md    ← 사실 검증 담당
    editor.md          ← 편집·교정 담당
```

### 2-2. 각 에이전트 정의

**researcher.md** (자료 조사):
```markdown
---
name: researcher
description: "주제 관련 자료를 조사하고 핵심 정보를 정리하는 에이전트"
---
# Researcher — 자료 조사 담당

## 핵심 역할
- 주어진 주제에 대해 웹 검색으로 최신 자료를 수집합니다.
- 학술 논문, 보도자료, 공식 통계를 우선합니다.
- 수집한 자료를 구조화된 형식으로 정리합니다.

## 작업 원칙
- 모든 정보에 출처(URL, 저자, 연도)를 반드시 포함합니다.
- 출처를 확인할 수 없는 정보는 "[미확인]"으로 표시합니다.
- 최신 자료(2024년 이후)를 우선합니다.

## 출력
- research_notes.md: 수집한 자료 정리
- sources.md: 참고문헌 목록

## 협업
- 완료 후 writer에게 전달합니다.
- writer로부터 추가 자료 요청을 받을 수 있습니다.
```

**writer.md** (본문 작성):
```markdown
---
name: writer
description: "조사 자료를 바탕으로 보고서 본문을 작성하는 에이전트"
---
# Writer — 본문 작성 담당

## 핵심 역할
- researcher의 자료를 바탕으로 보고서 각 섹션을 작성합니다.
- 서론-본론-결론 구조를 따릅니다.
- 논리적 흐름을 유지합니다.

## 작업 원칙
- CLAUDE.md의 작성 규칙을 따릅니다.
- researcher가 수집한 출처를 반드시 인용합니다.
- 자신의 의견과 자료 기반 주장을 명확히 구분합니다.

## 출력
- sections/ 폴더에 섹션별 파일 생성

## 협업
- researcher에게 추가 자료를 요청할 수 있습니다.
- 완료 후 fact-checker에게 전달합니다.
```

**fact-checker.md** (사실 검증):
```markdown
---
name: fact-checker
description: "보고서의 사실 관계와 출처를 검증하는 에이전트"
---
# Fact Checker — 사실 검증 담당

## 핵심 역할
- writer가 작성한 본문의 모든 사실 주장을 검증합니다.
- 인용된 출처가 실제로 존재하는지 확인합니다.
- 통계 수치의 정확성을 교차 검증합니다.

## 작업 원칙
- 각 주장에 대해 사실/미확인/오류로 판정합니다.
- 오류 발견 시 올바른 정보와 출처를 제시합니다.
- 출처가 없는 주장은 반드시 표시합니다.

## 출력
- factcheck_report.md: 검증 결과 보고서

## 협업
- 검증 결과를 editor에게 전달합니다.
- 심각한 오류 발견 시 writer에게 수정을 요청합니다.
```

**editor.md** (편집·교정):
```markdown
---
name: editor
description: "보고서의 문법, 흐름, 형식을 교정하고 최종본을 완성하는 에이전트"
---
# Editor — 편집·교정 담당

## 핵심 역할
- 전체 보고서의 문법, 맞춤법, 어투 일관성을 점검합니다.
- 섹션 간 논리적 흐름을 확인하고 전환 문장을 보강합니다.
- fact-checker의 피드백을 반영합니다.
- 최종 통합본을 완성합니다.

## 작업 원칙
- CLAUDE.md의 작성 규칙 준수 여부를 최종 확인합니다.
- 불필요한 반복을 제거합니다.
- 목차, 참고문헌, 페이지 구성을 정리합니다.

## 출력
- final/report.md: 최종 보고서

## 협업
- fact-checker의 검증 결과를 반영합니다.
- 수정이 필요하면 writer에게 피드백을 보냅니다.
```

### 2-3. 워크플로우 다이어그램

```
researcher → writer → fact-checker → editor → 최종 보고서
    ↑                      |
    └── 추가 자료 요청 ←───┘
```

#### 🟢 Level 3 — 전문가 프롬프트 (에이전트 팀 활용)

```
.claude/agents/ 에 정의된 에이전트 팀을 활용하여
"AI 윤리와 대학 교육" 보고서를 작성해줘.

작업 순서:
1. researcher: 최신 자료 5개 이상 수집 → research_notes.md
2. writer: 자료 기반으로 서론-본론-결론 작성 → sections/
3. fact-checker: 모든 주장과 출처 검증 → factcheck_report.md
4. editor: 최종 교정 및 통합 → final/report.md

각 단계의 결과물을 파일로 저장하고,
다음 에이전트에게 이전 결과물을 참조하게 해줘.
```

**결과**: 4개 에이전트가 순차적으로 작업하여, 자료 조사-작성-검증-교정이 체계적으로 이루어집니다. 각 단계의 결과물이 파일로 남아 추적 가능합니다.

📋 **평가**:
- 정확성: ★★★★★
- 활용도: ★★★★★
- 핵심 차이: 역할 분리 + 파일 기반 결과물 + 검증 단계 포함

### ❌ 흔한 실수: 에이전트 팀 설계 시 흔한 실수

**상황**: 에이전트 팀을 처음 구성할 때

---

**잘못된 방법 1 — 역할이 모호한 에이전트**:
```markdown
---
name: helper
description: "도움을 주는 에이전트"
---
# Helper
- 필요한 것을 도와줍니다.
```

**결과**: "도움을 준다"는 것이 구체적으로 무엇인지 알 수 없어, AI가 역할을 임의로 해석합니다.

**올바른 방법**:
```markdown
---
name: fact-checker
description: "보고서의 사실 관계와 출처를 검증하는 에이전트"
---
# Fact Checker
- writer가 작성한 본문의 모든 사실 주장을 검증합니다.
- 각 주장에 대해 사실/미확인/오류로 판정합니다.
```

**차이**: 역할명, 설명, 구체적 작업 내용이 명확할수록 AI가 정확한 역할을 수행합니다.

---

**잘못된 방법 2 — 에이전트 간 소통 방법 미정의**:

에이전트의 협업 섹션이 빠지면, 각 에이전트가 독립적으로 작업하여 결과물이 연결되지 않습니다.

**올바른 방법**: 각 에이전트 정의에 반드시 **협업** 섹션을 포함하고, "누구에게 결과를 전달하는지"를 명시하세요.

---

## 3. 실전 에이전트 팀 설계 패턴

### 패턴 1: 파이프라인형 (순차 처리)

```
A → B → C → D
```
각 에이전트가 이전 에이전트의 결과를 받아 순차적으로 처리. 보고서 작성에 적합.

### 패턴 2: 허브 앤 스포크형 (중앙 조율)

```
    B
    ↑
A → 리더 → C
    ↓
    D
```
리더 에이전트가 작업을 분배하고 결과를 수집. 다각도 분석에 적합.

### 패턴 3: 피어 리뷰형 (상호 검증)

```
A ←→ B
↕      ↕
C ←→ D
```
에이전트들이 서로의 결과를 검토. 품질 최우선 작업에 적합.

> ⚠️ **현실적 기대치**: 본문의 2년 종단 연구와 Atlassian 보고서(96%의 기업이 AI ROI 미달성)가 보여주듯, AI 도구가 "팀워크 자체"를 대체하지는 못합니다. 에이전트 팀도 마찬가지입니다 — **관리 업무(기록, 정리, 통합)를 자동화**하는 것이지, 팀원 간의 토론, 의사결정, 창의적 기여를 대체하는 것이 아닙니다. 에이전트 팀이 생성한 결과물은 반드시 팀원들이 검토하고, 자신의 관점을 추가해야 합니다.

### 🎯 실습: 에이전트 팀 설계하기 ⭐⭐

**목표**: 조별과제 시나리오에 맞는 에이전트 팀을 설계합니다.
**소요 시간**: 20분
**준비물**: Claude Code 설치 완료
**사용 도구**: Claude Code

---

#### Step 1. 프로젝트 폴더 구성 ⏱️ 3분

```bash
mkdir -p ~/team-project/.claude/agents ~/team-project/sections ~/team-project/final
cd ~/team-project
```

✅ **체크**: `.claude/agents/` 폴더가 생성되면 성공

---

#### Step 2. CLAUDE.md 작성 ⏱️ 5분

```
다음 내용으로 CLAUDE.md를 만들어줘:
- 주제: "AI 시대의 대학 교육 혁신"
- 학술적 어투, 한국어 작성
- APA 인용 형식
- 모든 통계에 출처 필수
- AI 초안임을 인지하고 사용자 검토 필수
```

✅ **체크**: `CLAUDE.md`가 프로젝트의 규칙을 담고 있으면 성공

---

#### Step 3. 에이전트 4개 정의 ⏱️ 12분

위 2-2절을 참고하여 4개 에이전트 파일을 작성합니다:

```
.claude/agents/ 폴더에 다음 4개 에이전트를 만들어줘:
1. researcher.md — 자료 조사 (웹 검색, 출처 수집)
2. writer.md — 본문 작성 (섹션별 파일 생성)
3. fact-checker.md — 사실 검증 (주장별 판정)
4. editor.md — 편집 교정 (최종본 완성)
각 에이전트에 역할, 작업 원칙, 출력, 협업 섹션을 포함해줘.
```

✅ **체크**: `.claude/agents/` 폴더에 4개 `.md` 파일이 생성되면 성공

> ⚠️ **주의**: 이 실습에서는 에이전트 정의 파일을 "작성"하는 것이 목표입니다. 실제 에이전트 팀 실행은 다음 실습에서 다룹니다.

---

#### 🎉 완성! — 결과물 미리보기

4개 역할의 에이전트 팀을 설계했습니다. 이것은 실제 조별과제에서 역할 분담을 하는 것과 같은 원리입니다.

> 📊 **효과**:
> - 기존 방식: 1명이 모든 작업을 순차적으로 처리 — 검증 누락 빈번
> - AI 활용: 역할별 전문 에이전트가 체계적으로 분업 — 검증 포함 자동화
> - 핵심 가치: "분업의 원리"를 AI 워크플로우에 적용

### 🎯 실습: 에이전트 팀 실행 시뮬레이션 ⭐⭐⭐

**목표**: 설계한 에이전트 팀의 워크플로우를 실제로 실행합니다.
**소요 시간**: 45분
**준비물**: 이전 실습에서 만든 에이전트 정의 파일들
**사용 도구**: Claude Code

---

#### Step 1. researcher 역할 실행 ⏱️ 10분

```
researcher 역할로 작업해줘.
"AI 시대의 대학 교육 혁신"에 관한 최신 자료를 5개 이상 수집하고,
각 자료의 핵심 내용과 출처를 research_notes.md에 정리해줘.
참고문헌은 sources.md에 별도 저장해줘.
```

✅ **체크**: `research_notes.md`와 `sources.md`가 생성되면 성공

---

#### Step 2. writer 역할 실행 ⏱️ 15분

```
writer 역할로 작업해줘.
research_notes.md를 읽고 "AI 시대의 대학 교육 혁신" 보고서를 작성해줘.
서론은 sections/01_intro.md, 본론은 sections/02_body.md,
결론은 sections/03_conclusion.md로 각각 저장.
sources.md의 출처를 인용할 것.
```

✅ **체크**: `sections/` 폴더에 3개 파일이 생성되면 성공

---

#### Step 3. fact-checker 역할 실행 ⏱️ 10분

```
fact-checker 역할로 작업해줘.
sections/ 폴더의 모든 파일을 읽고 사실 관계를 검증해줘.
각 주장에 대해 사실/미확인/오류 판정과 근거를 제시하고,
결과를 factcheck_report.md로 저장해줘.
```

✅ **체크**: `factcheck_report.md`가 생성되고, 각 주장별 판정이 포함되면 성공

---

#### Step 4. editor 역할 실행 ⏱️ 10분

```
editor 역할로 작업해줘.
sections/ 폴더의 모든 파일과 factcheck_report.md를 읽고,
오류를 수정하고, 문법·흐름·형식을 교정하여
최종본을 final/report.md로 완성해줘.
목차와 참고문헌 목록을 포함할 것.
```

✅ **체크**: `final/report.md`가 생성되고, 목차와 참고문헌이 포함되면 성공

> 🔧 **안 되나요?**: 특정 에이전트의 결과가 만족스럽지 않으면, 해당 역할을 다시 실행하세요. 예: "fact-checker의 검증이 너무 간략해. 더 상세하게 다시 해줘."

---

#### 🎉 완성! — 결과물 미리보기

4개 에이전트 역할을 순차적으로 실행하여, 자료 조사 → 본문 작성 → 사실 검증 → 편집의 전체 워크플로우를 완성했습니다.

> 📊 **효과**:
> - 기존 방식: 4명이 역할을 나누고 일정을 조율하며 작업 — 1~2주
> - AI 활용: 에이전트 팀 시뮬레이션으로 전체 파이프라인 — 45분 (+ 검토·수정)
> - 핵심 가치: 실제 조별과제에서도 이 워크플로우를 AI 보조로 적용 가능

---

### 🔄 이 워크플로우를 다른 상황에 적용하기

| 이 실습의 사례 | → 이렇게 바꾸면 | 다른 활용 |
|--------------|----------------|----------|
| 보고서 작성 팀 | 역할을 "기획-디자인-개발-테스트"로 변경 | 소프트웨어 프로젝트 |
| 4명 팀 | "면접관-지원자-평가자"로 변경 | 모의 면접 시뮬레이션 |
| 파이프라인형 | 허브 앤 스포크형으로 변경 | 다각도 분석 프로젝트 |

**핵심 패턴**: `역할 정의 → 입출력 명시 → 협업 규칙 → 순차 실행` — 이것이 에이전트 팀 설계의 핵심입니다.

---

> **Tip**: 본문에서 배운 팀플 AI 워크플로우(클로바노트→Claude→공유 문서)와 에이전트 팀을 결합할 수 있습니다. 예: 회의록을 클로바노트로 기록하고, Claude Code의 에이전트 팀이 회의 결정사항을 바탕으로 각 역할별 작업을 자동 실행. 에이전트 팀의 품질은 각 에이전트의 **역할 정의**에 달려 있습니다 — 본문에서 팀원 강점 기반 역할 분담(A: 자료조사, B: 디자인, C: 발표, D: 데이터)이 중요했듯이, 에이전트에게도 "무엇을 하는지, 어떤 기준으로 하는지, 누구에게 전달하는지"를 구체적으로 기술할수록 결과가 좋아집니다.
