## 1. Claude Code로 보고서 구조화 작성하기

### 1-1. 왜 보고서 작성에 Claude Code인가?

본문에서 배운 것처럼, "레포트 써줘" 한마디로 끝내려는 것이 가장 흔한 실수입니다. 섹션 단위로 나눠야 합니다. 웹 채팅(GUI)에서는 "같은 대화에서 이어서 쓰세요"라는 제약이 있었지만, Claude Code는 **파일 시스템에 직접 접근**하므로:

- 보고서를 **여러 파일로 분리**하여 섹션별로 작성 가능
- 기존 파일을 **읽고 수정**하면서 점진적으로 완성 가능
- CLAUDE.md에 작성 규칙을 설정하면 **일관된 톤과 형식** 유지
- 비대화형 모드(`-p`)로 **반복 작업 자동화** 가능

### 1-2. 프로젝트 폴더 구조 설계

보고서 작성 전에, 프로젝트 폴더를 체계적으로 구성하면 Claude Code의 효율이 극대화됩니다:

```
~/report-project/
├── CLAUDE.md           ← 작성 규칙 (어투, 인용 형식, 분량 등)
├── outline.md          ← 보고서 개요
├── sections/
│   ├── 01_intro.md     ← 서론
│   ├── 02_theory.md    ← 이론적 배경
│   ├── 03_analysis.md  ← 본론/분석
│   └── 04_conclusion.md ← 결론
├── references/
│   └── sources.md      ← 참고문헌 목록
└── final/
    └── report.md       ← 최종 통합본
```

### 1-3. CLAUDE.md로 보고서 작성 규칙 설정

```markdown
# 보고서 작성 규칙

## 형식
- 학술적 어투를 사용합니다.
- 각 섹션은 마크다운 형식으로 작성합니다.
- 문단은 3~5문장으로 구성합니다.

## 인용
- APA 7th 인용 형식을 사용합니다.
- 통계 수치 인용 시 반드시 (저자, 연도) 형태로 표기합니다.
- 출처를 확인할 수 없는 정보는 "[출처 확인 필요]"로 표시합니다.

## 검증
- 팩트체킹이 필요한 통계·주장에는 [출처 필요]를 표시합니다.
- 참고문헌은 AI가 생성한 것을 그대로 쓰지 않고, 반드시 직접 확인합니다.

## 윤리
- AI가 작성한 초안임을 인지하고, 모든 내용은 사용자가 검토·수정합니다.
- 표절하지 않고, 기존 텍스트를 반드시 재구성합니다.
```

> 💡 **본문 연결**: 본문에서 배운 "[출처 필요] 마커"를 CLAUDE.md에 규칙으로 설정하면, Claude Code가 작성하는 모든 섹션에서 자동으로 검증이 필요한 부분을 표시합니다. 본문의 Perplexity 팩트체킹 단계와 연결됩니다.

### 1-4. 단계별 보고서 작성 워크플로우

```
Step 1. 개요 생성
> outline.md를 만들어줘. 주제: "[보고서 주제]"
> 서론-이론적배경-분석-결론 구조로, 각 섹션의 핵심 논점 3개씩 포함.

Step 2. 섹션별 초안 작성
> outline.md를 참고하여 sections/01_intro.md에 서론을 작성해줘.
> 분량: 2페이지 (약 1000자).

Step 3. 이전 섹션 참조하며 후속 작성
> sections/01_intro.md를 읽고, 이 서론의 흐름을 이어서
> sections/02_theory.md에 이론적 배경을 작성해줘.

Step 4. 전체 통합
> sections/ 폴더의 모든 md 파일을 순서대로 읽고,
> 하나의 문서로 통합하여 final/report.md로 저장해줘.
> 섹션 간 연결 문장을 자연스럽게 추가해줘.
```

## 2. 비대화형 모드로 보고서 일괄 처리

본문에서 배운 "섹션 단위 작성"의 원칙을 기억하세요. Claude Code의 비대화형 모드(`-p`)를 사용하면, 개요 기반으로 모든 섹션을 한 번에 자동 생성할 수 있습니다.

### 2-1. 섹션별 자동 생성 스크립트

개요를 기반으로 여러 섹션을 자동 생성하는 스크립트:

```bash
#!/bin/bash
# generate_sections.sh — 개요 기반 보고서 섹션 자동 생성

sections=("01_서론" "02_이론적배경" "03_분석" "04_결론")

for section in "${sections[@]}"; do
  echo "📝 $section 작성 중..."
  claude -p "
    outline.md를 읽고, '$section' 부분을 작성하세요.
    CLAUDE.md의 작성 규칙을 따르세요.
    분량: 약 1000자.
    이전 섹션과의 흐름을 자연스럽게 연결하세요.
  " > "sections/${section}.md"
  echo "✓ sections/${section}.md 완료"
done

echo "🎉 전체 섹션 생성 완료!"
```

### 2-2. 보고서 품질 점검 자동화

작성된 보고서를 자동으로 점검하는 활용법:

```
> sections/ 폴더의 모든 md 파일을 순서대로 읽고 다음을 점검해줘:
> 1. 섹션 간 논리적 흐름이 자연스러운가?
> 2. 반복되는 내용이 있는가?
> 3. 출처가 필요한데 빠진 주장이 있는가?
> 4. 전체 분량이 적절한가?
> 결과를 review.md로 저장해줘.
```

**상황**: Claude Code로 긴 보고서를 작성할 때

**잘못된 방법 1 — 한 번에 전체 작성 요청**:
```
> 15페이지 분량의 보고서를 한 번에 써줘
```

**결과**: AI가 토큰 제한에 걸려 뒷부분이 잘리거나, 전반부는 상세하고 후반부는 성기게 됩니다. 수정도 전체를 다시 생성해야 해서 비효율적입니다.

**올바른 방법**:
```
> outline.md를 먼저 만들고, 섹션별로 나누어서 하나씩 작성해줘.
> 먼저 sections/01_intro.md부터 시작.
```

**차이**: 섹션별로 나누면 각 부분의 품질을 개별 확인하고, 필요한 섹션만 재생성할 수 있습니다.

---

**잘못된 방법 2 — 이전 맥락 무시**:
```
> 본론을 써줘
```

**결과**: 서론에서 제시한 연구 질문이나 이론적 배경과 동떨어진 본론이 생성됩니다.

**올바른 방법**:
```
> sections/01_intro.md와 sections/02_theory.md를 읽고,
> 서론의 연구 질문과 이론적 배경을 반영하여
> sections/03_analysis.md를 작성해줘.
```

**차이**: 이전 섹션을 명시적으로 참조하면, AI가 맥락을 유지하며 일관성 있는 글을 작성합니다.

## 3. 하네스 심화: 보고서 작성 에이전트 팀

긴 보고서를 작성할 때, 에이전트 팀으로 역할을 분리하면 품질이 향상됩니다:

```
.claude/
  agents/
    outliner.md      ← 개요 설계 전문 에이전트
    writer.md        ← 본문 작성 에이전트
    reviewer.md      ← 품질 검토 에이전트
```

**outliner.md**:
```markdown
---
name: outliner
description: "보고서 개요를 설계하는 에이전트"
---
# Outliner
- 주제를 분석하여 논리적 개요를 설계합니다.
- 각 섹션의 핵심 논점과 예상 분량을 제시합니다.
- 완료 후 writer에게 전달합니다.
```

**writer.md**:
```markdown
---
name: writer
description: "개요를 바탕으로 보고서 본문을 작성하는 에이전트"
---
# Writer
- outliner의 개요를 참조하여 섹션별 본문을 작성합니다.
- 이전 섹션과의 흐름을 유지합니다.
- CLAUDE.md의 작성 규칙을 엄격히 따릅니다.
- 완료 후 reviewer에게 전달합니다.
```

**reviewer.md**:
```markdown
---
name: reviewer
description: "작성된 보고서의 품질을 검토하는 에이전트"
---
# Reviewer
- 논리적 흐름, 인용 누락, 반복 내용을 점검합니다.
- 섹션 간 연결이 자연스러운지 확인합니다.
- 문제 발견 시 수정 제안을 작성합니다.
```

이렇게 구성하면, 개요 설계 → 본문 작성 → 품질 검토가 체계적으로 이루어집니다.
