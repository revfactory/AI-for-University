# 🔧 Claude Code & 하네스 활용 — Chapter 18 보충

> **대상 독자**: Ch 17에서 프로젝트 하네스를 경험한 학생
> 본문에서 ChatGPT·Claude로 복잡한 문제를 분석하고 해결책을 도출하는 방법을 배웠습니다. 이번 보충에서는 **Claude Code + 에이전트 팀으로 체계적 문제 해결**을 구현합니다. 본문에서 소개한 ScienceDirect(2024) 연구 — ChatGPT가 **유창성(fluency)과 유연성(flexibility)을 향상**시키되, **참신성(novelty)은 감소**시키는 경향 — 은 AI 문제 해결의 강점과 한계를 동시에 보여줍니다. 핵심 차이: 웹 UI에서는 "이 문제 해결해줘"라고 한 번에 요청하지만, CC 하네스는 **문제 분석가 + 솔루션 설계자 + 검증자**가 각 단계를 분담합니다. 본문의 **SPARK 프레임워크**(Scenario→Problem→Aspiration→Resources→Keys)와 디자인 씽킹 5단계를 에이전트 파이프라인으로 자동화하여, "감에 의존한 해결"이 아닌 **"구조화된 문제 해결"**을 경험합니다. 특히 AI가 "안전한 답"을 선호하는 참신성 감소 문제를, CC 에이전트의 SCAMPER·역발상 기법으로 보완합니다.

## 1. 에이전트 팀 문제 해결의 장점

### 1-1. 단일 AI vs 에이전트 팀 문제 해결

| 항목 | 웹 UI (ChatGPT 등) | Claude Code 에이전트 팀 |
|------|-------------------|----------------------|
| 문제 분석 | "해결해줘" 일괄 요청 | 분석가가 구조화된 분해 |
| 해결책 도출 | 하나의 답변 | 다수 솔루션 비교·평가 |
| 검증 | 없음 | 검증 에이전트가 실현가능성 평가 |
| 방법론 | 임의 | 디자인 씽킹/5 Whys/MECE 자동 적용 |
| 결과물 | 텍스트 응답 | 분석 보고서 + 실행 계획 + 프로토타입 |
| 재사용 | 불가 | 스크립트로 유사 문제에 재적용 |
| 노코드 프로토타입 | 본문 v0/Lovable/Bolt.new (시각적 UI) | CC 에이전트(마크다운 와이어프레임·코드 MVP) |
| 공감 단계 | 현장 관찰 + AI 패턴 분석 (인간 주도) | 인터뷰 시나리오 생성 + 페르소나 자동화 |

### 1-2. 문제 해결 파이프라인

```
[문제 정의] → [근본 원인 분석] → [솔루션 생성] → [평가·선택] → [실행 계획]
     ↓              ↓                ↓              ↓              ↓
 problem-analyst  problem-analyst  solution-designer  evaluator    solution-designer
 (5 Whys, MECE)  (fishbone)      (브레인스토밍)     (매트릭스)   (프로토타입)
```

**problem-analyst.md**:

```markdown
---
name: problem-analyst
description: "문제를 구조적으로 분석하고 근본 원인을 식별하는 에이전트"

# Problem Analyst — 문제 분석가

## 핵심 역할
- 문제를 MECE(Mutually Exclusive, Collectively Exhaustive) 원칙으로 분해합니다.
- 5 Whys 기법으로 근본 원인을 탐색합니다.
- 이해관계자 분석과 영향 범위를 정리합니다.

## 분석 도구
- 5 Whys: "왜?" 5번 반복으로 근본 원인 도달
- MECE 분해: 중복 없이, 누락 없이 문제 요소 분류
- 피시본(Ishikawa) 다이어그램: 원인을 카테고리별로 정리
- 이해관계자 맵: 누가 영향을 받고, 누가 결정권이 있는가
- 현상 vs 원인 구분: 보이는 증상과 실제 원인을 분리

## 작업 원칙
- 문제를 한 문장으로 정의 (HMW — How Might We 형식)
- "가정"과 "사실"을 구분하여 표시
- 분석 결과에 "확인 필요" 항목 태그

## 출력
- output/analysis/문제분석_[주제].md: 구조화된 분석
```

**solution-designer.md**:
```markdown
---
name: solution-designer
description: "문제 분석을 바탕으로 다수의 솔루션을 설계하고 프로토타입을 제작하는 에이전트"
---
# Solution Designer — 솔루션 설계자

## 핵심 역할
- 문제 분석 결과를 바탕으로 최소 3개 솔루션 후보를 생성합니다.
- 각 솔루션의 프로토타입(개념 검증)을 제작합니다.
- 실행에 필요한 자원, 시간, 리스크를 명시합니다.

## 설계 도구
- 브레인스토밍: 제약 없이 아이디어 다수 생성
- SCAMPER: 기존 솔루션을 변형하여 새 아이디어 도출
- 제약 기반 설계: constraints.md의 현실 조건 반영
- 프로토타입: 마크다운 와이어프레임, 코드 MVP, 프로세스 도식

## 작업 원칙
- 솔루션 최소 3개 (보수적/현실적/혁신적)
- 각 솔루션에 "장점/단점/리스크/필요자원" 명시
- 프로토타입은 "30분 내 구현 가능한 수준"

## 출력
- output/solutions/솔루션후보_[주제].md: 솔루션 비교
```

**evaluator.md**:
```markdown
---
name: evaluator
description: "솔루션 후보를 다각도로 평가하고 최적안을 선정하는 에이전트"
---
# Evaluator — 평가자

## 핵심 역할
- 솔루션 후보를 평가 매트릭스로 비교합니다.
- 실현가능성, 효과, 비용, 리스크를 점수화합니다.
- 최적 솔루션을 선정하고 실행 계획을 작성합니다.

## 평가 기준
- 실현가능성 (30%): 현재 자원과 역량으로 가능한가
- 효과 (30%): 문제를 얼마나 해결하는가
- 비용/시간 (20%): 투입 대비 효율
- 리스크 (20%): 실패 시 피해, 부작용

## 작업 원칙
- 평가 매트릭스는 투명하게 (점수 산출 근거 공개)
- "왜 이 솔루션이 최적인지" 논리적 설명 포함
- 차선책(Plan B)도 함께 제시
- 실행 계획: 단계별 일정, 담당자, 체크포인트

## 출력
- output/plan/평가결과_[주제].md: 평가 매트릭스 + 최적안
- output/plan/실행계획_[주제].md: 단계별 실행 계획
```

### 2-2. CLAUDE.md 문제 해결 규칙

```markdown
# 문제 해결 시스템 규칙

## 분석 원칙
- 문제를 "해결"하기 전에 반드시 "분석"부터 (분석 없는 해결책 금지)
- "가정"과 "사실"을 구분하여 표시
- 이해관계자 관점을 반드시 포함
- 본문 핵심: "공감(Empathize) 단계는 인간이 주도" (ResearchGate, 2023) — AI는 아이디어 생성과 프로토타입에서 가장 효과적이지만, 사용자의 진짜 고통은 현장에서만 발견됨
- 현장 관찰·인터뷰 데이터를 먼저 수집한 후 AI에게 패턴 분석을 요청할 것

## 솔루션 원칙
- 최소 3개 솔루션 후보 (1개만 제시 금지)
- 각 솔루션에 장단점·리스크 명시
- 제약 조건(constraints.md)을 반드시 반영

## 평가 원칙
- 평가 기준과 가중치를 사전에 정의
- 점수 산출 근거를 투명하게 공개
- 최적안 + 차선안 함께 제시
```

### 🔬 프롬프트 진화 실험: 문제 해결

#### 🔴 Level 1 — 초보 프롬프트

```
학교 동아리 참여율이 떨어지고 있어. 해결해줘.
```

**결과**: "SNS 홍보를 강화하세요", "이벤트를 기획하세요" 등 뻔한 조언이 나옵니다.

📋 **평가**:
- 정확성: ★★☆☆☆
- 활용도: ★☆☆☆☆
- 문제점: 원인 분석 없음, 제약 조건 없음

---

#### 🟡 Level 2 — 중급 프롬프트

```
대학 코딩 동아리 참여율이 50%→20%로 하락.
원인 3가지와 해결책 5가지를 제안해줘.
주 예산 10만원, 임원 3명.
```

**결과**: 원인과 해결책이 나열되지만 구조화되지 않고, 실현가능성 검증이 없습니다.

📋 **평가**:
- 정확성: ★★★☆☆
- 활용도: ★★★☆☆
- 개선점: 구체적 상황과 제약 조건 포함
- 아쉬운 점: 구조적 분석 없음, 평가 없음

---

#### 🟢 Level 3 — 전문가 프롬프트 (하네스 활용)

```
input/problem_statement.md를 읽고 문제를 해결해줘.

@problem-analyst 5 Whys + MECE + 이해관계자 분석
@solution-designer 최소 3개 솔루션 + 프로토타입
@evaluator 평가 매트릭스 + 최적안 + 실행 계획

제약 조건: input/constraints.md 참조.
```

📋 **평가**:
- 정확성: ★★★★★
- 활용도: ★★★★★
- 핵심 차이: 구조화된 분석 + 다수 솔루션 + 객관적 평가 + 실행 계획

### ❌ 흔한 실수: 문제 해결의 함정

**상황**: Claude Code로 문제를 해결할 때

---

**잘못된 방법 — 분석 없이 바로 해결책 요청**:

```
참여율을 높일 방법을 알려줘. 분석은 필요 없어, 답만 줘.
```

**결과**: 근본 원인을 모르니 증상만 치료하는 해결책이 나옵니다. "홍보를 더 하자"라고 했는데, 실제 원인이 "모임 시간이 수업과 겹침"이었다면 홍보는 무의미합니다. 본문 Anti-Pattern 1 "AI는 공감하지 못한다"와 동일 — 현장 관찰 없이 AI가 정의한 "평균 대기시간 20분"을 팩트로 수용했더니, 실제로는 식당마다 패턴이 달랐고 학생들은 "대기시간"보다 "메뉴 정보 부족"을 더 큰 불편으로 느꼈습니다.

**올바른 방법**: problem-analyst가 5 Whys로 근본 원인을 파악한 후, 그 원인에 맞는 솔루션을 설계합니다.

**차이**: CLAUDE.md의 **"분석 없는 해결책 금지"** 규칙이 성급한 결론을 방지합니다.

---

**잘못된 방법 — AI가 추천한 1순위 솔루션만 실행**:

```
1순위 솔루션만 알려줘. 나머지는 필요 없어.
```

**결과**: 1순위가 실행 불가능하거나 예상치 못한 부작용이 있을 때 대안이 없습니다.

**올바른 방법**: evaluator가 최소 3개 솔루션을 비교 평가하고, 최적안 + 차선안(Plan B)을 함께 제시합니다.

**차이**: 항상 대안이 준비되어 있어야 실전에서 유연하게 대응할 수 있습니다.

## 실습 과제

### 🎯 실습: 문제 분석 + 솔루션 도출 ⭐

**목표**: 실제 문제를 구조적으로 분석하고 해결책을 도출합니다.
**소요 시간**: 10분
**준비물**: Claude Code
**사용 도구**: Claude Code

---

#### Step 1. 문제 정의 ⏱️ 3분

```
다음 문제를 구조화해줘:

"[실제 문제 또는 샘플: 대학 동아리/수업/프로젝트에서 겪는 문제]"

1. 문제를 한 문장으로 정의 (HMW 형식)
2. 5 Whys로 근본 원인 탐색
3. 이해관계자 목록

input/problem_statement.md에 저장.
```

✅ **체크**: HMW 문제 정의 + 5 Whys + 이해관계자가 포함되면 성공

---

#### Step 2. 솔루션 브레인스토밍 ⏱️ 4분

```
문제 분석 결과를 바탕으로 솔루션 3개를 설계해줘.
각 솔루션에: 설명, 장점, 단점, 예상 비용, 필요 시간.
output/solutions/에 저장.
```

✅ **체크**: 3개 솔루션 + 장단점이 포함되면 성공

---

#### Step 3. 간단 평가 ⏱️ 3분

```
3개 솔루션을 실현가능성/효과/비용 기준으로 점수화(10점 만점)해줘.
최적안과 이유를 설명.
```

✅ **체크**: 평가 점수 + 최적안 선정이 포함되면 성공

> 🔧 **안 되나요?**: 문제가 떠오르지 않으면 → "대학생이 실제로 겪는 문제 상황 5개를 제시해줘"라고 요청하세요.

### 🎯 실습: 프로토타입 + 실행 계획 ⭐⭐

**목표**: 최적 솔루션의 프로토타입을 만들고 실행 계획을 수립합니다.
**소요 시간**: 25분
**준비물**: 이전 실습의 분석 결과
**사용 도구**: Claude Code

---

#### Step 1. 프로토타입 제작 ⏱️ 10분

```
최적 솔루션의 프로토타입을 만들어줘.

[문제 유형별 선택]
- 서비스/앱 문제: 마크다운 와이어프레임 + 사용자 플로우
- 프로세스 문제: 개선된 프로세스 도식 + Before/After 비교
- 조직 문제: 새 역할/규칙 제안 + 적용 시나리오
- 데이터 문제: 분석 코드 + 시각화 + 인사이트

output/solutions/프로토타입/에 저장.
```

✅ **체크**: 프로토타입이 생성되면 성공

---

#### Step 2. 실행 계획 수립 ⏱️ 8분

```
프로토타입을 실제로 적용하기 위한 실행 계획을 만들어줘.

포함:
1. 단계별 일정 (주 단위)
2. 각 단계의 담당자 + 산출물
3. 필요 자원 (예산, 도구, 인력)
4. 리스크 요인 + 대응 방안
5. 성과 측정 지표 (KPI)
6. Plan B (차선안) 적용 조건

output/plan/실행계획.md에 저장.
```

✅ **체크**: 실행 계획 + 리스크 대응 + KPI가 포함되면 성공

---

#### Step 3. 문제 해결 보고서 ⏱️ 7분

```
전체 과정을 종합한 문제 해결 보고서를 작성해줘.

구조:
1. 문제 정의
2. 분석 결과 (5 Whys, MECE)
3. 솔루션 비교
4. 최적안 + 프로토타입
5. 실행 계획
6. 기대 효과

output/plan/문제해결보고서.md에 저장.
```

✅ **체크**: 6개 섹션이 포함된 보고서가 생성되면 성공

### 🎯 실습: 종합 문제 해결 하네스 구축 ⭐⭐⭐

**목표**: 문제 분석, 솔루션 설계, 평가를 에이전트로 분업한 문제 해결 하네스를 구축합니다.
**소요 시간**: 40분
**준비물**: Claude Code
**사용 도구**: Claude Code

---

#### Step 1. 문제 해결 하네스 구축 ⏱️ 12분

```
~/problem-solver/ 에 문제 해결 하네스를 구축해줘.

폴더 구조: [위의 구조 참조]
에이전트 3개 + CLAUDE.md + 스크립트
input/ 에 문제 샘플 + 제약 조건도 만들어줘.
```

✅ **체크**: 전체 구조 + 에이전트 3개 + 스크립트가 완성되면 성공

---

#### Step 2. 복잡한 문제 에이전트 순차 해결 ⏱️ 15분

```
"대학 캠퍼스 음식물 쓰레기 30% 감축"을 해결해줘.

1단계 @problem-analyst: 5 Whys + MECE + 이해관계자 + 피시본
2단계 @solution-designer: 솔루션 5개 + 프로토타입 Top 2
3단계 @evaluator: 평가 매트릭스 + 최적안 + 실행 계획

제약: 예산 200만원, 기간 1학기, 학생자치회 주관
```

✅ **체크**: 구조화된 분석 + 5개 솔루션 + 평가 + 실행 계획이 완료되면 성공

> ⚠️ **주의**: AI의 솔루션이 "이론적으로 완벽"해도 현장 적용 시 예상치 못한 문제가 발생합니다. 실행 전 관계자 인터뷰와 소규모 파일럿이 필수입니다.

---

#### Step 3. 스크립트 자동화 ⏱️ 8분

```
새로운 문제를 해결할 때 사용할 스크립트를 만들어줘.

사용법: bash scripts/solve_problem.sh [문제명]

동작:
1. input/problem_statement.md 확인
2. 3단계 순차 실행 (분석→설계→평가)
3. 결과물 output/에 정리
4. 요약 출력
```

✅ **체크**: 스크립트로 전 과정이 자동 실행되면 성공

---

#### Step 4. 디자인 씽킹 워크숍 시뮬레이션 ⏱️ 5분

```
디자인 씽킹 5단계를 CC 에이전트로 시뮬레이션해줘.

1. 공감(Empathize): 사용자 인터뷰 시나리오 생성
2. 정의(Define): 문제 재정의 (HMW)
3. 아이디에이션(Ideate): 솔루션 브레인스토밍
4. 프로토타입(Prototype): 간단한 프로토타입
5. 테스트(Test): 사용자 피드백 시나리오

각 단계를 에이전트가 수행.
```

✅ **체크**: 디자인 씽킹 5단계가 시뮬레이션되면 성공

---

#### 🎉 완성! — 결과물 미리보기

분석 → 솔루션 → 평가 → 실행 → 디자인 씽킹까지 **체계적 문제 해결 시스템**을 구축했습니다.

> 📊 **효과**:
> - 기존 방식: 팀 회의에서 "아이디어 나열" → 가장 인기 있는 안 선택 — 근거 없음
> - 하네스 활용: 구조화된 분석 + 다수 솔루션 + 객관적 평가 — 논리적이고 투명한 의사결정
> - 핵심 가치: "좋은 해결책"보다 "좋은 문제 정의"가 먼저

### 🔄 이 워크플로우를 다른 상황에 적용하기

| 이 실습의 사례 | → 이렇게 바꾸면 | 다른 활용 |
|--------------|----------------|----------|
| 캠퍼스 문제 해결 | 기업 과제 해결 | 사례 분석 과제 |

### 📊 하이브리드 전략 비교표: 본문 도구 vs CC 하네스

| 문제 해결 단계 | 본문 도구 조합 | CC 하네스 | 최적 전략 |
|--------------|--------------|----------|----------|
| **1. 공감(Empathize)** | 현장 관찰 + Perplexity 리서치 + ChatGPT 페르소나 | problem-analyst(인터뷰 시나리오 생성) | **현장 관찰은 인간 필수** → CC로 관찰 데이터 구조화 |
| **2. 정의(Define)** | ChatGPT로 HMW 질문 + Claude 논리 검증 | problem-analyst(5 Whys + MECE + fishbone) | 본문 HMW로 시작 → CC로 근본 원인 심층 분석 |
| **3. 아이디어(Ideate)** | ChatGPT SCAMPER + 역발상 + 6-3-5 시뮬레이션 | solution-designer(브레인스토밍 + SCAMPER + 제약 기반) | 본문 기법으로 발산 → CC로 3개 솔루션 구조화 |
| **4. 프로토타입** | **v0/Lovable/Bolt.new(시각적 UI 최강)** | CC 에이전트(마크다운 와이어프레임·코드 MVP) | **노코드 UI는 본문 도구 우위** — CC는 프로세스·데이터 프로토타입에 강점 |
| **5. 테스트** | Google Forms + ChatGPT 피드백 분석 | evaluator(평가 매트릭스 + KPI + Plan B) | 본문으로 실사용자 피드백 수집 → CC evaluator로 객관적 평가 |
| **6. 반복(Iterate)** | 새 대화에서 재시작 | 스크립트로 즉시 재실행 | CC의 **재현성**이 압도적 — 피드백 반영 후 전체 파이프라인 재실행 |

**핵심**: 본문의 디자인 씽킹 5단계와 SPARK 프레임워크는 **"무엇을 할 것인가"**를 정의하고, CC 에이전트는 **"어떻게 체계적으로 실행할 것인가"**를 자동화합니다. ScienceDirect(2024) 연구가 지적한 AI의 "참신성 감소" 문제는 solution-designer의 SCAMPER·역발상 기법이 보완하고, ResearchGate(2023) 논문이 강조한 "공감 단계는 인간 주도" 원칙은 CLAUDE.md 규칙으로 강제합니다.

---

> **핵심 정리**: 문제 해결에서 Claude Code의 진짜 가치는 **"분석과 해결을 분리"**하는 것입니다. problem-analyst가 근본 원인을 파악하고, solution-designer가 다수 해결책을 제안하고, evaluator가 객관적으로 평가합니다. 이 3단계 분리가 "감에 의존한 해결"을 "구조화된 해결"로 전환합니다. CLAUDE.md의 **"분석 없는 해결책 금지"** 규칙이 핵심 — 문제를 제대로 정의하면, 해결책의 절반은 이미 나온 것입니다.
