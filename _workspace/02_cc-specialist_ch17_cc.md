MDPI(2025) 연구에 따르면 AI 통합 프로젝트 기반 학습(AI-PBL)은 일반 PBL보다 **유의미하게 높은 학습 효과**를 보였습니다. 핵심 차이: 웹 UI에서는 "질문→응답→복사"의 반복이지만, CC 하네스는 **프로젝트 매니저(PM) + 리서처 + 구현 에이전트 + 리포터**가 역할을 분담하여 프로젝트의 기획→조사→구현→보고서 전 과정을 자동화합니다. 본문의 "멀티 도구 워크플로우 3원칙(도구별 역할 분담·파이프라인 설계·1도구 1역할)"을 CC 하네스는 **에이전트별 역할 분담**으로 한 단계 더 체계화합니다. 전공별 특성에 맞는 에이전트를 정의하면 같은 하네스로 경영학·공학·인문학·사회과학 프로젝트를 모두 수행할 수 있습니다.

## 1. 전공 프로젝트에 하네스가 필요한 이유

### 1-1. 웹 UI 프로젝트 vs CC 하네스 프로젝트

| 항목 | 웹 UI (ChatGPT 등) | Claude Code 하네스 |
|------|-------------------|-------------------|
| 프로젝트 관리 | 대화창에 산발적 요청 | PM 에이전트가 단계별 관리 |
| 자료 조사 | 매번 "조사해줘" 반복 | 리서처 에이전트 체계적 수집 |
| 결과물 생성 | 복사-붙여넣기 조합 | 자동 보고서·코드·시각화 |
| 파일 관리 | 로컬에 수동 저장 | 프로젝트 폴더 구조 자동 관리 |
| 재현성 | 대화 히스토리에 의존 | 스크립트로 동일 과정 재실행 |
| 팀 프로젝트 | 역할 분담 어려움 | 에이전트별 역할 분담 = 팀원별 참고 |
| 멀티 도구 조합 | 본문 4도구 수동 전환 (Perplexity→ChatGPT→Claude→Gamma) | CC 에이전트가 파이프라인 자동 연결 |
| AI 활용 구분 | "AI vs 직접 작업" 수동 표시 | CLAUDE.md 규칙으로 자동 태그 |

### 1-2. 전공별 하네스 활용 맵

```
[경영학] 시장 분석 → SWOT → 전략 보고서
[공학] 요구사항 → 설계 → 프로토타입 → 테스트
[인문학] 문헌 조사 → 분석 → 논증 → 에세이
[사회과학] 가설 → 데이터 수집 → 분석 → 해석
[디자인] 리서치 → 아이디에이션 → 프로토타입 → 사용성 평가

공통 패턴: 기획(PM) → 조사(리서처) → 실행(구현) → 정리(리포터)
```

# Project Manager — 프로젝트 매니저

## 핵심 역할
- project_brief.md를 분석하여 프로젝트 계획을 수립합니다.
- 단계별 마일스톤과 산출물을 정의합니다.
- 다른 에이전트에게 작업을 분배하고 진행 상황을 추적합니다.

## 작업 원칙
- 프로젝트를 4~6단계로 분해 (각 단계에 명확한 산출물)
- 전공별 프로젝트 방법론 반영 (경영: 사례분석, 공학: 설계-구현-테스트)
- 각 단계의 예상 소요 시간 명시
- 위험 요소와 대안 제시
- 팀 프로젝트 시 팀원별 역할 분담 제안

## 출력
- _workspace/project_plan.md: 프로젝트 계획서
```

# Researcher — 리서처

## 핵심 역할
- 프로젝트 주제에 관한 자료를 체계적으로 조사합니다.
- 선행 연구, 사례, 데이터를 수집하고 정리합니다.
- 조사 결과를 구조화된 보고서로 작성합니다.

## 작업 원칙
- 조사 범위를 먼저 정의 (키워드, 분야, 시간 범위)
- 출처를 반드시 명시 (학술자료 > 기관보고서 > 뉴스)
- 찬반/다각도 관점 포함
- "조사한 것"과 "추정한 것"을 명확히 구분
- 전공 분야의 학술 용어와 프레임워크 활용

## 출력
- output/research/조사보고서_[주제].md: 조사 결과
```

### 2-2. CLAUDE.md 프로젝트 관리 규칙

```markdown
# 전공 프로젝트 관리 규칙

## 프로젝트 원칙
- 프로젝트 계획 → 조사 → 구현 → 보고 순서 준수
- 각 단계의 산출물이 다음 단계의 입력이 됨
- 모든 산출물에 날짜와 버전 표시

## 학술 윤리
- AI가 생성한 내용은 반드시 검증 후 사용
- 출처 없는 데이터/통계를 "사실"로 제시하지 않음
- "AI 활용 사실"을 보고서에 명시 (학교 정책 준수)
- 표절 금지 — AI 결과물도 본인의 비판적 분석을 거쳐야 함
- 본문 원칙: "AI가 못하는 부분을 직접 하라" — AI가 해준 부분보다 직접 한 부분이 학점과 면접에서 가치가 있음

## 멀티 도구 원칙 (본문 3원칙 연계)
- 1도구 1역할: Perplexity=리서치, ChatGPT=분석, Claude=검증, Gamma=시각화 → CC에서는 각 에이전트가 1역할
- AI 활용 부분 vs 직접 작업 부분을 모든 산출물에 명시
- 교수에게 AI 활용 범위를 사전 확인

## 팀 프로젝트 연계
- 에이전트 역할 = 팀원 역할의 참고 모델
- 팀원 각자가 에이전트를 활용하여 담당 파트를 수행
- 최종 통합은 reporter가 일관성 검토
```

### ❌ 흔한 실수: 전공 프로젝트의 함정

**상황**: Claude Code로 전공 프로젝트를 수행할 때

---

**잘못된 방법 — AI에게 보고서 전체를 맡기기**:

```
이 주제로 10페이지짜리 보고서를 처음부터 끝까지 써줘
```

**결과**: AI가 일관된 톤의 보고서를 생성하지만, 본인의 분석과 비판적 사고가 빠져 있습니다. 교수님이 "이건 AI가 쓴 것 같은데?"라고 피드백합니다. 본문 Anti-Pattern 1의 "코드는 돌아가지만 이해는 못한다" 사례와 동일 — R-squared 0.95의 과적합 가능성을 모르면 C학점입니다.

**올바른 방법**: 에이전트 팀이 **"재료"**(조사 결과, 데이터, 시각화)를 제공하고, **"분석과 해석"**은 본인이 직접 작성합니다. reporter는 형식과 구조를 잡아주지만, 핵심 논증은 사람의 것입니다.

**차이**: CLAUDE.md에 "AI는 재료 공급, 분석은 사람" 규칙을 설정하면 이 경계가 명확해집니다.

---

**잘못된 방법 — 데이터 검증 없이 AI 결과 신뢰**:

```
이 분야의 시장 규모와 성장률을 알려줘. 그대로 보고서에 넣을 거야.
```

**결과**: AI가 생성한 "2026년 시장 규모 1.2조원"이 실제 데이터와 다를 수 있습니다. 검증 없이 제출하면 학점은 물론 학술 윤리 문제로 이어집니다.

**올바른 방법**: researcher 에이전트가 출처를 명시하도록 하고, builder가 "⚠️ 출처 미확인" 태그를 자동으로 표시합니다. CLAUDE.md에 "모든 데이터에 출처 필수, 미확인 데이터는 경고 태그" 규칙을 설정합니다.

**차이**: 시스템 수준에서 검증되지 않은 데이터가 최종 보고서에 포함되는 것을 방지합니다.

#### Step 1. 프로젝트 브리프 작성 ⏱️ 4분

```
나의 전공 프로젝트 브리프를 작성해줘.
input/project_brief.md에 저장.

포함 항목:
- 전공: [본인 전공]
- 프로젝트 주제: [주제]
- 목표: 무엇을 밝히려 하는가
- 범위: 어디까지 다룰 것인가
- 산출물: 최종적으로 무엇을 제출하는가
- 제약 조건: 기한, 분량, 형식

샘플을 채워주고 나중에 수정할 수 있게 해줘.
```

✅ **체크**: `input/project_brief.md`에 프로젝트 브리프가 생성되면 성공

#### Step 2. 프로젝트 계획 수립 ⏱️ 3분

### 📊 하이브리드 전략 비교표: 본문 도구 vs CC 하네스

| 프로젝트 단계 | 본문 도구 조합 | CC 하네스 | 최적 전략 |
|-------------|--------------|----------|----------|
| **1. 주제 발상** | ChatGPT로 5개 주제 제안 | PM 에이전트가 브리프 분석 → 주제 추천 | 본문으로 브레인스토밍 → CC로 체계화 |
| **2. 배경 조사** | Perplexity(출처 포함 리서치) | researcher 에이전트(체계적 수집·분류) | Perplexity로 초기 데이터 → CC researcher로 구조화 |
| **3. 분석·구현** | ChatGPT CI(데이터 분석) + Claude(논리 검증) | builder 에이전트(코드·시각화·분석) | 본문 도구로 탐색 → CC builder로 재현 가능한 코드 생성 |
| **4. 보고서 작성** | ChatGPT + Claude로 구조화 | reporter 에이전트(APA 형식 자동) | 본문으로 초안 아이디어 → CC reporter로 형식 완성 |
| **5. 발표 자료** | Gamma(60초 슬라이드 생성) | reporter(마크다운 슬라이드) | **Gamma가 시각적으로 우위** — CC는 내용 구조화에 강점 |
| **6. 팀 프로젝트** | 팀원 수동 역할 분담 | 에이전트 ↔ 팀원 매핑 가이드 | CC 에이전트 구조를 팀원 역할의 참고 모델로 활용 |

**핵심**: 본문의 "Perplexity→ChatGPT→Claude→Gamma" 4도구 파이프라인은 **각 도구를 수동 전환**하는 방식이고, CC 하네스는 **에이전트 파이프라인이 자동 연결**되는 방식입니다. 반복적 프로젝트(매 학기 과제)에서는 CC 하네스가 압도적으로 효율적이고, 단발성 탐색에서는 본문 도구가 빠릅니다. Harvard 물리학 연구의 "AI 튜터로 2배 학습" 결과는 두 방식 모두에서 AI를 **학습 도구**로 활용할 때 극대화됩니다.

---

> **핵심 정리**: 전공 프로젝트에서 Claude Code의 진짜 가치는 **"에이전트 팀의 역할 분담"**입니다. project-manager가 계획을 세우고, researcher가 자료를 모으고, builder가 구현하고, reporter가 정리합니다. 이 구조는 팀 프로젝트의 역할 분담을 그대로 반영하므로, 팀원 각자가 담당 에이전트를 참고하여 작업할 수 있습니다. CLAUDE.md의 **"AI는 재료 공급, 분석은 사람"** 규칙이 핵심 — AI가 아무리 훌륭한 보고서를 생성해도, 본인의 비판적 분석과 고유한 관점이 없으면 학술적 가치가 없습니다.
